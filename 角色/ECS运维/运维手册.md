## 八、排坑经验

### 1. Hermes Agent 启动失败
**现象**：`RuntimeError: can't start new thread`

**原因**：官方镜像使用 Python 3.13，但 Hermes 只兼容 Python 3.11

**解决方案**：
- 等待官方发布兼容版本
- 或自行基于 Python 3.11 构建镜像

**相关文件**：
- 部署目录：`/root/songjian/docker-compose/hermes/`
- 配置目录：`/root/songjian/hermes/`

### 2. docker-compose 命令版本
**注意**：使用 `docker compose`（V2 命令）而非 `docker-compose`（V1 命令）

```bash
# 正确 ✅
docker compose up -d

# 错误 ❌（版本过旧）
docker-compose up -d
```

### 3. SRS 容器部署问题
**现象**：容器状态为 Created，无法启动，报错 `exec: "./objs/srs": stat ./objs/srs: no such file or directory`

**原因**：挂载目录覆盖了镜像内部的可执行文件目录
- 错误挂载：`/root/songjian/srs/logs:/usr/local/srs/objs` 覆盖了整个 objs 目录
- 导致 `/usr/local/srs/objs/srs` 可执行文件消失

**解决方案**：
- 不要挂载到 objs 目录，只挂载 config 和 data
- 添加 `command` 参数指定配置文件路径

**教训**：挂载目录前先检查镜像内部结构，避免覆盖关键文件

### 4. 直播卡顿问题（带宽瓶颈）
**现象**：HLS 和 FLV 播放都严重卡顿

**排查误区**：
- 一开始检查 CPU、内存、磁盘 IO、SRS 配置都正常
- 浪费时间优化 HLS 切片配置
- 最后才发现是带宽不足

**根本原因**：
- 阿里云 ECS 带宽 1 Mbps = 上下行共享，总共 1000 kbps
- 推流码率 2500+ kbps >> 1 Mbps 带宽上限

**解决方案**：
- 降低推流码率到 600k 视频 + 64k 音频 = 680 kbps
- 推流命令：
  ```bash
  ffmpeg -re -i <源流> \
    -c:v libx264 -preset ultrafast -g 60 -keyint_min 60 -b:v 600k -maxrate 800k -bufsize 1200k \
    -c:a aac -b:a 64k \
    -f flv rtmp://39.103.203.162:1935/live/test
  ```

**重要教训**：
- **排查性能问题第一步应该确认带宽上限**
- 阿里云 ECS 带宽是上下行共享，不是各算各的
- 推流 + 播放共享同一带宽池

**带宽规划建议**：
| 带宽 | 支持能力 |
|------|----------|
| 1 Mbps | 仅支持 1 路低码流（600-800 kbps）+ 少量观众 |
| 5 Mbps | 可支持 2-3 个观众（2500 kbps 码率） |
| 10 Mbps | 可支持 5-8 个观众 |
| 多观众 | 建议使用 CDN |

### 6. SSH 连接被拒（云盾拦截）
**现象**：SSH端口通（nc成功），但握手阶段被reset，`kex_exchange_identification: read: Connection reset by peer`

**排查过程**：
1. 一开始怀疑 fail2ban → 解封IP后仍被拒
2. 服务器上查不到连接记录 → 说明请求根本没到SSH层
3. 端口通但握手被reset → 阿里云云盾/DDoS防护在TCP层拦截

**根本原因**：阿里云云盾识别到频繁SSH连接（Agent自动化连接），在TCP层直接拦截，fail2ban都看不到

**解决方案**：
- 部署 Remote Gateway（HTTP命令网关），通过HTTP接口执行命令，绕开SSH
- Token认证 + IP白名单保证安全
- systemd 托管，开机自启

**教训**：云服务器频繁SSH连接会触发云盾，自动化Agent应使用HTTP API而非SSH

### 7. Docker 本地构建 OOM
**现象**：1.8G内存服务器 `docker build` 报 `lzma error: Cannot allocate memory`

**原因**：
- Debian/Ubuntu基础镜像的dpkg解压非常吃内存
- Docker BuildKit容器不受宿主机swap影响
- 即使加了2G swap，BuildKit容器内仍然OOM

**尝试方案及结果**：
| 方案 | 结果 |
|------|------|
| 加2G swap | ❌ BuildKit容器不受宿主机swap影响 |
| 关闭BuildKit用传统构建器 | ❌ 仍然OOM |
| 换Alpine基础镜像 | apk安装OK，但静态ffmpeg下载极慢 |

**最终方案**：放弃本地构建，使用预构建镜像 `ghcr.io/youbanzhishi/vcmix:core-latest`

**教训**：1.8G内存服务器**不要本地build镜像**，一律用预构建镜像pull

### 8. Swap 配置（2026-05-08）
**操作**：添加2G swap文件并持久化

```bash
# 创建swap
fallocate -l 2G /swapfile
chmod 600 /swapfile
mkswap /swapfile
swapon /swapfile

# 持久化（已写入/etc/fstab）
echo '/swapfile none swap sw 0 0' >> /etc/fstab

# 验证
free -h
swapon --show
```

**注意**：swap对Docker BuildKit容器无效，但对宿主机上直接运行的进程有效

### 9. OpenBLAS 线程创建失败
**现象**：容器启动报 `OpenBLAS blas_thread_init: pthread_create failed`，numpy加载失败

**原因**：容器内内存/进程限制太严，OpenBLAS默认尝试创建多线程

**解决方案**：docker-compose.yml 添加环境变量
```yaml
environment:
  - OPENBLAS_NUM_THREADS=1
  - OMP_NUM_THREADS=1
```

### 10. Gateway Proxy v2 使用方式
**用途**：替代旧版Remote Gateway，通过HTTP执行命令，支持角色权限分级+两步确认+审计日志

**访问格式**：
```bash
curl "http://39.103.203.162:1806/?token=ops_xl2026ecs&cmd=<URL编码的命令>"
```

**角色Token**（从 ecs-tokens.enc 解密获取）：
- admin: admin_xl2026ops（全权限）
- ops: ops_xl2026ecs（运维权限）
- dev: dev_xl2026rust（只读子集）
- readonly: ro_xl2026view（只读子集）

**两步确认**：灰/黑名单命令第一次请求返回confirm_code，带confirm参数再次请求执行

**管理命令**：
```bash
systemctl start|stop|restart gateway-proxy    # 服务管理
cat /root/songjian/gateway-proxy/audit.log   # 审计日志
```

**配置文件**：`/root/songjian/gateway-proxy/config.json`（端口/角色Token/IP白名单）

**详细文档**：`角色/ECS运维/knowledge/Gateway-Proxy运维.md`

**⚠️ 旧版Remote Gateway(7772端口)已停用**，不要再用 `xl2026cmd` token

## 九、紧急处理流程

### 服务异常排查步骤

1. **检查容器状态**
   ```bash
   docker ps -a | grep <服务名>
   ```

2. **查看日志**
   ```bash
   docker logs <容器名> --tail 200
   ```

3. **检查资源**
   ```bash
   docker stats --no-stream
   free -h
   df -h
   ```

4. **检查网络**
   ```bash
   docker network ls
   docker network inspect c_bridge
   ```

### 服务无法启动

1. 检查端口是否被占用：`netstat -tlnp | grep <端口>`
2. 检查配置文件语法：`docker compose config`
3. 检查日志错误信息
4. 尝试删除容器重建：`docker compose down && docker compose up -d`
