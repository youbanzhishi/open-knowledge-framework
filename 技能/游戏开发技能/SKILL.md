## 全流程 Pipeline

### Phase 1: 策划 (Planning)

#### 1.1 创意立项
- 产出：`创意提案.md`（游戏名称、类型、核心玩法一句话、目标平台、参考竞品）
- 决策：用户确认方向后进入下一阶段

#### 1.2 游戏设计文档 (GDD)
- 产出：`GDD.md`（核心玩法、操作方式、关卡设计、数值体系、美术风格、音效需求、技术选型）
- 模板：`./references/gdd-template.md`
- 决策：用户审核通过后进入开发

#### 1.3 原型验证
- 产出：可交互的最小原型（核心玩法验证）
- 目标：3-7天内出原型，快速验证fun factor
- 决策：原型通过后进入正式开发

### Phase 2: 美术 (Art)

#### 2.1 美术风格定义
- 产出：`美术规范.md`（色彩方案、风格参考、尺寸规范、命名规范）
- 工具：image_generate / search_images / 外部美工技能

#### 2.2 素材制作清单
- 产出：`素材清单.md`（角色、场景、UI、特效、图标等，标注优先级和状态）
- 按优先级分批制作，不阻塞程序开发

#### 2.3 素材产出
- 角色图：角色设计 → 立绘 → 动画帧（如需）
- 场景图：场景概念 → 背景图 → 前景装饰
- UI：界面布局 → 切图 → 图标
- 特效：粒子效果 / 序列帧

**自我兜底方案**：使用像素风/扁平风/简笔画风格，image_generate + 代码绘制均可出活。

### Phase 3: 开发 (Development)

#### 3.1 技术选型决策树

```
目标平台？
├── 微信小游戏 / H5
│   ├── 简单2D → 原生Canvas + TypeScript
│   ├── 中等2D → Cocos Creator
│   └── 复杂2D/轻3D → Cocos Creator / Laya
├── Steam / PC
│   ├── 2D → Godot / Unity
│   └── 3D → Unity / Godot
├── 移动端 (iOS/Android)
│   ├── 2D → Unity / Cocos Creator
│   └── 3D → Unity / Unreal
└── 多平台同发
    └── Unity / Godot（跨平台优势）
```

#### 3.2 开发规范
- 产出：`技术方案.md`（架构设计、模块划分、技术栈、目录结构）
- 代码规范：统一命名、注释标准、Git提交规范
- 模块化开发：核心逻辑 / UI / 数据 / 音效 分层

#### 3.3 开发里程碑
- M1: 核心玩法可运行（无美术，用占位符）
- M2: 美术素材集成 + UI完成
- M3: 音效/特效 + 完整关卡
- M4: 打包构建 + 平台适配

### Phase 4: 测试 (Testing)

#### 4.1 测试清单
- 产出：`测试用例.md`（功能测试、性能测试、兼容性测试、边界测试）
- 模板：`./references/test-template.md`

#### 4.2 测试执行
- 自测：逐项执行测试用例，记录结果
- 协作：可调用测试技能辅助
- 云手机实测：mobile_use 在真机环境验证

#### 4.3 Bug 管理
- 产出：`bug-list.md`（严重/一般/建议 三级分类）
- 修复 → 回归 → 关闭

### Phase 5: 发布 (Publishing)

#### 5.1 发布清单
- 平台账号准备
- 隐私政策 / 用户协议
- 应用图标 / 截图 / 宣传图
- 版本号 / 更新日志

## 原型开发踩坑经验（2026-04-30 拖鞋大冒险）

### 触屏操控
1. **虚拟摇杆位置**：CSS position relative/absolute 要分清，子元素的 left/top 是相对父元素的，不能用 getBoundingClientRect 的视口坐标直接设
2. **摇杆常驻显示**：初始 opacity: 0.6，触摸时变亮，不要用 display:none 切换
3. **摇杆初始化时机**：getBoundingClientRect 在 DOM 渲染前返回0，用 setTimeout 延迟初始化
4. **攻击按钮变量命名**：`setupButton(btn, 'attackPressed', isHold=true)` 中 held 字段名要用 `inputKey.replace('Pressed', 'Held')` 生成，避免 `attackPressedHeld` vs `attackHeld` 不匹配
5. **触屏攻击统一入口**：不要按武器类型分触屏攻击判断，用 `attackPressed || attackHeld` 统一触发，performAttack 内部根据 currentWeapon 处理
6. **按钮事件加 stopPropagation**：防止触摸事件冒泡到 canvas

### 打击反馈
1. **震屏参数（手机）**：普通命中 intensity=3 duration=5，击杀 intensity=6 duration=10，衰减系数 0.8
2. **Hit Stop 在手机上体验差**：会导致卡顿+主角被怪物攻击，不要用。爽感靠震屏+浮动文字+击飞效果
3. **CD计算**：weaponCooldowns 直接存秒数（如0.35），每帧减 1/60，不要乘60再减

### 横屏适配
1. **手机强制横屏容易出问题**：Canvas 尺寸/渲染逻辑容易搞乱，竖屏优先更稳
2. 如需横屏，后续用 CSS transform rotate 整体旋转方案，不要改 Canvas 内部逻辑

### 原型阶段边界（重要！）
1. **原型使命是验证核心玩法，不是做产品** — 验证完操控、打击感、核心机制就停
2. **引擎有的能力不要在原型上造轮子** — 摄像机、物理、场景管理、UI系统，这些Cocos Creator现成
3. **手感微调留给引擎** — 弹跳物理、碰撞精度这些在引擎里调更高效
4. **判断标准**：要做的功能引擎有现成组件 → 立刻停原型转引擎开发

### 跳跃Z轴系统
1. **jumpForce必须是正数** — jumpZ正值=在空中，jumpVZ正值=向上
2. **重力方向**：`jumpVZ -= gravity`（让VZ逐渐减小，从正变负=下落）
3. **敌人Y轴不受重力** — 2.5D场景中Y表示纵深位置，`enemy.vy += gravity` 会导致敌人全部沉到下方
4. **受伤弹起**：用 `jumpVZ = 8` 不用 `vy = -8`

### 核弹/特殊技能
1. **核弹按钮独立于武器系统** — 按核弹直接triggerNuke()，不要改currentWeapon，否则影响后续攻击
