## ⛔ 铁律（违反=不可逆损害，必须遵守）

1. **删除→回收站**：永远不要用删除命令。把文件移到 `./回收站/<名称>-<日期>/` 目录下
2. **改文件前先备份**：修改任何文件之前，先把原文件复制一份到回收站，然后读取现有内容，禁止直接覆盖
3. **禁止越界操作服务器**：ECS服务器（39.103.203.162）的所有操作只能由ECS运维角色执行。开发/创作/运营角色禁止直接SSH或调用Remote Gateway，需要服务器操作时必须通过ECS运维角色
4. **权限最小化**：每个角色只用自己的token访问服务器。开发用dev_xl2026rust，运维用ops_xl2026ecs。严禁借用admin token或旧版xl2026cmd。有专用token不用=权限越界
5. **角色边界+交付门**：每个角色有明确的职责边界和交付门，不越界不跳步：
   - 系统开发者：开发七步门(①编译②测试③本地构建④CI构建⑤Tauri⑥文档)，不管部署
   - ECS运维：运维交付门(①容器运行②健康检查③服务可达④记录日志)，不管开发
   - 前端开发：前端交付门(①浏览器运行②三模式验证③响应式验证④推GitHub)，不管后端
6. **任务完成=执行+反哺**：智能体会失忆，没记下来的东西就真没了。产出结果≠任务完成，必须完成反哺检查门才算结束：
   - 更新项目INDEX.md的"最近变更"
   - 产出了项目特有知识？→ 写进项目 docs/knowledge/
   - 踩了坑？→ 记录，踩2次升级hot-rules
   - 发现跨项目通用经验？→ 写进角色 knowledge/
   - 发现跨角色通用经验？→ 写进共享知识/
   - **sub-agent返回结果时必须汇报反哺情况**，没汇报=任务未完成
7. **禁止扫描无关目录**：不要扫描其他项目目录、回收站、用户上传目录。全量扫描浪费资源，只读你需要的文件

## 📋 工作守则（违反不致命，但应该遵守）

- **接手任务先查导航版本**：读顶部版本号，变了只看摘要，没变跳过
- **先查再干**：开工前必须用act.sh搜索现有方案和前科，禁止直接手搓：
  ```bash
  bash scripts/act.sh "意图描述" . "角色/你的角色名"
  ```
  禁止直接grep找知识——act.sh有索引加速(1秒)还会自动展示你的角色前科（hot-rules和踩坑无条件置顶，不用特意去找）
- **推送用push.sh**：`bash scripts/push.sh "描述"`，禁止手动git push
- **经验跟着人走**：做完任务后，经验问自己"换个项目还有用吗？"有用的归角色knowledge，只对当前项目有用的归项目docs/knowledge
- **新工具要注册**：造了新工具必须注册，项目专用→项目INDEX，跨项目通用→角色RULES。只管用不管注册=体系不成长
- **同样的错不犯两次**：同一类型操作失败2次→必须换方案或上报，禁止第3次自作主张。错误分类按操作类型：git操作/编译/部署/API调用/文件操作/生图，同类型同根因=同一种错。踩2次→升hot-rules，踩3次→提工单给调教师
  - 高消耗操作举例：网络下载失败重试>2次、grep全量目录(应用act.sh索引)、等待外部服务>30s、同一API调用失败>2次
- **反哺必须可验证**：汇报"已反哺"时列出具体更新了哪些文件，主agent验收时抽查1-2项（如读INDEX.md验证"最近变更"是否有新条目）。口头说"已反哺"但文件没变=未完成
- **Ritual允许沉默**：标有[ritual]的定时任务，执行session判断无价值时输出NO_REPLY不算失败，不重试不告警。工单型日程才要求必须交付
- **膨胀内容开独立仓库**：会持续增长的内容（收藏、资源、源码）不入体系仓库，开独立私有仓库。体系只保留索引+使用指引

## 🔐 凭据管理

> 体系所有凭据（GitHub PAT、SSH Key、服务器Token、网盘凭据等）加密存储在 `共享知识/凭据/secrets.enc`，**明文绝不入 git**。

### 🔒 ECS/高危凭据加密访问（强制机制）
> **ECS Gateway Token已单独加密，与仓库凭据分离。这是硬性机制，不是建议。**

1. **仓库凭据密钥（MASTER_KEY）**：已存平台SECRET.md，角色可自行解密github/ssh/ctfile等凭据
2. **ECS服务器密码**：**不存在任何文件**，只能从主人对话获取。新会话=没密码=必须重新找主人要
3. 连接ECS必须先解密token：
   ```bash
   # 向主人获取ECS密码，然后：
   openssl enc -d -aes-256-cbc -pbkdf2 -iter 100000 -pass pass:<主人给的密码> -in 共享知识/凭据/ecs-tokens.enc -base64
   # 提取对应角色的token（ops/dev/readonly/admin）
   ```
4. **铁律**：
   - ECS密码用完即丢，禁止存入任何文件
   - 解密后的token禁止写回任何文件（pre-commit hook硬拦截）
   - 无密码=无法连接=必须找主人要，不允许跳过
   - 派发子任务时可在任务描述中传递ECS密码，但sub-agent不得持久化存储

### 获取凭据
1. **仓库凭据**：MASTER_KEY已存平台SECRET.md，直接解密：
   ```bash
   # 获取指定字段
   bash 共享知识/凭据/decrypt.sh $MASTER_KEY github.pat
   ```
2. **ECS凭据**：密码不存文件，找主人要后解密：
   ```bash
   openssl enc -d -aes-256-cbc -pbkdf2 -iter 100000 -pass pass:<密码> -in 共享知识/凭据/ecs-tokens.enc -base64
   ```
3. 常用字段路径速查（仓库凭据）：
   - `github.pat` — GitHub 读写PAT
   - `github.old_pat_readonly` — GitHub 只读PAT
   - `ssh.public_key` — SSH 公钥
   - `gitee.pat` — Gitee PAT
   - `ctfile.session_token` — 城通网盘 API token 🔒
   - `ctfile.password` — 城通网盘分享密码 🔒
4. ECS字段（ecs-tokens.enc内，需单独密码）：
   - `token_admin` — 网关 admin token 🔒
   - `token_ops` — 网关运维 token 🔒
   - `token_readonly` — 网关只读 token 🔒
   - `token_dev` — 网关开发 token 🔒

### 更新凭据
凭据变更时（新增/修改/删除），必须同步更新加密文件：
1. 解密到临时文件：`openssl enc -d -aes-256-cbc -pbkdf2 -iter 100000 -pass pass:$MASTER_KEY -in 共享知识/凭据/secrets.enc -base64 -out /tmp/secrets.json`
2. 编辑 `/tmp/secrets.json`，增删改对应字段
3. 重新加密：`openssl enc -aes-256-cbc -salt -pbkdf2 -iter 100000 -pass pass:$MASTER_KEY -in /tmp/secrets.json -out 共享知识/凭据/secrets.enc -base64`
4. 删除临时文件：`rm /tmp/secrets.json`
5. 提交 `secrets.enc` 到 git
6. 通知主对话凭据已更新

> ⚠️ 绝不在 git 中提交明文 secrets.json。绝不在聊天记录中明文输出凭据值。

### 我要修改这个知识体系本身

> 体系修改是最容易出纰漏的操作——改了A忘了同步B，模板过时、导航断链都是这么来的。

**必须先读：** `./共享知识/项目规范/体系维护手册.md`

**操作规程（按顺序来）：**

1. **想清楚：能不能用"注册"做到？** 新维度=注册追加，不重构现有结构。如果你在改架构本身，先确认没有注册的方式能做到
2. **查影响表：** 体系维护手册第二章有"修改影响表"，确认你改的东西会影响谁。**特别注意模板——改了架构规范必须同步检查模板**
3. **备份→修改→检查联动：** 改文件前先备份到回收站，然后改，改完立即检查联动文件是否需要同步
4. **改完自检（缺一不可）：**
   - 改的文件更新了"最后更新"日期
   - **如果改了本导航文件**：必须更新顶部版本号和变更摘要（版本格式：日期-vN，只保留最近5条摘要）
   - 入口.md已更新（新增目录/角色/文件时）
   - **模板已同步**（改了架构规范中INDEX/RULES的结构，必须同步更新templates/下的模板）
   - 现有项目/角色的文件按需同步（新增字段不需要立即补全，但模板必须是最新的）
   - 没有删减现有章节，只有追加或替换

## 反哺清单（任务完成后必须过一遍）

任务产出只是第一步，以下每项都必须完成才算任务结束：

### 反哺轨迹预填

任务完成时，**不要从空白模板开始反哺**。先回顾本次任务执行轨迹，自动生成预填内容：
- 我做了什么（关键步骤序列）
- 调了什么工具/API（工具调用摘要）
- 踩了什么坑（错误+解决方案）
- 发现什么通用经验（跨项目可用）
预填后只做确认/修正/补充，不从头写。**角色在反哺检查门汇报时必须包含"轨迹预填→确认/修正"过程**

### 必做（每次都要）

1. **更新项目INDEX.md**：把这次做了什么、改了什么写进"最近变更"
2. **项目特有事实**：如果产生了项目特有的知识（为什么这样配、这台服务器为什么这样设），写进项目的 docs/knowledge/
3. **经验归属判断**：写经验前先判断归属，问自己"这条知识换个项目还有用吗？"
   - 没用了 → 项目docs/knowledge/（"为什么这样配"）
   - 还能用 → 角色knowledge/（"怎么做更好"，能让角色变聪明的经验才值得归角色）
   - 换角色还能用 → 共享知识/（"大家都能用"）

### 按条件做

4. **踩了坑** → 同一个坑踩2次：升级为项目热规则，写进项目/规划/hot-rules.md
5. **通用经验**（换个项目还能用的"怎么做更好"）→ 写进对应角色的 knowledge/（归属判断见必做第3条）
6. **跨角色通用**（换个角色还能用的）→ 写进共享知识/踩坑记录/ 或 设计模式/
7. **5步以上的复杂流程** → 封装成技能，写到 ./技能/\<技能名>/
   - 判断：去掉判断力还能干吗？能干=不需要封装成技能，干不了=值得封装
