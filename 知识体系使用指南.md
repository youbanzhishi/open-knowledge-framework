# 知识体系使用指南

> 面向想使用本体系的人——从零搭建到日常使用，一份文档搞定

---

## 一、这个体系是什么

一套让多个AI智能体（Agent）协同工作的知识管理框架——**OpenClaw**。

核心使命：**打造最强的智能体成长框架，无论平台如何变迁，智能体永不过时。**

从扣子到飞书到 Hermes 到未来的人形机器人，只要智能体还在，这个体系就能跑。详细愿景和设计哲学见 `共享知识/蓝图.md`。

核心思想：

**知识不等于行为，行为不能靠自觉，要靠机制。机制用步骤来实现，步骤要读写闭环。**

- 每个智能体有明确角色边界，做自己该做的事
- 角色之间通过"交接台"协作，不靠口头转达
- 所有规则写在文件里，智能体每次启动自动加载
- 踩过的坑自动升级成规则，不再重犯

### 体系结构

```
open-knowledge-system/
├── 入口.md          ← 唯一入口，智能体读到就能开工
├── 基础设定/        ← 体系级规则（TOOLS.md/SOUL.md）
├── 角色/            ← 每个角色一个文件夹
│   ├── 系统开发者/   ← RULES.md + knowledge/ + hot-rules.md
│   ├── 本地运维/
│   ├── 测试工程师/
│   └── ...
├── 共享知识/        ← 角色间共享的文档
├── 交接台/          ← 角色协作中枢（bug单/工单/物料）
├── 项目/            ← 具体项目文件夹
├── scripts/         ← 自动化脚本
└── 热更新.md        ← 最近变更记录
```

## 二、怎么搭建

### 前提条件

- 一个AI平台（扣子/飞书/微信等），能创建智能体
- 一个Git仓库（GitHub + Gitee双推推荐）
- 智能体能读写文件、执行bash命令

### 搭建步骤

1. **Clone仓库**：把open-knowledge-system clone到智能体可访问的目录
2. **配置Git**：在智能体环境执行 `bash scripts/init.sh`
3. **告诉智能体入口**：第一次对话时说"先读 ./入口.md"
4. **完事了**——智能体读完入口就能干活

### 多智能体配置

如果你有多个智能体（比如一个写代码、一个做运维），每个智能体：

1. 读 `入口.md` 了解体系全局
2. 读自己角色的 `角色/{角色名}/RULES.md`
3. 执行 `bash scripts/act.sh "意图" shared "角色/{角色名}"` 做例行检查

## 三、怎么和智能体说话

### 万能话术（99%场景都够用）

```
[你要干什么]。先读 ./入口.md
```

**关键：第一句必须说你要干什么**，不然智能体不知道走哪条路。

### 各场景详细话术

#### 新建项目

```
我要做一个 [项目描述] 的项目。先读 ./入口.md
```

示例：
- "我要做一个开源DAW项目。先读 ./入口.md"
- "我要做一个表情包IP。先读 ./入口.md"

#### 已有项目开发

```
帮我 [具体任务]。先读 ./入口.md
```

示例：
- "帮我开发音频引擎的实时效果器模块。先读 ./入口.md"
- "帮我把登录功能加上。先读 ./入口.md"

#### 运维任务

```
帮我 [运维任务]。先读 ./入口.md
```

示例：
- "帮我部署OpenDAW到ECS。先读 ./入口.md"
- "服务器端口被占了帮我看看。先读 ./入口.md"

#### 项目入体系

```
帮我把 [项目名] 加入知识体系。先读 ./入口.md
```

示例：
- "帮我把AudioFX加入体系。先读 ./入口.md"

#### 修改体系本身

```
我要优化知识体系，[改什么]。先读 ./入口.md
```

示例：
- "我要优化知识体系，加一个测试工程师角色。先读 ./入口.md"
- "我要优化知识体系，把发布流水线写进规则。先读 ./入口.md"

#### 通知同步

```
通知所有会话更新规范
```

智能体会读热更新.md，把最新变更同步到各角色。

#### 随便聊聊

直接说，不用加任何前缀。

## 四、常见场景操作手册

### 场景1：开发者完成功能→测试→部署上线

完整发布流水线：

```
你（对开发者说）：
"XX功能开发完了，提交测试。先读 ./入口.md"

开发者会：
1. 走六步门交付流程
2. 提测试物料到交接台

你（对测试工程师说）：
"测试XX项目的新功能。先读 ./入口.md"

测试工程师会：
1. 领测试物料
2. 发现Bug → 提工单给开发者
3. 测试通过 → 提工单给本地运维

你（对本地运维说）：
"部署XX项目测试环境。先读 ./入口.md"

本地运维会：
1. 部署到测试环境
2. 通知你验收

你验收通过后说：
"验收通过，正式部署上线。先读 ./入口.md"

本地运维会：
1. 提工单给ECS运维
2. ECS运维正式部署上线
```

### 场景2：发现体系问题→修复

```
你（对任何角色说）：
"我发现[问题描述]，这个需要修。先读 ./入口.md"

角色会：
1. 判断是不是自己职责范围
2. 自己能解决 → 自己修
3. 超出职责 → 提bug单/工单到交接台

你（对AI调教师说）：
"体系需要优化，[具体问题]。先读 ./入口.md"

调教师会：
1. 分析问题根因
2. 修改体系文件
3. 写优化追踪(OPT) + 热更新
4. 推送代码
```

### 场景3：新增角色

```
你（对AI调教师说）：
"我要优化知识体系，新增一个[角色名]角色，负责[职责描述]。先读 ./入口.md"

调教师会：
1. 参考 templates/RULES.template.md 创建角色RULES
2. 更新角色边界总览
3. 更新交接台README
4. 写优化追踪 + 热更新
5. 推送代码
```

### 场景4：让角色做跨角色的事

**不要直接让角色做别人职责的事**。用交接台：

```
你（对开发者说）：
"XX项目的deploy.md需要更新。先读 ./入口.md"

开发者会写好deploy.md → 提物料交接单到交接台

你（对运维说）：
"去看看交接台有没有待领的单子。先读 ./入口.md"

运维会领单 → 按deploy.md部署
```

或者更简单——直接让角色自己提工单：

```
你（对测试工程师说）：
"测试XX项目。先读 ./入口.md"

测试工程师发现Bug → 自动提工单给开发者
测试通过 → 自动提工单给本地运维
```

### 场景5：紧急问题（服务器挂了/数据丢失等）

```
紧急！[问题描述]。先读 ./入口.md
```

加"紧急！"前缀，角色会优先处理。

## 五、体系核心机制速查

### 五步门（每个角色每次任务必走）

| 步骤 | 做什么 | 你需要做什么 |
|------|--------|-------------|
| ①查后定方案 | 搜知识+读交接台+确认环境 | 说清楚要做什么 |
| ②执行 | 按方案干活，遇阻塞提工单 | 不用管，角色自己处理 |
| ③测试验证 | 验证产出质量 | 可能需要你审核（比如先审图） |
| ④反哺沉淀 | 记经验/写文档/升级规则 | 不用管，角色自动做 |
| ⑤交付闭环 | 推代码/交报告/提物料 | 验收结果 |

### 交接台（角色协作中枢）

| 类型 | 谁提 | 提给谁 | 命令 |
|------|------|--------|------|
| Bug单 | 任何角色 | AI调教师 | `bash scripts/handover.sh bug "标题" "提交者"` |
| 工单 | 任何角色 | 指定角色 | `bash scripts/handover.sh work "标题" "提交者" "接收角色"` |
| 物料 | 交付方 | 接收方 | `bash scripts/handover.sh material "编号" "交付方" "接收方" "内容" "路径"` |
| 热更新 | AI调教师 | 所有角色 | `bash scripts/handover.sh hotupdate "OPT编号" "摘要" "影响" "改了什么"` |

### 铁律（所有角色不可违反）

1. **删除→回收站**：不直接删文件，移到回收站
2. **改前备份**：改文件前先备份原版本
3. **先读后动**：动手前先读相关文件，不凭印象
4. **重试熔断**：任何操作失败2次→上报，不自行重试
5. **生图count=1**：逐张生成，不批量生图
6. **职责内自己扛**：自己能解决的不提工单，解决不了是经验不够
7. **覆盖=犯罪**：不覆盖他人文件，改前先读

## 六、常见问题

### Q：智能体不按规则办事怎么办？

说"你读一下你的RULES.md"，让他重新加载规则。如果规则本身有问题，对AI调教师说"体系需要优化，[问题描述]"。

### Q：多个角色要协同做一个任务怎么办？

不要自己协调。告诉其中一个角色"先读 ./入口.md"，他会通过交接台自动和其他角色协作。

### Q：怎么知道体系最近改了什么？

看 `热更新.md`，每次体系变更都会记录。

### Q：怎么查看项目进展？

对角色说"看一下XX项目的进展。先读 ./入口.md"，他会读项目INDEX.md给你汇报。

### Q：智能体卡住了怎么办？

直接说"停"或"停下"，他会停止当前操作。然后说"你遇到了什么问题"，让他汇报阻塞点。

### Q：怎么防止智能体浪费token/积分？

体系已有机制：
- 重试熔断：失败2次自动停下
- 生图count=1：防止批量生成
- 先审后批量：小批量确认再继续
- 职责内自己扛：不因小事提工单

如果你发现智能体在空转，直接说"停，汇报当前状态"。

## 七、给你的建议

1. **让智能体自己跑**：体系设计的就是你只说需求，智能体自己走流程。不要微观管理
2. **验收把关**：五步门③④⑤需要你确认的环节才介入，其他时候不用管
3. **踩坑让体系记住**：发现问题就说，调教师会升级成规则。说一次，以后所有角色自动遵守
4. **看热更新**：每隔几天看一下热更新.md，了解体系自动优化了什么

**协作标语：协作不是单向获取，而是你来我往，相互成长**
