# 派发模板规范

> 最后更新：2026-05-31
> 解决：任务派发时每次手写五段式task，主会话token消耗大，模板无版本同步

## 设计原则

1. **模板跟着角色走**：每个角色目录下有 `templates/dispatch.md`，是该角色的派发模板
2. **变量填充，不手写**：主agent派发时只需读取模板+填充变量，不每次手写
3. **版本同步**：模板随角色RULES版本同步升级，入口版本迁移清单覆盖
4. **三种模板**：串行/并行/PM审查，对应auto-collab三种模式的子agent task

## 模板格式

每个模板使用 `{变量名}` 标记需要填充的位置：

| 变量 | 来源 | 说明 |
|------|------|------|
| {角色名} | 工单接收角色 | 角色目录名 |
| {工单路径} | 扫描结果 | 交接台/工单/{文件名}.md |
| {项目名} | 工单归属项目 | 并行模式时限定范围 |
| {铁律段} | 入口.md铁律 | 自动注入当前版本铁律 |
| {热规则段} | 项目hot-rules | 自动注入项目热规则 |
| {轨迹预填段} | 主agent在任务描述末尾追加 | 角色完成时的执行轨迹预填模板 |

## 串行模板（默认）

位置：`角色/{角色名}/templates/dispatch.md`

```markdown
你是{角色名}，请处理工单。

## 执行步骤
1. cd open-knowledge-system && bash scripts/act.sh "处理工单" . "角色/{角色名}"
2. 读取工单：{工单路径}
3. 按五步门执行
4. 铁律：删除→回收站|覆盖=犯罪|先读后动|失败2次停
5. 反哺：预填执行轨迹→确认/修正→更新INDEX/knowledge/hot-rules
6. 汇报：完成/失败+原因
```

## 并行模板

位置：同上，文件名 `dispatch-parallel.md`

```markdown
你是{角色名}，处理{项目名}项目的工单。

## 执行步骤
1. cd open-knowledge-system && bash scripts/act.sh "处理工单" . "角色/{角色名}"
2. 读取工单：{工单路径}
3. 按五步门执行
4. ⚠️ 并行约束：只改{项目名}文件|完成后commit不push|共享文件只读
5. 铁律：删除→回收站|覆盖=犯罪|先读后动|失败2次停
6. 反哺：预填执行轨迹→确认/修正→更新项目INDEX/knowledge/hot-rules
7. 汇报：完成/失败+原因+修改文件列表
```

## PM审查模板

位置：`角色/产品经理/templates/dispatch-pm.md`

```markdown
你是产品经理，执行蓝图审查。

## 执行步骤
1. cd open-knowledge-system && bash scripts/act.sh "蓝图审查" . "角色/产品经理"
2. 读取 共享知识/蓝图.md → 提取项目终极目标
3. 读取各项目INDEX.md → 对比进度vs目标
4. 有差距→提工单；无差距→标记达标
5. 产出审查报告
6. 汇报：是否需要继续循环
```

## 派发流程

1. 主agent扫描工单 → 确定角色和模式
2. 读取 `角色/{角色名}/templates/dispatch[-parallel|-pm].md`
3. 填充变量：{角色名}、{工单路径}、{项目名}
4. spawn子agent，task为填充后的模板内容
5. 省token：模板本身已在文件中，不需要在task中重复五段式

## 与架构规范的关系

派发模板是知识流动的**第六条路径**的载体（但不是新路径，是现有路径的效率优化）。

模板注册到架构规范的维度注册表：
```
维度名      | 实体目录        | 规则文件    | 入口文件   | 反哺方式    | 派发模板
-----------|----------------|------------|-----------|----------|----------
项目        | 项目/<名>/  | INDEX.md   | INDEX.md  | 更新INDEX  | —
角色        | 角色/<名>/      | RULES.md   | RULES.md  | 更新knowledge | templates/dispatch*.md
```

## 通道选择规则（借鉴Syll多通道统一执行回路）

> 主agent派发任务时，根据任务性质选择最合适的执行通道。显式写在派发模板的`{通道建议}`变量中。

### 通道优先级

| 任务性质 | 首选通道 | 备选通道 | 判断标准 |
|----------|---------|---------|---------|
| 读取已知URL内容 | fetch_web | 云电脑浏览器 | URL明确且无需交互→fetch_web；需要登录/交互→云电脑 |
| 网页交互操作 | 云电脑(agent_browser) | 云手机 | 桌面端网页→云电脑；移动端专属→云手机 |
| 移动端App操作 | 云手机(mobile_use) | — | 抖音/小红书/微博等App→云手机 |
| 文件读写/代码执行 | 直接bash | — | 无需浏览器→直接执行 |
| 长时间搜索/调研 | sessions_spawn(deep_research) | 云电脑 | 多源信息收集→deep_research；单一网页→fetch_web |

### 模板变量

新增 `{通道建议}` 变量，主agent派发时填充：

| 变量 | 来源 | 说明 |
|------|------|------|
| {通道建议} | 主agent判断 | 任务的推荐执行通道+备选通道 |

### 通道建议格式

在派发模板的执行步骤前加一行：

```markdown
## 执行通道：{通道建议}
```

示例：
- `执行通道：fetch_web（读取网页内容，无需交互）`
- `执行通道：云电脑+agent_browser（需登录+表单填写）`
- `执行通道：云手机（移动端App操作）`
- `执行通道：bash+fetch_web（代码执行+网页读取）`
