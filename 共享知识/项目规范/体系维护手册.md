# 知识体系维护手册

> 最后更新：2026-05-10
> 给任何session看的——怎么在这个体系上继续建设，不改坏、不遗漏
> 如果你是一个新session，接到"优化/修改/补充知识体系"的任务，先读这个

---

## 一、这个体系的设计意图

### 核心哲学

**新功能=注册扩展，架构本身永远不需要改。**

这套哲学来自OpenDAW的Extension Registry，三个体系同构：
- 代码架构：新插件=注册扩展，宿主不改
- 知识体系：新维度=注册路径，结构不改
- 平台适配：新平台=注册适配层，知识不动

**如果你想改架构，先问自己：能不能通过"注册"来做到？**

### 五层为什么要分五层

| 层 | 为什么在这层 | 为什么不往下放 | 为什么不往上提 |
|---|---|---|---|
| T0 SOUL | 身份认知，最底层 | 不能放T1，因为"我是谁"比"我怎么工作"更根本 | — |
| T1 TOOLS | 工作方式，跨项目通用 | 不能放T2，因为操作铁律比共享规范更底层 | 不能提T0，因为工具≠身份 |
| T2 宪法 | 所有项目共享的规则 | 不能放T3，因为项目不该各自定义"删除→回收站" | 不能提T1，因为宪法是约定不是铁律 |
| T3 项目INDEX+热规则 | 单个项目的特有知识 | 不能放T4，因为INDEX是知识索引不是操作手册 | 不能提T2，因为项目特例不是通用规则 |
| T4 技能 | 操作手册（5步以上复杂流程） | — | 不能提T3，因为技能是跨项目可分享的 |

**判断原则：这条规则管多广？** 越广越往上，越窄越往下。

### 知识为什么要流动

静态知识会腐烂。4条流动路径的设计：

```
项目踩坑 → 项目热规则（防犯，自动化程度最高）
项目经验 → 角色knowledge（跨项目复用）
角色经验 → 共享知识（跨角色复用）
角色能力 → 技能导出（跨人复用）
```

**方向：从具体到通用，从私有到可分享。** 反向流动是禁止的——不要把通用规则搬进项目热规则。

## 二、文件依赖链（改一个会影响哪些）

```
SOUL.md
  └→ TOOLS.md（引用SOUL中的身份来决定工作方式）
      └→ 共享知识/项目规范/（TOOLS铁律是宪法的一部分来源）
          ├→ 各项目INDEX.md（引用宪法+声明本地偏差）
          ├→ 各项目hot-rules.md（热规则规范定义了格式和升级条件）
          ├→ 各角色RULES.md（角色宪法，引用共享规范但不复制）
          └→ 知识体系架构规范.md（体系自身的架构）
              ├→ templates/INDEX.template.md（项目模板，架构规范改结构时必须同步）
              ├→ templates/RULES.template.md（角色模板，架构规范改结构时必须同步）
              └→ 任务派发模板（三段式：铁律+热规则+反哺）
```

### 修改影响表

| 你想改的 | 影响范围 | 必须同步更新 | 注意事项 |
|----------|---------|-------------|---------|
| SOUL.md | 全局 | TOOLS.md可能需要对应更新 | 身份变更影响所有行为 |
| TOOLS.md铁律 | 所有任务派发 | 任务派发模板中的铁律段 | 新增铁律要同步到架构规范的派发模板 |
| 目录结构规范 | 所有项目 | 各项目INDEX.md的自愈命令 | 不要删已有目录，只加新目录 |
| 协作规范 | 所有项目+角色 | 无需同步（引用不复制） | 改完宪法文件更新日期即可 |
| 热规则规范 | 所有项目 | 无需同步 | 格式变更要向后兼容 |
| 架构规范 | 体系自身+模板+入口 | **templates/下所有模板**（INDEX/RULES/等）、入口.md | 改了INDEX/RULES的结构定义，必须同步更新templates/下的模板。否则新建项目/角色用的是旧模板，体系改了等于白改 |
| templates/下任何模板 | 使用该模板的所有项目/角色 | 无需同步（模板是创建时用的） | 已有项目/角色不需要立即对齐，但模板必须是最新的 |
| 项目INDEX.md | 仅该项目 | 无需同步 | 修改后更新"最后更新"日期 |
| 项目hot-rules.md | 仅该项目 | 无需同步 | 升级/降级要更新优先级标记 |
| 角色RULES.md | 该角色的所有任务 | 无需同步 | 角色是独立的，不影响其他角色 |
| 角色knowledge/ | 该角色的能力 | 无需同步 | 经验追加不修改已有文档 |
| 入口.md | 入口 | 无需同步 | 新增目录时要更新入口 |

## 三、修改/补充的操作规范

### 原则

1. **追加不改写**：新内容追加到文件末尾或对应章节，不改已有内容
2. **注册不重构**：新增维度/路径/角色，通过注册追加，不改现有结构
3. **向后兼容**：修改规范时，已有项目不需要立即对齐，通过版本追踪自然同步
4. **更新日期**：任何修改都要更新文件的"最后更新"日期

### 具体操作

#### 新增一个项目类型变体（比如"数据类"项目）
1. 在目录结构规范.md的"按项目类型变体"章节追加新类型
2. 新增自愈命令模板
3. 更新目录结构规范的"最后更新"日期
4. 不需要通知已有项目（版本追踪法会自然同步）

#### 新增一个角色（比如"内容运营"）
1. **先检查是否已存在**：`ls ./角色/` 看目录列表
2. **如果已存在**：不需要新建，复用已有的RULES+knowledge+hot-rules，经验追加到knowledge/
3. **如果不存在**：创建 `./角色/<角色名>/` 目录
4. 写 RULES.md + hot-rules.md
5. 创建 knowledge/ + templates/ + log/ 子目录
6. 在入口.md追加角色条目
7. 如果该角色有跨项目经验，注册知识流动路径（在架构规范追加）

#### 新增一条知识流动路径
1. 在架构规范的"知识流动全景图"追加新路径
2. 在"反哺清单"追加对应行
3. 更新架构规范的"最后更新"日期
4. 这就是"注册路径"——现有路径不受影响

#### 修改现有规范
1. 先确认影响范围（查上面的修改影响表）
2. 用edit_file的replace模式修改（不要用write_file覆盖）
3. 更新"最后更新"日期
4. 如果影响任务派发模板，同步更新架构规范中的模板

#### 补充知识文档
1. 判断知识类型：项目特有？跨项目？跨角色？跨人？
2. 按类型写入对应位置（查入口.md）
3. 如果是踩坑经验，判断是否需要升级为热规则
4. **文档双写检查**：如果产出了操作文档，是否需要同时产出知识沉淀？如果是，在共享知识/设计模式/下创建对应的决策文档
5. **交叉引用检查**：操作文档顶部是否有📖引用指向知识沉淀？知识沉淀是否有🔧指向操作文档？

#### 文档双写规范（2026-05-11确立）

> 核心原则：操作文档给执行者看（怎么做），知识沉淀给决策者看（为什么这么做）。两者互不替代，必须同时产出。

**什么时候需要双写**：
- 产出了有决策过程的架构/方案文档时
- 方案对比选择后，操作文档只记录最终方案，决策过程需另存

**双写位置**：
- 操作文档 → 角色knowledge/（执行者照着走）
- 知识沉淀 → 共享知识/设计模式/（决策者回顾思考）

**交叉引用格式**：
- 操作文档顶部：`**📖 想了解决策背景和方案取舍？** → [路径](相对路径)`
- 知识沉淀顶部：`**🔧 运维操作文档？** → [路径](相对路径)`

## 四、禁止事项

1. ❌ **不要用write_file覆盖已有规范文件**——用edit_file追加/替换
2. ❌ **不要删减现有章节**——只能追加新章节或替换段落
3. ❌ **不要把通用规则搬进项目**——项目只引用不复制
4. ❌ **不要把项目特例搬进宪法**——特例属于INDEX.md的本地偏差段
5. ❌ **不要改变五层体系的层次关系**——可以新增层间路径，不重新分层
6. ❌ **不要重构目录命名**——已有项目的目录名是约定，改名=破坏所有引用

## 六、知识体系同步

### 什么时候同步

- 每次知识体系有变更后（新增/修改/删除文件）
- 定时任务定期执行
- 手动触发时

### 同步什么

- 整个知识体系目录推送到Git仓库
- 使用 `基础设定/scripts/sync.sh` 执行

### 同步脚本位置

`./基础设定/scripts/sync.sh`

### 同步步骤

1. 进入知识体系同步目录
2. `git add -A` 暂存所有变更
3. `git commit` 提交
4. `GIT_SSH_COMMAND="ssh -i /root/.ssh/id_ed25519_openks -o StrictHostKeyChecking=no" git push origin main` 推送

### 前提条件

- 已配置SSH密钥：/root/.ssh/id_ed25519_openks
- Git远程仓库可访问

### 禁止事项

- ❌ 不要修改已推送的内容
- ❌ 不要强制推送（--force）
- ❌ 冲突时不要自动合并

## 七、规范热更新（广播机制）

### 问题

规范改了，正在跑的agent还在用旧版。智能体平台没有"广播给所有session"的API，真正的推模式做不到。

### 方案：三层覆盖

| 层 | 覆盖对象 | 怎么做 | 时机 |
|----|---------|--------|------|
| 1. 任务描述 | 一次性任务 | 派任务时加一句"先读入口.md" | 每次派任务 |
| 2. 定时任务 | 长期运行的cron | 修改cron的任务描述，加入"开工前读入口.md" | 改规范后手动更新一次 |
| 3. 心跳检查 | 持续运行的heartbeat | 在heartbeat.md加一条"查入口.md版本号，变了通知用户" | 改规范后手动加一次 |

### 关键认知

- **一次性agent不需要广播**：它干完就没了，下次来的新session自然会读最新版
- **需要广播的是长期agent**：定时任务和心跳，它们反复运行，会一直用旧版直到你更新它们的任务描述
- **更新方式**：改完规范后，去修改相关定时任务/心跳的描述，加入新版本信息。这比让它们自己发现要可靠

### 操作步骤

规范改完后：

1. 更新入口.md的版本号和变更摘要
2. git commit + push
3. 检查正在跑的定时任务和心跳——如果这次改动影响它们的工作方式，更新它们的任务描述

第3步是手动操作，但通常规范改动不会影响所有定时任务，只需要更新相关的几个就行。
