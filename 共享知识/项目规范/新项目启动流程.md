# 新项目启动流程

> 最后更新：2026-05-09
> 任何新会话开始新项目时，按此流程操作，确保0成本接入协作体系。

## ⛔ 第一件事：文件安全铁律（比读宪法还优先）

**在你做任何文件操作之前，必须遵守：**

```
1. 删除→mv到./回收站/<项目名>-<日期>/，绝对禁止rm
2. 覆盖=犯罪，先读后动
3. 整目录清理先移回收站
4. mv同名目录加后缀
```

这不是建议，是铁律。违反=任务失败，无例外。

## 第二件事：读宪法

很多新手session上来就建文件、写代码，然后目录乱七八糟、规范不统一、跟其他项目无法协作。

**正确顺序**：先读规范 → 再建结构 → 再写INDEX → 最后才写代码

⚠️ **目录自愈铁律**：定时清理可能删除空目录。任何session操作项目前，必须先运行INDEX.md中的目录自愈命令，确保结构完整。这不是建议，是铁律。

## 启动5步法

### 第1步：读宪法（最重要）

```
读取 ./共享知识/项目规范/ 下所有文件
- 目录结构规范.md → 知道目录该怎么建（选哪种项目变体）
- 协作规范.md → 知道怎么跟其他项目协作
- 新项目启动流程.md → 你正在读的这个
```

**这一步决定了后续一切是否规范。跳过=白费力气。**

### 第2步：确定项目类型+建目录结构

先判断项目类型，再按对应变体建目录：

| 项目类型 | src/内容 | 特殊目录 | 典型例子 |
|----------|----------|----------|----------|
| 🔨 开发 | 代码 | tests/ + justfile + .github/ | OpenLink, OpenDAW |
| ✍️ 内容 | 草稿+大纲 | assets/templates/ | 小说, 课程 |
| 📊 运营 | 素材+文案 | feedback/ | 引流, 社媒 |
| 🎵 混合 | 代码+内容 | 全部 | DAW, 音乐制作 |

建目录命令（开发类）：
```bash
mkdir -p ./项目/<项目名>/{
  规划,src,tests/{integration,fixtures},
  docs/{dev-log,knowledge,adr},
  assets/{images,templates,data},
  output,config,scripts,feedback,
  .github/workflows
}
# 决策记录模板
cp ./templates/adr-template.md ./项目/<项目名>/docs/adr/000-template.md
```

建目录命令（内容/运营类）：
```bash
mkdir -p ./项目/<项目名>/{
  规划,src/{drafts,outlines},
  docs/{dev-log,knowledge,decisions},
  assets/{images,templates,data},
  output,config,scripts,feedback
}
# 决策记录模板
cp ./templates/dr-template.md ./项目/<项目名>/docs/decisions/000-template.md
```

### 第3步：写核心文件

按优先级：
1. **INDEX.md** — 按templates/INDEX.template.md填写，特别注意：关联角色+可用工具+反哺指南+**领域术语段**
2. **README.md** — 一句话定位+快速上手
3. **规划/roadmap.md** — 项目往哪走（Phase/里程碑/优先级）
4. **决策记录目录** — 按类型建：
   - 开发类：`docs/adr/`（编号ADR-{N}），放模板`000-template.md`
   - 内容/运营类：`docs/decisions/`（编号DR-{N}），放模板`000-template.md`
   - 规范见`共享知识/项目规范/决策记录规范.md`
5. **规划/roadmap.md** — 项目往哪走（Phase/里程碑/优先级）
6. **规划/decisions.md** — 关键决策索引（引用ADR/DR编号，不重复内容）
7. **docs/蓝图与设计文档.md** — 完整设计展开（架构/数据模型/API/流程），引用ADR不重复

**三份规划文档的关系（互补不冲突）**：
| 文档 | 回答什么 | 例子 |
|------|---------|------|
| decisions.md | 为什么选A不选B | ADR-001: 为什么选Rust重写 |
| 蓝图与设计文档 | 怎么做（完整设计） | 架构图+数据模型定义+API端点+流程图 |
| roadmap.md | 什么时候做什么 | Phase 2: 文本摄入+关键词搜索 |

蓝图引用decisions.md的ADR编号（如"详见ADR-001"），不重复决策理由；蓝图定义设计，roadmap引用蓝图安排时间。

### 第3.5步：写文档体系（开发类项目必做）

开发类项目（Rust/Python/Go等）必须标配以下文档：

| 文档 | 路径 | 说明 |
|------|------|------|
| 用户文档 | docs/user-guide.md | 完整使用指南（安装/配置/功能/FAQ） |
| Agent指南 | docs/agent-guide.md | AI智能体内置指南（项目定位/API速查/工作流/接入步骤） |
| Agent发现 | GET /.well-known/agent.json | 智能体自发现端点（capabilities/API/配置/links） |
| 部署指南 | docs/deployment.md | Docker+二进制+源码+生产环境 |

**Agent发现协议规范**：
- 端点：`GET /.well-known/agent.json`
- 返回JSON，包含：schema_version、name、description、version、base_url、auth、capabilities（API端点列表）、links（文档/源码/健康检查）
- 智能体请求此URL即可了解项目全部能力，不需要人教
- 路由注册时注意：`/.well-known/agent.json` 必须在通配路由（如 `/:code`）之前，避免被错误匹配

### 第3.6步：文档双写+交叉引用（所有项目必做）

**文档分两类，必须同时产出，互不替代：**

| 类型 | 位置 | 受众 | 风格 | 例子 |
|------|------|------|------|------|
| 操作文档 | 角色knowledge/ | 执行者（运维/开发） | 简洁可执行，命令可复制 | 云电脑与ECS互补架构.md |
| 知识沉淀 | 共享知识/设计模式/ | 决策者（主人/自己） | 详细记录决策过程、方案取舍、踩坑经验 | 云电脑与ECS互补架构决策.md |

**交叉引用规范**：
- 操作文档顶部必须加 `📖` 引用块，指向相关知识沉淀文档
- 知识沉淀文档中引用操作文档时用 `🔧` 标记
- 格式：`**📖 想了解XXX的决策背景？** → [文档名](相对路径)`

**交叉引用示例**：
```markdown
# 操作文档顶部
**📖 想了解决策背景和方案取舍？** → [共享知识/设计模式/XXX决策.md](../../共享知识/设计模式/XXX决策.md)
**📦 需要操作YYY？** → [YYY操作指南.md](YYY操作指南.md)

# 知识沉淀文档顶部
**🔧 运维操作文档？** → [角色/ECS运维/knowledge/XXX.md](../../角色/ECS运维/knowledge/XXX.md)
```

**同步更新铁律**：
- 修改任何一份时，必须同步更新另一份，不允许只改一边
- 操作文档改了步骤→知识沉淀补决策变更理由
- 知识沉淀改了方案→操作文档更新对应命令/配置
- 新增内容时双写，修改内容时双改，缺一不可

**为什么必须双写**：
- 操作文档只告诉你"怎么做"，不告诉你"为什么这么做"
- 知识沉淀记录完整决策链：背景→方案对比→取舍→教训
- 后来回顾时能还原当时的思考，不丢失上下文
- 运维看完操作文档想深入，点链接直达知识沉淀

### 第4步：在入口注册项目

编辑 `./入口.md`，在项目清单里追加新项目条目。

### 第4.5步：接入OpenDev生态门户（Open系列项目必做）

Open系列项目必须在生态门户中注册：
1. 在 `opendev-portal` 仓库中添加产品页（`src/pages/<项目名>.astro`）
2. 在首页项目矩阵中添加项目卡片
3. 在 `public/.well-known/agent.json` 生态清单中追加项目条目
4. 导航栏添加新项目链接

**原则**：新项目接入零成本——加一个页面+一条路由即可。

### 第5步：才开始干活

前面的都搞好了，代码/内容才不会乱放、文档才不会缺、其他session接手才不会懵。

### 第5.5步：Phase完成必须更新进度（断点续推的生命线）

每个Phase完成commit时，**必须同步更新**以下文件：

| 文件 | 更新内容 | 为什么 |
|------|---------|--------|
| 规划/roadmap.md | 对应Phase条目打勾 `- [x]` | 续推agent读roadmap就知道做到哪了，不用读代码 |
| INDEX.md | 更新进度+最近变更 | 任何人接手项目先看INDEX |
| docs/dev-log/ | 记录踩坑和关键决策 | 后续agent避免重复踩坑 |

**这是断点续推的生命线**：项目写了一万行代码后agent挂了，新agent不需要读一万行代码，只需要读roadmap（几十行）就知道全部进度。不更新roadmap=续推agent瞎子摸象。

### 第5.6步：开发完成≠交付完成——必须走完七步门

**代码push了不等于开发完成了。** 开发交付必须走完七步门：

| 步 | 门 | 做什么 | 谁管 |
|---|---|---|---|
| ① | 编译0错误 | cargo check | 开发 |
| ② | 测试0失败 | cargo test | 开发 |
| ③ | 本地构建验证 | cargo build --release + smoke test | 开发 |
| ④ | CI全平台构建 | GitHub Actions 4平台构建+推Release | 开发 |
| ⑤ | 桌面应用构建 | Tauri构建(仅GUI项目) | 开发 |
| ⑥ | 文档同步更新 | README/CHANGELOG/知识体系/运维文档 | 开发 |

**六步全过才算开发完成。开发不管运维的事，部署由运维角色负责。**

🔴 **体系执行防bug**：知识≠行为，写了不等于做了。必须加主动约束：
- **派发前必检**：派发开发类任务前，主agent必须读系统开发者RULES.md铁律部分，确认任务范围不越界
- **验收时对照七步门**：sub-agent返回"代码写完了"，主agent必须逐项对照七步门验收，不能凭感觉就当完成
- **反面教训**：五步门写进RULES.md但派部署任务时完全没读，直接跳到ECS部署，越界+漏步骤

## 凭据检查（建仓库前必做）

> **先查现有凭据，不要重复申请密钥。**

新建项目/仓库需要凭据时：
1. 查 `共享知识/凭据/secrets.enc` 是否已有可复用的凭据（如 GitHub PAT 读写权限已覆盖所有仓库）
2. 如果已有 → 在项目 INDEX.md「凭据」段填入字段路径即可
3. 如果需要新凭据 → 按凭据更新流程添加到 `secrets.enc`（见 `共享知识/凭据/README.md`），然后在 INDEX 和关联角色 RULES 中登记字段路径

常见场景：
- **新建 GitHub 仓库** → 不需要新 PAT，现有 `github.pat` 已有读写权限
- **需要服务器访问** → 查 `ecs_gateway.*` 现有角色 token，按需复用
- **需要新服务 API Key** → 这是需要新增的，按更新流程加入 `secrets.enc`

## 多平台三推配置（建仓库后必做）

> **新建GitHub仓库后，必须同时配置Gitee+Gitea三推（容灾备份）。一个平台挂了，代码还在另外两个平台。**

配置流程：
1. **GitHub建仓库** — 正常创建，可见性按需设置
2. **Gitee建私有仓库** — ⚠️ 必须设为私有，绝不公开
3. **Gitea建私有仓库** — ⚠️ 必须设为私有，绝不公开
4. **运行三推配置脚本**：
   ```bash
   MASTER_KEY=xxx bash scripts/multi-remote-setup.sh <本地路径> <仓库名>
   ```
5. **验证三推**：`git push` 后检查 GitHub、Gitee 和 Gitea 三个平台都有最新提交

> 📖 原理和详细文档见 入口.md 的「多平台容灾备份」段，或 共享知识/凭据/README.md 的多平台容灾备份段。

## 已有项目入体系流程

> 散落的项目文件怎么收拢，怎么搬才不会炸

### ⚠️ 前置检查：关联项目

在执行3步法之前，**必须先检查关联项目**：

```
1. 问用户："这个项目是否有关联项目？比如同一代码库的其他仓库、同一产品线的其他模块？"
2. 如果有关联项目 → 确认是否一起入体系，在INDEX.md中声明联盟关系
3. 不要遗漏：用户只提了一个项目名，不代表只有一个项目
```

**典型场景**：用户说"OpenDAW入体系"，但OpenDAW和AudioFX是独立Git仓库+知识强关联的联盟项目。只处理一个会遗漏关联关系。

### 原则

**只补不搬，逐步对齐。**

项目入体系 ≠ 一次性大搬迁。现有文件的路径可能被代码/脚本/CI引用，乱搬会破坏运行。

### 3步法

#### Step 1：建骨架（零风险）

按目录结构规范建好目录骨架 + INDEX.md + hot-rules.md。这一步**不碰任何现有文件**。

```bash
# 1. 目录自愈
mkdir -p ./回收站 && mkdir -p ./项目/<项目名>/{
  规划,src,tests,docs/{dev-log,knowledge},
  assets/{images,templates,data},
  output,config,scripts,feedback
}

# 2. 写 INDEX.md（从现有项目提炼，不是凭空编）
# 3. 写 规划/hot-rules.md（从踩坑经验提炼）
# 4. 写 规划/roadmap.md（从项目现状总结）
```

#### Step 2：增量归位（新文件走新规矩）

从现在起，**新产生的文件按规范放**。旧的先不动。

```
新开发日志  → docs/dev-log/          （不再散落）
新知识沉淀  → docs/knowledge/        （不再写在README里）
新配置      → config/                （不再丢根目录）
新脚本      → scripts/               （不再到处放）
新素材      → assets/                （不再混在代码里）
```

#### Step 3：迁移旧文件（逐个确认，搬完验证）

只有以下情况才搬旧文件，每搬一个都要验证：

| 文件类型 | 能不能搬 | 验证方法 |
|----------|---------|----------|
| 文档/笔记/日志 | ✅ 安全 | 搬完读一下确认内容完整 |
| 配置文件 | ⚠️ 谨慎 | 检查是否有代码引用了旧路径 |
| 脚本 | ⚠️ 谨慎 | 跑一下确认还能正常执行 |
| 代码文件 | ❌ 不搬 | 代码路径改动需要全项目验证，成本太高 |
| CI/CD配置 | ❌ 不搬 | 路径写死，搬了CI就炸 |

**迁移操作规范**：
1. 搬前：`cp` 复制到新位置（不是mv）
2. 验证：确认新位置文件可正常使用
3. 旧文件：移到`./回收站/<项目名>-<日期>/`（不是rm）
4. 更新引用：如果有代码/脚本引用旧路径，改引用指向新位置
5. 再验证：跑一遍项目核心流程确认没炸

### 你跟agent说的话

```
帮我把 [项目名] 加入知识体系。
先读 ./入口.md，再读 ./共享知识/项目规范/ 下所有规范，
然后按"已有项目入体系流程"操作：
1. 建骨架（不碰现有文件）
2. 写INDEX.md + hot-rules.md
3. 列出散落文件清单和迁移建议，等我确认后再搬
```

**关键：第3步只列建议不执行，等你确认。** 文件迁移是破坏性操作，必须人确认。
