# 多平台知识体系同步方案

> 最后更新：2026-05-09
> 解决扣子Coze、ECS OpenClaw等不同平台间知识体系数据同步问题

## 核心问题

- 知识文件在一个平台上更新了，其他平台不知道
- 不同平台的模型能力不同（有的笨有的聪明有的自作聪明）
- 修改冲突：两个平台同时改同一个文件怎么办
- ECS带宽限制：出站1Mbps

## 设计原则

1. **Git是唯一真相源**——所有平台从Git拉取，向Git推送，不平台间直连
2. **先拉后推**——任何修改前先pull，修改后立刻push
3. **单写者原则**——同一文件同一时刻只有一个平台在写（复用项目锁机制）
4. **冲突靠人**——自动合并只处理简单冲突，复杂冲突报警等人处理
5. **笨模型也能跑**——同步流程必须防呆，不需要模型理解Git原理

## 同步架构

```
GitHub（唯一真相源）
  ├── open-knowledge-system/    ← 知识体系专用仓库
  │   ├── 宪法层/               ← ./共享知识/
  │   ├── 入口.md
  │   ├── 角色/                 ← ./角色/
  │   └── 体系测试记录.md
  │
  ↕ pull/push（每个平台独立操作）
  │
  ├── 扣子Coze（主平台）       ← 日常开发+体系优化
  └── ECS OpenClaw（生产环境） ← 运维+执行任务
```

### 为什么不用平台间直连

```
❌ 扣子 → ECS 直传     带宽1Mbps，大文件慢；需要双向都要推
❌ 共享云存储           需要第三方服务，增加依赖
✅ Git中转              已有GitHub，零额外成本；版本历史天然可回滚
```

### 不同步的内容

| 内容 | 原因 | 怎么处理 |

## 同步流程

### 初始化（新平台首次接入）

```bash
# 1. Clone仓库到工作区根目录
cd /path/to/workspace
git clone git@github.com:youbanzhishi/open-knowledge-system.git ./

# 2. 软链接到体系目录（或直接复制）
# 方案A：软链接（推荐，改一处自动同步）
ln -s ./共享知识 ./共享知识
ln -s ./角色 ./角色

# 方案B：复制（简单但需手动同步）
cp -r ./共享知识 ./
cp -r ./角色 ./
```

### 日常同步（每次开工前+收工后）

```
开工前：
1. git pull --rebase          ← 拉取最新
2. 如果有冲突 → 按冲突处理流程走
3. 正常干活

收工后（反哺段中新增一步）：
1. git add -A
2. git commit -m "sync: [简要描述变更]"
3. git push
4. 如果push失败 → 先pull再push
```

### 防呆版同步（给笨模型用）

写成入口的一个铁律：

```
铁律N：跨平台同步
1. 开始干活前 → git pull --rebase
2. 改完文件后 → git add -A && git commit -m "sync: xxx" && git push
3. push失败 → git pull --rebase → 再push
4. 还失败 → 告诉用户"同步失败，需要手动处理"
```

## 冲突处理

### 预防（项目锁跨平台）

项目锁文件不同步，但可以用**Git分支锁**替代：

```
想改某个宪法文件前：
1. 检查Git远程是否有未合并的commit
2. 有 → 先pull → 如果别人在改同一个文件 → 等或换任务
3. 没有 → pull → 改 → push
```

### 处理（冲突发生时）

| 冲突类型 | 自动处理 | 例子 |
|----------|---------|------|
| 不同文件 | ✅ 自动合并 | A改了协作规范，B改了热规则规范 |
| 同文件不同章节 | ✅ 自动合并 | A改了协作规范的"角色体系"，B改了"项目锁" |
| 同文件同章节 | ❌ 报警，人处理 | 两人都改了项目锁机制那段 |
| 同文件末尾追加 | ❌ **会冲突** | 两个平台都在文件末尾加了新内容 |

**追加不改写降低了大部分冲突，但同章节追加仍会冲突。** 根本解法不是在追加层面防，而是在写入层面防。

### 跨平台写锁：Git commit作为锁

复用"同一文件同一时刻只有一个写者"的原则，跨平台版用**Git远程状态**当锁：

```
写入前：
1. git pull --rebase          ← 先拉最新
2. 检查要改的文件是否有人刚改过
   git log --oneline -5 -- <文件路径>
3. 如果最近有其他平台的commit → 询问用户"XX平台刚改了这个文件，要继续吗？"
4. 用户确认 → 修改 → 立刻push（趁没人改赶紧推）
5. push失败 → 说明有人抢先了 → pull → 重新改 → 再push

写入后：
1. git add -A
2. git commit -m "sync: [平台名] [简要描述]"
3. git push
4. push失败 → git pull --rebase → 再push
5. 还失败 → 告诉用户"同步冲突"
```

**核心逻辑**：pull时看到的文件状态=你的写入基础，改完立刻push=抢锁成功。push失败=被人抢先了，需要重新基于最新状态改。

### 单平台写入权（特定场景可选，非强制）

> ⚠️ 这个方案在两个平台同时干活时不现实，只适用于"某个知识域天然只在一个平台产生"的情况

如果某个知识域**天然只在一个平台产生**，可以约定写权：

```
规则示例：
- 宪法层 → 只在扣子改（体系设计在这边做）
- 运维知识 → 只在ECS改（运维在那边执行）
- 角色知识 → 两个平台都能改（都会积累经验）
```

但角色知识两边都会写，这才是真正的冲突场景。见下面的并发写入方案。

### 并发写入的真实解法：文件级分流 + 频繁小推

两个平台同时写经验，不可能互相等。解法是**让它们写不同的文件**：

```
❌ 两个平台都往 协作规范.md 追加 → 冲突
✅ 每个平台写各自的时间戳文件 → 天然不冲突
```

**具体方案**：经验先写到独立文件，再定期合并到主文件

```
写入时（不冲突）：
  扣子：./角色/系统开发者/knowledge/rust/clippy-1.95.md     ← 独立文件
  ECS：./角色/系统开发者/knowledge/rust/内存优化.md          ← 独立文件
  → 不同文件，Git自动合并 ✅

合并时（低频，手动或半自动）：
  定期把独立文件的知识合并到主知识文档
  合并时先pull → 合并 → push
  因为是低频操作，冲突概率极低
```

**这其实就是我们已有机制的天然优势**——knowledge/目录下是独立文件，天然不冲突！冲突只发生在"同一个.md文件被两个平台同时改"的情况。

### 实际冲突风险分析

| 文件类型 | 冲突风险 | 原因 |
|----------|---------|------|
| 角色/knowledge/*.md | 🟢 低 | 独立文件，新经验=新文件 |
| 角色/hot-rules.md | 🟡 中 | 同一文件，但改的频率低 |
| 项目INDEX.md | 🟡 中 | 同一文件，但通常只有一个平台在管 |
| 宪法层文件 | 🔴 高 | 同一文件，且两边都可能改 |
| 知识注册表 | 🟡 中 | 同一文件，追加新行 |

**结论**：大部分写入（knowledge/下的独立文件）天然不冲突。冲突集中在少数"公共文件"（宪法层、热规则、注册表）。

### 公共文件的并发写入规则

对于冲突风险高的公共文件，用**"先占后推"**规则：

```
1. 想改公共文件前 → git pull --rebase
2. 改完 → 立刻 git push
3. push失败 → pull → 看冲突：
   a. 能自动合并 → push ✅
   b. 冲突标记 → 看两个版本：
      - 追加类冲突（都在末尾加了东西）→ 合并两份保留 → push
      - 修改类冲突（都改了同一段）→ 告诉用户"需要你决定"
4. 搞不定 → 不改，告诉用户
```

**关键**：知识文件都是文本，冲突合并比代码简单得多。大部分冲突只是"两边都追加了内容"，合并=两份都保留就行。

### 最实际的方案：频繁小推

```
原则：改一点推一点，不要攒一堆再推
- 改了一个文件 → 立刻push
- 不要攒5个文件再一起push

攒得越久 → 跟远程差异越大 → 冲突概率越高
频繁小推 → 每次差异小 → 冲突概率低
```

### 冲突兜底（不可完全避免）

```
1. git push 失败 → git pull --rebase
2. rebase自动合并成功 → push ✅
3. rebase有冲突 → 看冲突标记：
   - 两个版本都值得保留 → 合并两份 → push
   - 只保留一份 → 手动选一个 → push
   - 搞不定 → 告诉用户"⚠️ 同步冲突需要你决定"
4. 知识文件都是文本，冲突合并比代码简单得多
```

**总结**：冲突不可完全避免，但三个策略叠加把概率压到很低：
1. 频繁小推（改一点推一点）
2. 文件级分流（knowledge/独立文件天然不冲突）
3. Git锁检测（pull时看最近的commit是谁的）

遇到冲突不怕——Git能检测到就是安全的，可怕的是静默覆盖。

### 巧合冲突的真实场景与处理

两个平台恰好同时改了同一个文件的同一章节——**会碰上，但概率低，且后果可控**：

```
场景：扣子和ECS同时往 知识注册表 追加了新行

Git行为：
  | 系统开发者 | rust, python | ./角色/系统开发者/ |
  | ECS运维 | linux, docker | ./角色/ECS运维/ |     ← ECS刚加的
  | 系统开发者 | rust, python | ./角色/系统开发者/ |
  | 小程序开发者 | js, python | ./角色/小程序开发者/ | ← 扣子刚加的

处理：两边都保留，删掉冲突标记 → push
  | 系统开发者 | rust, python | ./角色/系统开发者/ |
  | ECS运维 | linux, docker | ./角色/ECS运维/ |
  | 小程序开发者 | js, python | ./角色/小程序开发者/ |
```

**知识文件冲突 ≈ 两边都追加了内容** → 合并=两份都保留，不需要理解逻辑，不需要取舍。

**对比代码冲突**：代码冲突需要理解语义才能合并；知识文件冲突通常只是"谁加的内容在前谁在后"，两边都有价值，全保留即可。

**最坏情况**：两个平台改了同一段的同一个字 → 问用户选哪个。但这几乎不会发生——谁会同时改同一段的同一个字？

## 模型适配层

不同平台不同模型，能力参差不齐。入口需要**三层防呆**：

### 第一层：流程防呆（入口.md本身）

- 每一步都是"读X → 判断Y → 做Z"，不需要推理
- 所有选项列出来，不需要记忆
- 所有判断都有表格，不需要逻辑

### 第二层：模板防呆（回复模板）

- 身份确认模板 → 复制填空，不需要理解格式
- 自纠模板 → "发现XX → 告诉用户YY"，不需要决策
- 反哺模板 → 清单式，逐项打勾

### 第三层：铁律防呆（底线规则）

- 删除→回收站（不需要判断该不该删）
- 先读后动（不需要记忆文件内容）
- 追加不改写（不需要合并逻辑）
- 重试≤2次（不需要评估该不该继续）

**目标：最笨的模型只要能"读文件、按步骤执行"，就能跑这套体系。**

## 与现有机制的关系

```
项目锁（.项目锁）   ← 平台内并发控制，不同步
知识注册表          ← 跨角色发现，同步
宪法同步机制        ← 平台内版本追踪，同步到Git
自动沉淀            ← 平台内实时触发，同步到Git
跨平台同步          ← 新增机制，Git作为中转
```

**扩展即注册**：跨平台同步不是改体系，是在平台适配层注册一个新的同步通道。
