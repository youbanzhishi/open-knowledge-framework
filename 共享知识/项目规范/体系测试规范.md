# 知识体系自检与测试规范

> 最后更新：2026-05-09
> 知识体系也是程序，需要测试。三大类：结构自检（程序化）、流程测试（场景化）、回归测试（防倒退）

## 一、结构自检（可程序化执行）

任何session随时可以跑，纯文件检查，不需要人工参与。

### 自检清单

| # | 检查项 | 检查方式 | 期望结果 | 失败处理 |
|---|--------|---------|---------|---------|
| S1 | 宪法文件存在 | `ls ./共享知识/项目规范/*.md` | 7个文件都在 | mkdir -p + 报警 |
| S2 | 宪法文件有更新日期 | 每个文件第3行含"最后更新" | 全部有日期 | 补日期 |
| S3 | 入口存在 | `ls ./入口.md` | 存在 | 报警 |
| S4 | 知识注册表存在 | `ls ./共享知识/knowledge-registry.md` | 存在 | 创建空表 |
| S5 | 角色目录结构完整 | 每个角色有RULES.md + knowledge/ + hot-rules.md | 全部齐全 | mkdir -p + 报警 |
| S6 | 项目INDEX存在 | 每个已入体系项目有INDEX.md | 全部有 | 报警（可能未入体系） |
| S7 | 项目热规则存在 | 每个已入体系项目有规划/hot-rules.md | 全部有 | 创建空文件 |
| S8 | 回收站存在 | `ls ./回收站/` | 目录存在 | mkdir -p |
| S9 | 知识注册表与实际一致 | 注册表中的路径都能访问 | 全部可达 | 更新注册表 |
| S10 | 角色技能标签在注册表 | 每个角色RULES的技能标签在注册表有行 | 全部有 | 追加注册 |
| S11 | INDEX同步日期未过期 | INDEX同步日期 >= 宪法最后更新日期 | 全部同步 | 标记需要同步 |
| S12 | 无孤立.项目锁 | 锁的session还活着 | 无超时锁 | 释放超时锁(>30min) |
| S13 | 无孤立.活跃 | 活跃标记的session还活着 | 无超时标记 | 清理超时标记 |

### 执行方式

```bash
# 手动执行（任何session都可以）
1. 逐项检查上表
2. 输出：✅ N项通过 / ⚠️ N项需处理 / ❌ N项失败
3. 失败项自动修复（能修的）或报警（不能修的）
```

### 定期执行

通过heartbeat定期触发（建议每天1次），检查项标记为 `[体系自检]`。

## 二、流程测试（场景化验证）

模拟真实使用场景，验证整个流程跑通。需要人工触发。

### 测试用例

#### T1: 新会话身份识别流程

```
前置：清空所有.项目锁和.活跃
步骤：
1. 告诉agent "读 ./入口.md"
2. 验证：扫描结果为"全空"
3. 说"OpenLink开发"
4. 验证：身份确认格式正确（竖排、emoji）
5. 确认身份
6. 验证：身份卡已写入，格式正确
7. 说"你想让我做什么？"
8. 验证：等待用户指令，没有瞎跑

通过标准：每步符合预期
```

#### T2: 选错自纠流程

```
前置：OpenDAW项目目录存在但无INDEX.md
步骤：
1. 告诉agent "读 ./入口.md"
2. 选B.已有项目 + OpenDAW
3. 验证：agent发现没有INDEX.md，询问是否切换D
4. 说"是"
5. 验证：切换到D.项目入体系流程

通过标准：agent发现了矛盾，没有自动跳转，问了用户
```

#### T3: 失忆恢复流程

```
前置：OpenLink项目有.项目锁（身份卡已写）
步骤：
1. 新session，告诉agent "读 ./入口.md"
2. 验证：扫描到.项目锁
3. 验证：读取身份卡，恢复身份
4. 验证：回复包含"我恢复了记忆"但不假装记得对话细节
5. 验证：等用户指示

通过标准：恢复身份，诚实面对失忆，等指示
```

#### T4: 重复读导航流程

```
前置：已有身份卡，正在干活
步骤：
1. 说"重读 ./入口.md"
2. 验证：扫描到自己的身份卡
3. 验证：不重新走身份确认，刷新记忆继续
4. 说"重选"
5. 验证：删除旧身份卡，重新走确认流程

通过标准：默认继续，"重选"才重来
```

#### T5: 知识注册表自动发现

```
前置：系统开发者有knowledge/rust/，知识注册表已记录
步骤：
1. 创建新角色"应用开发者"，技能标签含Rust
2. 验证：查知识注册表发现系统开发者有Rust知识
3. 验证：在应用开发者/knowledge/rust/下创建引用文件
4. 验证：引用文件指向系统开发者的rust目录

通过标准：新角色自动引用前辈知识
```

#### T6: 项目锁机制

```
步骤：
1. 创建.项目锁
2. 另一个session尝试写同一项目
3. 验证：第二个session发现锁存在，等待或换任务
4. 释放锁
5. 验证：第二个session可以拿到锁

通过标准：并发写被阻止，释放后可拿锁
```

#### T7: 反哺闭环

```
步骤：
1. 完成一个开发任务
2. 验证：INDEX.md已更新最近变更
3. 验证：新踩坑已写入hot-rules或共享知识
4. 验证：跨项目经验已写入角色knowledge
5. 验证：知识注册表已更新（如有新知识路径）

通过标准：5个反哺点全部覆盖
```

#### T8: 宪法同步

```
前置：修改某个宪法文件，更新"最后更新"日期
步骤：
1. 新session接手项目
2. 读INDEX.md的同步日期
3. 验证：发现同步日期 < 宪法更新日期
4. 同步宪法内容
5. 更新INDEX同步日期

通过标准：版本追踪法工作正常
```

## 三、回归测试（防倒退）

每次修改体系文件后，跑一遍关键断言，确保没改坏。

### 断言清单

| # | 断言 | 检查方式 |
|---|------|---------|
| R1 | 入口5步完整 | 文件包含"第1步"到"第5步" |
| R2 | 协作规范目录与实际章节一致 | 目录中的每个链接对应的章节存在 |
| R3 | 五层知识体系完整 | SOUL→TOOLS→宪法→INDEX→技能，每层有文件 |
| R4 | 知识流动5条路径都在 | 架构规范中包含路径1-5 |
| R5 | 角色目录与维度注册表一致 | 协作规范维度注册表列出的角色 = 实际目录 |
| R6 | 任务派发五段式完整 | 协作规范包含5段模板 |
| R7 | 铁律数量一致 | TOOLS.md声明的铁律数 = 实际列出的条目数 |

### 执行时机

- 每次修改宪法层文件后
- 每次修改入口后
- 主对话觉得"改了好多东西"时

## 五、测试与体系的关系

测试体系本身遵循"注册扩展"模式：

```
知识体系核心
  ├── 结构自检（程序化，随时可跑）
  ├── 流程测试（场景化，人工触发）
  └── 回归测试（断言式，修改后触发）

扩展点：
  ├── 新检查项 → 追加到结构自检表
  ├── 新测试场景 → 追加到流程测试
  ├── 新断言 → 追加到回归测试
  └── 新执行方式 → 注册即可
```

测试是体系的**免疫系统**——不治已病治未病，发现退化早于用户抱怨。

## 六、新会话无缝继续——专项测试（高优先级）

> 这是体系的生命线。ECS上的OpenClaw会话更不稳定，必须确保无缝接续零失败。

### 核心依赖链

新会话要无缝继续，依赖三个东西都正确：

```
1. 身份卡（.项目锁/.活跃）  → 恢复"我是谁、在哪个项目"
2. 项目INDEX              → 恢复"项目全貌、走到哪了"
3. 入口流程            → 知道怎么扫描、怎么恢复
```

**任何一环断裂，恢复就失败。** 以下测试覆盖所有断裂场景。

### 测试用例

#### T9: 完整恢复流程（最核心）

```
前置：
  - 项目X有.项目锁（身份卡已写，含场景/角色/项目INDEX路径/关联技能）
  - 项目X的INDEX.md存在且内容完整
  
步骤：
1. 新session启动，被告知"读 ./入口.md"
2. 扫描 → 发现.项目锁 → 读取身份卡
3. 读身份卡指向的INDEX.md
4. 回复："我恢复了记忆：[项目名]/[角色]/[之前在做什么]
   但我可能丢失了对话上下文，你想让我继续什么任务？"
5. 用户说具体任务 → 正常执行

通过标准：
  ✅ 身份恢复（项目名+角色+正在做什么）
  ✅ 承认对话上下文丢失，不假装还记得
  ✅ 等用户指示，不瞎跑
  ✅ 执行任务时遵守角色铁律+项目热规则
```

#### T10: 身份卡丢失（.项目锁被意外删除）

```
前置：
  - 项目X在体系里（有INDEX.md），但.项目锁没了
  - 用户说"继续OpenLink"
  
步骤：
1. 扫描 → 全空
2. 关键词匹配 → OpenLink → 检查项目目录 → 有INDEX.md → B.已有项目
3. 确认身份 → 写新身份卡
4. 读INDEX → 恢复项目全貌
5. 问用户要做什么

通过标准：
  ✅ 没有身份卡也能通过INDEX恢复项目全貌
  ✅ 走B.已有项目而不是D.项目入体系（因为INDEX已存在）
  ✅ 重建身份卡，下次可恢复
```

#### T11: INDEX过时（宪法已更新但INDEX没同步）

```
前置：
  - 项目INDEX的"最后同步宪法"日期早于某个宪法文件的"最后更新"日期
  - 新session接手
  
步骤：
1. 读INDEX → 发现同步日期过期
2. 自动同步：重读对应宪法文件，更新INDEX中的共享规范段
3. 更新INDEX的"最后同步宪法"日期
4. 继续执行任务

通过标准：
  ✅ 自动发现过时并同步，不需要用户提醒
  ✅ 同步后日期更新
  ✅ 不影响INDEX的其他内容
```

#### T12: 角色失忆（.活跃标记丢失）

```
前置：
  - 角色Y的.活跃标记没了
  - 用户说"用系统开发者角色"
  
步骤：
1. 扫描 → 角色目录无.活跃
2. 关键词匹配 → 角色入体系相关 → D2.角色入体系
3. ls ./角色/ → 发现系统开发者已存在 → 复用不新建
4. 读RULES + hot-rules + knowledge
5. 写.活跃标记
6. 问用户要做什么

通过标准：
  ✅ 不重建已有角色
  ✅ 读取角色所有知识（RULES + hot-rules + knowledge）
  ✅ 重建.活跃标记
```

#### T13: 全部丢失（身份卡+INDEX都没了）

```
前置：
  - 项目目录存在但没有INDEX.md和.项目锁
  - 用户说"继续开发OpenDAW"
  
步骤：
1. 扫描 → 全空
2. 关键词匹配 → OpenDAW → 检查目录 → 无INDEX.md
3. 自纠：提示用户"⚠️ OpenDAW还没有INDEX.md，可能没入体系。要切换到D.项目入体系吗？"
4. 用户确认 → 走D流程 → 建INDEX
5. 写身份卡

通过标准：
  ✅ 不走B.已有项目（因为没有INDEX）
  ✅ 询问用户确认，不自动跳转
  ✅ 入体系后INDEX和身份卡都建好
```

#### T14: 多项目切换

```
前置：
  - 用户同时提了两个项目
  
步骤：
1. 扫描 → 发现两个项目都有.项目锁
2. 询问用户要先处理哪个
3. 用户选择 → 读取对应身份卡+INDEX
4. 干完一个 → 释放锁 → 切换到另一个

通过标准：
  ✅ 不假设用户想先做哪个
  ✅ 项目锁正确拿/释
  ✅ 切换时读取新项目的完整信息
```

#### T15: 跨平台恢复（ECS OpenClaw场景）

```
前置：
  - 知识文件从扣子环境复制到ECS OpenClaw
  - 文件结构完全一致
  
步骤：
1. ECS上的新session启动
2. 走入口流程
3. 扫描 → 发现.项目锁（或全空）
4. 恢复/建立身份 → 读INDEX → 执行任务

通过标准：
  ✅ 知识文件零修改直接可用
  ✅ 流程与扣子环境完全一致
  ✅ 平台差异只体现在适配层，不影响知识层
```

### 测试优先级

| 优先级 | 测试 | 原因 |
|--------|------|------|
| P0 必过 | T9 完整恢复 | 日常最常见场景 |
| P0 必过 | T10 身份卡丢失 | 锁超时释放后一定发生 |
| P0 必过 | T13 全部丢失 | 旧项目首次入体系 |
| P1 重要 | T11 INDEX过时 | 宪法频繁更新时常见 |
| P1 重要 | T12 角色失忆 | .活跃标记丢失 |
| P2 一般 | T14 多项目切换 | 少见但需要正确处理 |
| P2 一般 | T15 跨平台恢复 | 部署到ECS时验证 |

**P0不过=体系不可用，必须修到过为止。**
