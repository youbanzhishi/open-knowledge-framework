# 知识体系架构规范

> 最后更新：2026-06-03
> 知识体系自身的架构规范——跟OpenLink同构：扩展即注册，架构永不改

## 🗺️ 全景速览（一图看全貌）

```
五层知识体系：
  T0 SOUL.md          ← 我是谁
  T1 TOOLS.md         ← 我怎么工作
  T2 宪法层（共享知识）  ← 所有项目共享
  T3 项目INDEX+热规则   ← 单个项目
  T4 技能              ← 操作手册

知识归属原则：经验在人（角色）身上
  项目知识："为什么这样配" → 换个项目就没用了 → 项目docs/knowledge/
  角色经验："怎么做更好"   → 换个项目还能用   → 角色knowledge/
  共享模式："大家都能用"   → 换个角色还能用   → 共享知识/

五条知识流动路径：
  项目踩坑 → 项目热规则（项目内防犯，踩坑2次升级）    🟢全自动
  项目经验 → 角色knowledge（跨项目复用，强制反哺）    🟡半自动
  角色经验 → 共享知识（跨角色复用，通用化提炼）       🟡半自动
  角色能力 → 技能导出（跨人复用，冻结快照）          🟠手动
  角色进化 → 技能自动更新（增量同步，保持技能鲜活）   🟡半自动

角色→技能导出4原则：
  1. 保留"为什么"，去掉"在哪"
  2. 保留"怎么做"，去掉"对谁"
  3. 保留"坑在哪"，去掉"我踩的"
  4. 保留"模板"，去掉"配置"

技能封装判断：去掉判断力还能干吗？能干=工作流，干不了=技能

三重同构：
  Extension Registry（新功能=注册扩展）≅ 知识体系（新维度=注册路径）≅ 平台适配器（新平台=注册适配层）
  → 开放体系的核心不是功能全，而是扩展成本低

平台无关性5个最小接口：读文件 / 写文件 / 派任务 / 会话管理 / 定时触发

铁律4条+工作守则4条：
  铁律（违反=不可逆损害）：删除→回收站 / 改前备份 / 做完立即记录 / 禁止扫描无关目录
  工作守则（违反不致命但应遵守）：先查再干 / 经验跟着人走 / 新工具要注册 / 同样的错不犯两次
```

> 详细内容见下方各章节：一～七（架构基础）| 八（流动全景）| 九（平台无关性）| 十（同构）

## 核心问题

1. **反哺靠自觉**：DAW只吸取不反哺，需要主人提醒
2. **扩展靠大改**：新增一个维度就要改整个体系
3. **被动发现**：知识躺在那里等人翻，不会主动流动

## 解法：知识体系也用Extension Registry模式

OpenLink的核心哲学是"新功能=注册扩展，架构本身永远不需要改"。
知识体系应该遵循同样的哲学。

### 三条元规则

1. **反哺是任务完成条件，不是可选行为**
2. **体系扩展=注册新维度，不改动现有结构**
3. **知识自动流动，不依赖人提醒**

## 一、反哺自动化

### 问题

现在：session完成任务 → 任务结束 → 知识沉淀是可选项 → 大部分时候不做
期望：session完成任务 → 反哺是强制的 → 不反哺=任务没完成

### 解法：反哺纳入任务闭环

```
任务启动 → 执行 → 验收 → 【强制反哺】 → 任务才算完成
```

#### 反哺清单（每次任务完成后必须过一遍）

##### 反哺轨迹预填（借鉴Syll示教即技能）

传统反哺：任务完成→空白模板→从零填写→遗漏多→质量差
轨迹预填：任务完成→自动回顾执行轨迹→生成预填内容→确认/修正→高质量反哺

预填来源（agent自回顾，不需要额外工具）：
1. 执行步骤序列：我按什么顺序做了什么
2. 工具调用摘要：调了哪些工具/API，结果如何
3. 踩坑记录：遇到了什么错误，怎么解决的
4. 通用经验提炼：哪些经验换个项目还有用

角色汇报反哺时，格式为：
- 轨迹预填：[自动生成的4项内容]
- 确认/修正：[哪些确认了，哪些修正了，补充了什么]
- 最终沉淀：[写入的具体文件和内容摘要]

| 反哺目标 | 什么时候 | 反哺什么 | 存哪 | 判断标准 |
|----------|----------|----------|------|----------|
| 项目INDEX | 每次必做 | 最近变更+文档索引更新 | 项目/INDEX.md | — |
| 项目docs/knowledge | 每次必做 | 项目特有事实 | 项目/docs/knowledge/ | 换个项目就没用了（为什么这样配） |
| 项目热规则 | 踩了2次坑 | 新增/升级热规则 | 项目/规划/hot-rules.md | — |
| 角色knowledge | 发现通用经验 | 方法论/踩坑/方案 | 角色/<角色名>/knowledge/ | 换个项目还能用（怎么做更好） |
| 共享知识 | 发现跨角色通用模式 | 踩坑记录/设计模式 | 共享知识/ | 换个角色还能用（大家都能用） |
| 宪法同步 | 宪法有更新 | 更新INDEX同步日期 | 项目/INDEX.md | — |

#### 主对话的执行方式

派发任务时，任务描述**必须包含以下五段**（按顺序）：

**第零段：导航版本检查（每个任务必须有）**
```
开工前读 ./入口.md 顶部版本号，变了只看摘要，没变跳过。
```

**第一段：铁律+工作守则（每个任务必须有）**
```
⛔ 铁律（违反=不可逆损害，必须遵守）：
1. 删除→回收站，禁止直接删除。移到 ./回收站/<名称>-<日期>/
2. 改文件前先备份再读，禁止直接覆盖
3. 做完立即记录，智能体会失忆，没记下来的东西就真没了
4. 禁止扫描无关目录（回收站/其他项目/用户上传）

📋 工作守则（违反不致命，但应该遵守）：
- 先查再干：开工前查项目INDEX和角色RULES的可用工具，有现成方案不手搓
- 经验跟着人走：做完后问"换个项目还有用吗？"有用的归角色knowledge
- 新工具要注册：造了新工具必须注册，项目专用→INDEX，跨项目→RULES
- 同样的错不犯两次：同一错误≤2次，第3次换方案
```

**第二段：项目热规则（派发项目相关任务时注入）**
```
📌 项目热规则（来自 项目/规划/hot-rules.md）：
[粘贴该项目热规则内容]
```

**第三段：目录自愈+进度汇报（每个任务必须有）**
```
🔧 目录自愈：操作项目前先跑 INDEX.md 中的 mkdir -p 命令

📊 进度汇报（让用户知道你在干活）：
1. 接到任务 → 汇报"收到，开始干 [摘要]，预计分N步"
2. 每完成一个关键步骤 → 汇报"第X/N步完成：[做了什么]，下一步：[要做什么]"
3. 遇到阻塞/报错 → 立刻汇报"⚠️ 第X步阻塞：[错误]，正在[换方案/需要决策]"
4. 同一步骤超过3分钟 → 汇报"还在处理第X步：[当前在干什么]"
5. 任务完成 → 汇报"完成：[结果摘要]，[产出物位置]"
沉默上限5分钟，超5分钟无输出=异常
```

**第四段：强制反哺（每个任务必须有）**
```
【强制·任务完成条件】
任务产出只是第一步，以下反哺必须完成才算任务结束：
1. 更新项目INDEX.md的最近变更
2. 项目特有事实 → 项目docs/knowledge/（判断：换个项目就没用了）
3. 如有新踩坑 → 项目hot-rules或共享知识/踩坑记录/
4. 如有通用经验 → 对应角色的knowledge/（判断：换个项目还能用）
5. 如发现跨角色通用模式 → 共享知识/（判断：换个角色还能用）
6. 如发现宪法更新 → 同步INDEX.md的同步日期
7. ⚡ 效率审计（必做）：本次任务有没有可以抽取复用的？有没有可以自动化的？标记出来

知识写入位置速查（核心原则：经验在人身上）：
  项目特有事实 → ./项目/<项目名>/docs/knowledge/  ← "为什么这样配"，换项目没用
  角色通用经验 → ./角色/<角色名>/knowledge/          ← "怎么做更好"，换项目还有用
  跨角色通用   → ./共享知识/踩坑记录/ 或 设计模式/    ← "大家都能用"，换角色还有用
  操作手册     → ./技能/<技能名>/（5步以上复杂流程）
  项目热规则   → ./项目/<项目名>/规划/hot-rules.md
```

#### 自动化程度

| 反哺动作 | 自动化程度 | 怎么做到 |
|----------|-----------|----------|
| INDEX最近变更 | 🟢 完全自动 | sub-agent任务末尾强制执行 |
| 项目热规则 | 🟢 完全自动 | 踩坑2次自动升级，sub-agent判断 |
| 角色knowledge | 🟡 半自动 | 需要主对话验收时判断"这是否跨项目通用" |
| 共享知识 | 🟡 半自动 | 同上，需要判断"是否对所有项目通用" |
| 宪法同步 | 🟢 完全自动 | 版本追踪法，日期不一致就同步 |

**关键转变**：反哺从"可选的good practice"变成"任务完成的强制条件"。

## 二、体系扩展性

### 问题

现在：新增一个维度（比如"角色"概念） → 改协作规范 → 改目录规范 → 改启动流程 → 大改
期望：新增维度 = 注册到现有框架，不改动现有结构

### 解法：维度注册制

知识体系的每一"类"实体是一个**维度**。新维度=注册，不改现有维度。

#### 维度注册表

```
维度名      | 实体目录        | 规则文件    | 入口文件   | 反哺方式    | 派发模板
-----------|----------------|------------|-----------|----------|----------
项目        | 项目/<名>/  | INDEX.md   | INDEX.md  | 更新INDEX  | —
角色        | 角色/<名>/      | RULES.md   | RULES.md  | 更新knowledge | templates/dispatch*.md
```

**未来新增维度只需一行**：

```
维度名      | 实体目录        | 规则文件    | 入口文件   | 反哺方式    | 派发模板
-----------|----------------|------------|-----------|----------|----------
社区        | 社区/<名>/      | CHARTER.md | CHARTER.md| 更新动态    | —
知识源      | 知识源/<名>/    | FEED.md    | FEED.md   | 更新条目    | —
```

不用改任何现有文件。新维度注册即可。

#### 维度通用接口

不管什么维度，实体必须实现以下"接口"：

```
每个实体必须有：
1. 入口文件：新session接手时读的第一个文件（INDEX/RULES/CHARTER/...）
2. 热规则文件：执行任务时注入的规则
3. knowledge目录：经验积累
4. log目录：操作记录
5. 反哺能力：能向其他实体输出经验
6. 派发模板：templates/dispatch.md（主agent派发时的task模板）
```

这6个是"接口"，不同维度可以有自己的"实现"：
- 项目：入口=INDEX.md，热规则=规划/hot-rules.md
- 角色：入口=RULES.md，热规则=hot-rules.md，派发模板=templates/dispatch*.md
- 未来维度：入口=CHARTER.md，热规则=charter-rules.md

**接口不变，实现可扩展。** 跟Rust的trait一样。

#### 目录结构也是注册制

当前目录结构规范列出了固定目录。未来新增目录类型怎么办？

解法：**共性目录必须统一，个性目录注册即用**。

```
共性目录（所有实体必须有）：
  - 入口文件
  - 热规则
  - knowledge/
  - log/

个性目录（按需注册，命名约定）：
  - 项目的 src/ tests/ output/ feedback/
  - 角色的 templates/
  - 未来的 任何新需求/
```

新增个性目录时：
1. 在目录结构规范中注册（追加，不改现有）
2. 说明适用范围（所有实体/特定维度/特定实体）
3. 命名遵循约定（小写+连字符）

## 三、知识归属与自动流动

### 核心原则：经验在人身上

**类比现实**：一个工程师做了三个项目，项目结束了文档归档，但工程师变强了。换个新项目，他不用从零踩坑。

**映射到体系**：角色=人，项目=工作台。通用经验跟着角色走，项目只存项目特有事实。

### 知识归属判断标准

**判断一个问题：这条知识换个项目还有用吗？**

| 判断 | 归属 | 举例 |
|------|------|------|
| 换个项目就没用了 | **项目** docs/knowledge/ | 这台ECS为什么端口7772、这个项目的cargo配置为什么CARGO_BUILD_JOBS=2 |
| 换个项目做同类事还能用 | **角色** knowledge/ | 低内存环境要限制编译并发、FDN混响高频闪烁怎么解决、网文黄金三章怎么写 |
| 换个角色做不同领域的事也能用 | **共享知识** | Git分支策略、CI通用模式、token消耗黑洞 |

**一句话判断**：项目知识="为什么这样配"，角色经验="怎么做更好"，共享模式="大家都能用"。

### 知识自动流动

知识有三层，从具体到通用，自动向上提炼：

```
第3层：实体级（项目hot-rules / 角色knowledge）
  ↓ 踩坑2次 / 发现跨项目模式
第2层：维度级（共享知识/项目规范/）
  ↓ 发现通用模式
第1层：全局级（TOOLS.md / SOUL.md）
```

**流动是单向的**：具体→通用。不会从通用流向具体（通用是约束，不是知识）。

**角色是知识的主载体**：项目是知识的产生地，但通用经验的归宿是角色。项目结束可以归档，角色经验持续积累。

### 三层单向关联

项目、角色、技能之间是单向关联：

```
项目INDEX → 声明关联角色（消费者找提供者）
角色RULES → 声明关联技能 + 可用工具（人拿工具）
技能       → 独立存在（工具不关心谁在用）
```

**不需要双向关联**：项目需要角色，但角色不需要知道自己在服务哪些项目。类比现实：分诊台需要知道"外科医生擅长什么"，但外科医生不需要知道今天会来哪些病人。角色是能力的提供者，项目是能力的消费者。双向关联维护成本高，收益低。

**工具也在人身上**：跨项目通用的工具（如open-dev-tools）不应在每个项目里重复声明，应归到角色RULES的"可用工具"段。项目通过关联角色自动获得工具。项目专用的工具写在项目INDEX的"可用工具"段。

**新工具必须注册**：开发过程中造了新工具（脚本/模板/流程），必须注册到对应位置。只管用不管注册=不反哺=体系不成长。注册位置判断同经验归属：项目专用→项目INDEX，跨项目通用→角色RULES。

### 入口的使用哲学

**铁律是安全培训，不是参考书。**

类比现实：百姓不需要通读法律，但犯罪率很低，因为大家知道哪些事一定不能做。入口的铁律就是"底线法律"——agent不需要通读架构规范，但必须知道哪些事一定不能做。

**铁律判断标准：违反了会造成不可逆损害吗？**

| 条目 | 违反后果 | 是铁律？ |
|------|---------|---------|
| 删除→回收站 | 文件永久丢失，不可逆 | ✅ 铁律 |
| 改文件前先备份 | 覆盖原内容，不可逆 | ✅ 铁律 |
| 做完立即记录 | 信息丢失（智能体会失忆） | ✅ 铁律（失忆不可逆） |
| 禁止扫描无关目录 | token浪费，大量浪费不可接受 | ✅ 铁律（资源不可逆） |
| 先查再干 | 浪费时间，但可逆 | ❌ 工作守则 |
| 经验跟着人走 | 知识放错位置，可修正 | ❌ 工作守则 |
| 新工具要注册 | 工具不可发现，可补注册 | ❌ 工作守则 |
| 同样的错不犯两次 | 浪费资源，不会不可逆损害 | ❌ 工作守则 |

核心逻辑：铁律只保留"违反=不可逆损害"的条目。犯了可补救的，降为工作守则。条目越少，agent越容易记住——8条记不住，4条刻在脑子里。

三层加载：
1. 铁律（必读，4条，10秒读完）— 不读规范也不会犯底线错误
2. 入口文件（INDEX/RULES，按需）— 分诊台指向
3. 详细规范（架构规范/目录规范，按需）— 需要时才查

#### 流动触发条件

| 从 | 到 | 触发条件 | 谁负责 |
|----|----|----------|--------|
| 项目踩坑 | 项目热规则 | 同一个坑踩2次 | sub-agent自动 |
| 项目踩坑 | 角色knowledge | 经验跨项目通用 | 主对话验收时判断 |
| 角色knowledge | 共享知识 | 经验跨角色通用 | 主对话判断 |
| 共享知识 | TOOLS.md | 影响所有行为 | 主对话判断 |
| 任何层级 | SOUL.md | 涉及身份/原则 | 仅主人决定 |

#### 反向流动：通知

知识从通用层更新后，需要通知下游：
- 宪法更新 → 各项目INDEX同步日期过期 → session自检时发现并同步
- 角色knowledge更新 → 下次角色驱动项目时自动带上新经验
- TOOLS.md更新 → 下次派任务时自动注入新铁律

**通知是被动触发的，不需要主动推送。** 但任务描述中包含"检查宪法同步"作为强制步骤，确保不遗漏。

## 三-B、Ritual情境触达（借鉴Syll主动陪伴机制）

### 核心区别

| 维度 | 工单型日程(Calendar) | Ritual情境触达 |
|------|---------------------|----------------|
| 定位 | 必须执行的工单 | 可选的情境关怀 |
| 失败 | 失败需重试 | 沉默是合法响应 |
| 输出 | 明确的交付物 | 可能无输出（NO_REPLY） |
| 触发 | 固定时间 | 时间+情境判断 |
| Token | 必须消耗 | 可零消耗（判断后沉默） |

### 内置Ritual（4个）

| Ritual | 触发时间 | 触达条件 | 内容 |
|--------|---------|---------|------|
| 晨间要点 | 每日9:00 | 有活跃项目且昨天有变更 | 昨日变更摘要+今日待办 |
| 晚间收束 | 每日22:00 | 有未完成任务或待领工单 | 未完成项提醒+明日建议 |
| 记忆浮现 | 每周二/五15:00 | 历史记忆与当前任务有关联 | 翻出关联历史经验/踩坑/决策 |
| 周结回顾 | 每周日21:00 | 本周有项目进展 | 本周完成项+下周预览+体系健康度 |

### Ritual执行规则

1. **触发时先判断**：读当前项目状态→判断是否有价值→无价值→输出NO_REPLY（不算失败）
2. **有价值的标准**：至少满足一项——有未读变更/有遗忘风险/有跨项目经验可关联/有工单堆积
3. **输出格式**：简短+情境化+可操作。不说"你有3个待办"，而是"OpenDAW的CI从6月1号挂了，还没人领"
4. **沉默优先**：不确定有没有价值→沉默。宁可漏报不可打扰

### 与Calendar的配合

Ritual类型日程在Calendar创建时，description中标注 `[ritual]` 前缀，执行session看到前缀即按ritual规则处理（允许NO_REPLY，不要求强制交付物）。

### 在主agent（小龙）侧的落地

主agent创建Calendar日程时，如果是ritual类型：
- summary格式：`[ritual] 晨间要点` / `[ritual] 记忆浮现` 等
- description中写明ritual类型、判断条件、沉默规则
- 不设time_range（ritual对时间不严格，避免整点竞争资源即可）

## 四、体系演进保障

### 问题

以后可能需要很多现在想不到的扩展，不想每次大改。

### 解法：体系本身的Extension Registry

知识体系本身遵循"注册扩展"模式：

```
知识体系核心（永不改）
  ├── 五层架构：SOUL→TOOLS→宪法→实体→技能
  ├── 维度注册表：项目/角色/未来维度
  ├── 反哺闭环：任务完成→强制反哺

  ├── 新维度：社区/知识源/...
  ├── 新目录类型：在目录规范注册
  ├── 新反哺路径：在流动规则注册
  ├── 新实体类型：在维度注册表注册
  └── 新规则类型：在热规则规范注册
```

**核心层永不需要改。** 所有扩展通过注册实现。

### 未来可能扩展（现在不需要建，但架构预留）

|------|------|------|

| 经验评分 | 新规则类型 | 热规则的冷却升级机制更智能 |

这些现在都不需要建。等需求出现时，在对应位置注册即可。

## 七、技能的本质与封装规范

### 什么是技能？

**技能 = 步骤 + 判断力。判断力是核心。**

判断标准：**去掉判断力，这事还能不能干？** 能干→工作流，干不了→技能。

| 判断 | 类型 | 归宿 | 举例 |
|------|------|------|------|
| 核心不需要判断力，脚本就能跑 | 工作流 | 项目/scripts/ + 定时任务 | 掘金签到：调API→抽奖→记录 |
| 核心就是判断力，每次情况不同 | 技能 | ./技能/<技能名>/ | 混音：看频谱→判断问题→选方案→验证 |
| 判断力是核心但流程也有 | 技能 | ./技能/<技能名>/ | 知识沉淀：判断什么值得沉淀+格式化 |

**工作流 vs 技能的本质区别**：工作流的异常处理可能需要判断力，但核心路径不需要。技能的核心路径就需要判断力。

**不该封装为技能的**：
- 固定流程+定时触发（掘金签到、元宝收藏导出）→ 作为轻量运营项目
- 数据采集+报表（数据统计）→ 项目/scripts/
- 系统维护操作（git push同步）→ 基础设定/scripts/

**项目技能不存在**：技能是通用的，脱离任何项目也能独立执行。只对某个项目有用的操作手册不是技能，是项目文档。

### 工作流的归宿：轻量运营项目

**工作流 ≠ 技能**，工作流不需要角色来承载，也不需要技能的封装。工作流是**有自己状态的运营流程**，天然适合项目结构。

**工作流归角色的坏处**：
- 角色knowledge装的是经验（能让角色变聪明的经验才更值得归到角色中），工作流装的是流程（不会让角色变聪明）
- 工作流有独立状态（cookie、历史记录、执行日志），塞在角色目录下管理混乱
- 角色RULES变胖，每次加载浪费token

**工作流归项目的好处**：
- 项目目录天然支持状态管理（config/、scripts/、docs/）
- 定时任务描述直接指向项目INDEX，sub-agent读INDEX就能干
- 角色保持轻量，只装经验不装流程，按需加载省token

**轻量运营项目的最简结构**（不需要roadmap/decisions）：

```
项目/<工作流名>/
├── INDEX.md              ← 一句话描述 + 执行步骤 + 注意事项
├── scripts/              ← 自动化脚本
├── docs/knowledge/       ← 操作要点（API方法、配置说明等）
└── config/               ← 配置文件、历史数据
```

**判断一句话**：这个流程有自己独立的状态需要管理吗？有→轻量项目，没有→可能只是一段脚本。

### 角色→技能导出通道

### 场景

我训练了一个"Rust开发者"角色，积累了大量实战经验。我想把这份能力打包给别人用——导出为一个完善的技能。

### 角色 vs 技能的本质差异

| 维度 | 角色 | 技能 |
|------|------|------|
| 拥有者 | 我（特定环境+特定经验） | 任何人（通用环境） |
| 知识来源 | 实战积累（带环境细节） | 角色提炼（去环境依赖） |
| 运行方式 | 持续存在，越用越强 | 静态加载，按需使用 |
| 可分享性 | 不直接可分享 | 天然可分享 |

**角色是"活的"，技能是"冻结的快照"。** 导出=把角色的当前能力冻结成一个可移植的包。

### 导出流程：角色 → 技能

```
角色（活的，环境相关）
  ├── RULES.md          → 技能SKILL.md的"铁律"段
  ├── knowledge/        → 技能SKILL.md的"方法论"段 + references/
  ├── hot-rules.md      → 技能SKILL.md的"注意事项"段
  ├── templates/        → 技能的scripts/ + templates/
  └── log/              → 不导出（私有操作记录）
```

**导出时必须做的事**：

1. **去环境依赖**：把"必须用~/.cargo/bin/cargo"改成"确认cargo版本≥1.75"，去掉特定路径
2. **去私有信息**：去掉服务器IP、密钥路径、个人偏好
3. **泛化经验**：把"OpenLink必须CARGO_BUILD_JOBS=2"改成"低内存环境限制编译并发"
4. **补全说明**：加上背景说明，让陌生人能理解"为什么这样做"
5. **组织结构**：按SKILL.md标准格式组织

### 导出后的技能结构

```
./技能/<技能名>/
├── SKILL.md              ← 从角色RULES+knowledge+hot-rules提炼
│   ├── 技能信息
│   ├── 功能描述
│   ├── 使用场景
│   ├── 铁律（← RULES.md提炼，去环境依赖）
│   ├── 方法论（← knowledge提炼，泛化经验）
│   ├── 注意事项（← hot-rules提炼，泛化警告）
│   └── 加载方式
├── references/           ← 从knowledge/精选的关键文档
│   ├── workspace组织.md
│   ├── 编译环境配置.md
│   └── ...
├── scripts/              ← 从templates/提炼的可执行脚本
│   ├── setup-rust.sh
│   └── cargo-mirror.sh
└── templates/            ← 从templates/复制
    ├── Cargo.toml
    ├── ci.yml
    └── justfile
```

### 导出的关键原则

1. **保留"为什么"，去掉"在哪"**
   - 保留："编译并发需要限制，否则OOM"（为什么）
   - 去掉："CARGO_BUILD_JOBS=2因为1.8G内存"（具体环境）
   - 改成："低内存环境(≤4G)设置CARGO_BUILD_JOBS=2"

2. **保留"怎么做"，去掉"对谁"**
   - 保留："workspace组织：core/store/api/sdk分离"（怎么做）
   - 去掉："OpenLink的workspace结构是..."（特定项目）
   - 改成："Rust项目推荐workspace组织方式..."

3. **保留"坑在哪"，去掉"我踩的"**
   - 保留："系统cargo可能版本过旧，确认用正确版本"（坑在哪）
   - 去掉："必须用~/.cargo/bin/cargo"（我的路径）
   - 改成："确认cargo版本≥1.75，rustup安装的优先"

4. **保留"模板"，去掉"配置"**
   - 保留：Cargo.toml模板、CI模板、justfile模板
   - 去掉：镜像地址、服务器配置、密钥路径

### 自动导出的可能性

未来可以做半自动导出：
1. 读取角色的RULES.md + knowledge/ + hot-rules.md
2. AI自动去环境依赖、泛化经验
3. 生成SKILL.md初稿 + 组织references/
4. 人工审核调整
5. 打包发布

但现在手动导出就够了——积累到值得导出的时候再做。

### 技能元数据：progressive vs always-on（借鉴Syll技能加载模式）

> 解决长技能全量注入浪费token的问题

#### 两种模式

| 模式 | 加载时机 | 注入内容 | 适用场景 |
|------|---------|---------|---------|
| **progressive**（按需加载） | 触发关键词/场景匹配时 | 先注入摘要，触发后全量加载 | 大型技能（>500行）、项目级技能、低频技能 |
| **always-on**（常驻注入） | 每次会话 | 全量注入system prompt | 短小核心规则（<200行）、高频技能、铁律级规则 |

#### SKILL.md元数据格式

在SKILL.md的"技能信息"段中新增mode字段：

```markdown
## 技能信息
- 技能名称：xxx
- 技能类型：xxx
- 加载模式：progressive  # progressive | always-on
- 触发关键词：[关键词1, 关键词2]  # 仅progressive模式需要
- 摘要：一句话描述  # 仅progressive模式需要，注入system prompt的摘要
```

#### 加载逻辑

1. **always-on技能**：skill_load时全量注入，和现在一样
2. **progressive技能**：
   - skill_load时只注入"技能名+摘要+触发关键词"到system prompt
   - 当对话中命中触发关键词时，再次调用skill_load全量加载
   - 已加载的技能不重复加载

#### 现有技能分类

| 技能 | 建议模式 | 理由 |
|------|---------|------|
| openlink-dev | progressive | 项目级，只在开发OpenLink时用 |
| openvault-dev | progressive | 项目级，只在开发OpenVault时用 |
| 小红书笔记技能 | progressive | 低频，只在创作小红书内容时用 |
| 成长复盘技能 | progressive | 低频，只在复盘时用 |
| 网文创作技能 | progressive | 低频，只在写网文时用 |
| 动漫制作技能 | progressive | 低频，只在做动画时用 |
| 游戏开发技能 | progressive | 低频，只在开发游戏时用 |

当前所有技能都建议设为progressive——none是高频到需要always-on的。随着体系发展，如果有<200行的核心规则级技能，可设为always-on。

### 技能自动封装与更新机制

> 角色进化了，技能快照也得刷新

#### 触发条件

以下任一条件满足，自动触发技能封装/更新：

| 触发条件 | 动作 | 说明 |
|----------|------|------|
| 角色knowledge新增文件 ≥ 3个 | 提示"可导出" | 积累到值得封装了 |
| 角色hot-rules变更 | 更新已有技能的"注意事项"段 | 热规则=最容易犯的错，必须同步 |
| 用户说"导出技能"/"更新技能" | 立即执行 | 手动触发 |
| 定期巡检发现角色knowledge比技能内容多 | 提示"可更新" | 技能落后于角色了 |
| **任务结束Flush检查：本次任务足够复杂** | **多步骤流程，下次遇到可直接套用→建议用户确认封装** | 🆕落地触发：不依赖平台能力，靠反哺铁律自检 |

#### 封装流程（半自动）

```
1. 检测：角色knowledge文件数/内容量达到阈值
2. 提示：告诉用户"[角色名]已积累N个知识文件，可以导出/更新技能"
3. 等确认：用户说"导出"才执行（不自动覆盖）
4. 提炼：AI自动执行4个导出原则
   - 保留"为什么"，去掉"在哪"
   - 保留"怎么做"，去掉"对谁"
   - 保留"坑在哪"，去掉"我踩的"
   - 保留"模板"，去掉"配置"
5. 生成：SKILL.md + references/ + templates/
6. 写入：放到 ./技能/<技能名>/
7. 通知：告知用户导出完成，列出变更摘要
```

#### 更新流程（增量）

已有技能的更新不需要重新导出全部，只更新变化的部分：

```
1. 对比角色knowledge vs 技能SKILL.md
2. 只提炼新增/变更的知识
3. 增量更新SKILL.md对应段落
4. 更新references/（如有新文档）
5. 更新templates/（如有新模板）
6. 保留用户手动调整过的内容（不覆盖）
```

#### 质量保障

- **不自动发布**：封装完必须等用户确认
- **保留手动调整**：如果技能文件被用户手动改过，自动更新时保留用户改动
- **版本标记**：每次更新记录版本号和日期
- **回滚能力**：旧版本保留在references/的versions/子目录

#### 与反哺闭环的关系

技能自动封装是反哺的**第五条流动路径**：

```
路径1  项目踩坑 → 项目热规则        🟢全自动
路径2  项目经验 → 角色knowledge      🟡半自动
路径3  角色经验 → 共享知识            🟡半自动
路径4  角色能力 → 技能导出            🟠手动
路径5  角色进化 → 技能自动更新        🟡半自动（新增）
```

路径5和路径4的区别：路径4是首次导出（手动），路径5是后续更新（半自动）。

### 导出通道是注册扩展

导出通道不是"改体系"，是在知识流动规则中注册一条新路径：

```
已有流动路径：
  项目踩坑 → 项目热规则（项目内防犯）
  项目经验 → 角色knowledge（跨项目复用）
  角色经验 → 共享知识（跨角色复用）

新增流动路径（注册）：
  角色能力 → 技能导出（跨人复用）
```

**体系不变，路径追加。** 跟OpenLink的Extension Registry完全同构。

## 八、知识流动全景图

> 四条流动路径，覆盖从"项目内防犯"到"跨人复用"的完整知识生命周期

### 全景视图

```
知识流动全景：

路径1  项目踩坑 ──→ 项目热规则
      触发：同一个坑踩2次
      方向：项目内闭环
      产出：项目/规划/hot-rules.md 新增/升级条目
      自动化：🟢 完全自动（sub-agent判断）

路径2  项目经验 ──→ 角色knowledge
      触发：任务完成强制反哺 + 主对话判断"跨项目通用"
      方向：项目→角色（跨项目复用）
      产出：角色/<角色名>/knowledge/ 新增文档
      自动化：🟡 半自动（需要主对话验收时判断通用性）

路径3  角色经验 ──→ 共享知识
      触发：主对话判断"跨角色通用"
      方向：角色→共享（跨角色复用，通用化提炼）
      产出：共享知识/ 对应目录新增文档
      自动化：🟡 半自动（需要主对话判断通用性 + 去角色依赖泛化）

路径4  角色能力 ──→ 技能导出
      触发：角色经验积累到值得打包 / 用户主动要求导出
      方向：角色→技能（跨人复用，去环境依赖冻结快照）
      产出：技能/<技能名>/ 完整技能包（SKILL.md + references/ + scripts/ + templates/）
      自动化：🟠 手动（需要去环境依赖、泛化经验、补全说明）

路径5  角色进化 ──→ 技能自动更新
      触发：角色knowledge新增≥3文件 / hot-rules变更 / 定期巡检发现技能落后
      方向：角色→技能（增量更新，保持技能与角色同步）
      产出：技能/<技能名>/ 增量更新（只更新变化段落）
      自动化：🟡 半自动（自动检测+提示，用户确认后执行增量更新）
```

### 流动层级与覆盖范围

```
覆盖范围    1人1项目    1人多项目    多人多项目    跨人通用
           ─────────────────────────────────────────────→
路径1  项目踩坑→项目热规则    ✦
路径2  项目经验→角色knowledge      ✦
路径3  角色经验→共享知识                ✦
路径4  角色能力→技能导出                      ✦
路径5  角色进化→技能自动更新                  ✦（增量）
```

**知识流动的终极方向**：从"只有我知道、只在当前项目有用"到"任何人都能用、在任何环境都能跑"。

## 九、平台无关性原则

> 核心设计哲学：**知识在文件中，平台只是运行时**

### 问题

今天是OpenClaw，明天是Hermes，后天可能还有更强大的平台。
如果知识体系跟平台绑定，每次换平台就要重新调教——这违背了"架构永不改"的根本原则。

### 五条核心原则

#### 1. 纯文本优先

所有知识用标准Markdown存储，不依赖任何平台专有格式/API。

```
✅ 正确：知识写在 .md 文件中，任何人/任何平台都能读
❌ 错误：知识存在平台的"知识库"功能中，换个平台就没了
❌ 错误：知识依赖平台特定的API调用格式
```

#### 2. 知识与运行时分离

知识体系是"数据"，平台是"引擎"，换引擎数据不动。

```
类比：
  知识体系 = 数据库（存储了所有经验、规则、方法论）
  平台    = 数据库引擎（MySQL/PostgreSQL/SQLite...）
  换引擎  = 数据导出→导入新引擎，数据本身不变
```

具体含义：
- 知识文件不包含任何"怎么在当前平台执行"的指令
- 平台特定的执行逻辑放在适配层，不放知识层
- 任何新平台只需要实现适配层，知识文件原样复制即可

#### 3. 平台适配器模式

每个平台只需实现一个薄适配层，知识体系本身零修改。

```
知识体系（平台无关）
  ├── SOUL.md / TOOLS.md / 宪法层 / INDEX.md / hot-rules.md
  ├── 角色knowledge / 共享知识
  └── 技能SKILL.md + references/

  ↕ 最小接口（5个能力）

平台适配层（平台相关）
  ├── 文件读写：读/写Markdown知识文件
  ├── 任务派发：主对话→sub-agent的任务分配
  ├── 会话管理：session的创建/切换/销毁
  ├── 定时触发：心跳/巡检/定期任务的调度
  └── 通知机制：知识更新后的推送/自检
```

**每个平台实现这5个能力，就能跑整个知识体系。**

#### 4. 自描述性

每个知识文件都是自包含的——读它不需要先读过系统手册，新平台的新session拿起来就能用。

```
✅ 正确：hot-rules.md 每条规则都写明了"为什么"和"什么情况触发"
✅ 正确：INDEX.md 包含完整的共享规范引用，不需要额外查找
✅ 正确：SKILL.md 自带使用场景和前置条件说明

❌ 错误：规则写"参见TOOLS.md第3节"——新平台没有TOOLS.md
❌ 错误：规则写"按之前讨论的执行"——新session没有之前讨论的上下文
```

#### 5. 迁移接口标准化

知识体系的最小接口只有5个能力，任何平台实现这5个就能跑：

| # | 能力 | 说明 | 为什么是最小集 |
|---|------|------|---------------|
| 1 | 读文件 | 读取知识文件内容 | 知识在文件中，读是基础 |
| 2 | 写文件 | 写入/追加/编辑知识文件 | 反哺需要写，没有写就没有知识流动 |
| 3 | 派任务 | 主对话向sub-agent派发任务 | 多agent协作的必要条件 |
| 4 | 会话管理 | session的创建/切换/销毁 | 项目隔离和角色切换的基础 |
| 5 | 定时触发 | 按时间自动执行任务 | 宪法同步/巡检/反哺检查的驱动力 |

**5个能力=知识体系的完整运行条件。** 少一个都跑不起来，多一个都不是必须的。

### 迁移成本分析

| 知识类型 | 迁移成本 | 原因 |
|----------|---------|------|
| SOUL.md | 🟢 零成本 | 纯Markdown，无任何平台依赖 |
| TOOLS.md | 🟢 零成本 | 纯Markdown，铁律是文本不是代码 |
| 宪法层（共享知识/项目规范/） | 🟢 零成本 | 纯Markdown，只包含规范和规则 |
| 项目INDEX.md + hot-rules.md | 🟢 零成本 | 纯Markdown，自描述自包含 |
| 角色knowledge/ | 🟢 零成本 | 纯Markdown，经验文档无环境依赖 |
| 技能（SKILL.md + references/） | 🟡 低成本 | SKILL.md本身是纯Markdown，但scripts/可能包含平台特定脚本（如shell命令），需要重新封装 |
| 定时任务/心跳 | 🟠 需重新实现 | 每个平台的调度机制不同，但逻辑相同（读文件→检查→执行→写回），只需映射到新平台的调度API |
| 通知机制 | 🟠 需重新实现 | 宪法更新的自检逻辑相同，但推送方式依赖平台（OpenClaw用session消息，其他平台可能用webhook） |

**结论**：核心知识（占90%以上）全部零成本迁移，只有"执行层"需要适配。

## 十、与Extension Registry同构

> 知识体系的设计跟OpenDAW的Extension Registry是同构的——不是巧合，是设计哲学的一致

### 三重同构

```
Extension Registry（OpenDAW）：
  新功能 = 注册扩展，架构不改
  → 新插件注册到Plugin Host，宿主不改动
  → 新效果器注册到信号链，路由不改动

知识体系（知识架构）：
  新维度 = 注册路径，结构不改
  → 新角色注册到维度注册表，五层架构不改动
  → 新流动路径注册到流动规则，现有路径不改动

平台适配器（平台无关性）：
  新平台 = 注册适配层，知识不动
  → 新平台实现5个最小接口，知识文件原样复制
  → 新执行环境接入适配层，知识体系零修改
```

### 同构不是巧合

这三重同构共享同一个设计哲学：

> **开放体系的核心不是功能全，而是扩展成本低。**

- Extension Registry：加一个插件不需要改宿主代码 → 扩展成本低
- 知识体系：加一个维度不需要改现有结构 → 扩展成本低
- 平台适配：换一个平台不需要改知识文件 → 迁移成本低

**低扩展成本 = 长期存活率。** 一个每次扩展都要大改的体系，最终会因为改不动而死亡。

### 同构的实践意义

当你在任何一个体系中思考"怎么加一个新东西"时，答案都是一样的：

1. **不要改现有结构**——先问"能不能通过注册来扩展"
2. **定义清晰接口**——新东西只需要实现接口，不需要理解整个体系
3. **保持向后兼容**——新东西的加入不影响已有东西的运行

这个思维模型，三个体系通用。


## 自动沉淀机制

> 解决"聊出来的洞察没人记"的问题——不等用户提醒，发现就沉淀

### 问题

反哺机制只覆盖"任务完成后"，但对话中会产生高价值洞察：
- 新的认知模型（如"项目+角色+技能三顶帽子"）
- 新的机制设计（如"身份卡"、"失忆恢复"）
- 发现的体系漏洞（如"项目锁防止并发写"）
- 通用经验提炼（如"确认要分维度，改一个不影响另一个"）

这些如果等用户说"记一下"才沉淀，大部分会丢失。

### 自动沉淀触发条件

以下任一条件满足，立即沉淀，不需要用户提醒：

| 触发条件 | 例子 | 沉淀到哪 |
|----------|------|----------|
| 发现新认知模型/框架 | "三顶帽子"模型 | `./共享知识/项目规范/` 对应规范 |
| 设计出新机制 | 身份卡、项目锁 | `./共享知识/项目规范/协作规范.md` |
| 发现体系漏洞 | 并发写冲突 | 对应规范补规则 |
| 提炼出通用经验 | "确认分维度比合在一起好" | `./共享知识/` 或角色knowledge |
| 用户说"这点很重要"/"记一下"/"别忘" | 任何 | 用户指定的位置 |
| 踩坑第2次 | 同一错误出现2次 | 项目hot-rules → 角色knowledge |
| 用户疑惑解答产生新知识 | "技能只属于角色吗？"→引出技能三种归属 | `./共享知识/项目规范/` 对应规范 |

> **关键补充**：用户问的"疑惑"看似只是求知，但解答中产生的新认知/新分类/新模型同样是高价值知识，必须沉淀。不要只沉淀"大洞察"，答疑解惑中的新认知同样值得记。

### 沉淀原则

1. **即时**：发现就写，不等对话结束
2. **最小化**：只写核心洞察，不写对话过程
3. **归位**：写对地方（结构级→宪法层，操作级→TOOLS，项目级→INDEX/hot-rules）
4. **去重**：写前搜一下，已有则补充不重复建

### 与反哺闭环的关系

自动沉淀是反哺的**实时触发器**，和"任务完成后反哺"互补：

```
实时触发（对话中）：发现洞察 → 立即沉淀
定时触发（任务结束）：反哺段 → 检查遗漏 → 补沉淀
```

双重保障：实时捕获不遗漏，定时兜底不丢失。

## 竞品定位：vs Hermes Agent

> 沉淀时间：2026-05-09
> 来源：用户提问"Hermes 是否比我们强"的深度分析

### 核心差异

**Hermes 解决单体进化，我们解决群体智慧。** 这是本质区别，不是功能差异。

| 层次 | Hermes | 我们 |
|------|--------|------|
| 单体内化 | ✅ GEPA闭环学习、技能自动创建、periodic nudge | ⚠️ 半自动，靠铁律触发 |
| 跨角色流通 | ❌ 每个实例独立进化，经验不流通 | ✅ 角色桥梁+知识注册表 |
| 跨项目关联 | ❌ 无 | ✅ 联盟检测+项目INDEX |
| 体系治理 | ❌ 无 | ✅ 宪法层+测试体系 |
| 新会话恢复 | ⚠️ 记忆注入 | ✅ 启动发现+身份卡 |

### 单实例 vs 多实例

- **单 Hermes 实例**装我们体系 → 60%重复、30%小幅增强、10%真正有用（宪法层、INDEX结构）
- **多 Hermes 实例**装我们体系 → **质变**。Hermes 的致命弱点是"每个实例独立进化，A踩过的坑B还得再踩"

### 最佳组合

**Hermes 做单体内化引擎，我们做群体协作层。**

```
Hermes实例 → 自动学习变强 → 反哺到角色knowledge → 知识注册表 → 其他实例引用
```

Hermes 让每个 agent 自己变聪明，我们让一群聪明的 agent 不再重复造轮子。

### 我们的差异化机会

Hermes 50K star 但没解决"多实例知识流通"问题。这是我们的核心价值——**1+1>2 的群体智慧**，不是单个 agent 更强。

## 十一、知识升级路径：knowledge → hot-rules 自动升级

> v5.0新增：解决"记录经验≠经验自动生效"的核心问题

### 核心问题

knowledge是被动存储（要搜才有），hot-rule是主动注入（自动加载）。
"记录了经验"不等于"经验自动生效"——差距就在knowledge不会自动变成hot-rules。

### 升级触发：同类knowledge出现2次

- **不需要精确计数**：靠agent在沉淀时自检
- **沉淀时自检**：每次写knowledge前，检查是否已有类似记录
- **发现类似记录** → 标记`[二次]` → 建议升级为hot-rule

### 完整升级路径

```
踩坑/发现 → 记录到knowledge（被动存储）
          → 同类出现2次 → 自动提炼为hot-rule（半主动）
          → hot-rule内容同步到TOOLS.md项目热规则区 → 下次会话自动注入（主动生效）
```

### 更新后的知识流动全景图

```
知识流动全景（v5.0更新）：

路径1  项目踩坑 ──→ 项目hot-rules
      触发：同一个坑踩2次
      产出：项目/规划/hot-rules.md
      自动化：🟢 完全自动

路径2  项目经验 ──→ 角色knowledge
      触发：任务反哺+判断跨项目通用
      产出：角色/<角色名>/knowledge/
      自动化：🟡 半自动

路径3  角色经验 ──→ 共享知识
      触发：判断跨角色通用
      产出：共享知识/对应目录
      自动化：🟡 半自动

路径4  角色能力 ──→ 技能导出
      触发：经验积累值得打包
      产出：技能/<技能名>/
      自动化：🟠 手动

路径5  角色进化 ──→ 技能自动更新
      触发：knowledge新增≥3文件
      产出：技能/增量更新
      自动化：🟡 半自动

路径6  同类knowledge ──→ hot-rule ──→ TOOLS.md（新增）
      触发：沉淀时自检发现同类记录2次以上
      产出：项目hot-rules.md → TOOLS.md项目热规则区
      自动化：🟡 半自动（自动检测，沉淀时触发）
```

### hot-rule提炼格式规范

**不是复制knowledge内容，而是提炼为"触发条件→标准动作"**：

```
❌ 被动记录（knowledge风格）：
CI报错E0499，原因是trait bound没满足，解决方案是加where子句约束

✅ 主动规则（hot-rule风格）：
CI报E0499 → 检查trait bound，加where子句
```

### 降级机制

如果hot-rule连续10次任务无违反，说明已内化为习惯：
```
🔴铁律 → 连续10次无违反 → 🟡警告 → 连续10次 → 🟢提醒 → 移回knowledge
```

### 升级记录

在knowledge-registry.md的升级记录区域登记：
- 升级时间、来源knowledge、提炼为hot-rule、触发次数、状态
- 降级时标记"已降级"，保留历史记录

### 设计原则

1. **不增加认知负担**：升级检查集成到沉淀流程中
2. **格式简洁**：hot-rule一行能看懂，不需要读大段文字
3. **可追溯**：每次升级有记录
4. **可降级**：hot-rule过时了可以退回knowledge

## 仓库分层原则

> 新增时间：2026-05-11
> "框架与私有数据分离"的进阶版——更细化地处理"会膨胀的素材"问题

### 核心问题

体系仓库会膨胀失控。收藏导出、资源收集、项目源码等内容会持续增长，如果全放体系仓库，仓库体积无限膨胀、克隆变慢、搜索变慢、维护成本失控。

### 核心原则

**体系仓库=方法论（怎么干活），独立私有仓库=素材库（干活用的内容）。**

- 方法论精炼不膨胀——规范、流程、索引、使用指引
- 素材会增长放独立仓库——收藏数据、资源文件、项目源码
- 体系只保留**索引+使用指引**，需要素材时先查索引定位再从仓库取

### 判断标准

| 判断维度 | 放体系仓库 | 开独立私有仓库 |
|----------|-----------|--------------|
| 增长性 | 稳定或缓慢增长 | 会持续膨胀 |
| 性质 | 方法论（怎么干） | 素材（干活用的内容） |
| 查找方式 | 直接在体系内读 | 先查体系内索引，再去仓库取 |
| 典型内容 | 规范、流程、模板、索引、使用指引 | 收藏数据、资源集合、项目源码 |

**简单判断：这个内容会不会随时间越变越大？会→开独立仓库，不会→放体系。**

### 体系仓库中的索引格式

体系仓库中用索引文件指向独立私有仓库，格式示例：

**收藏索引.md**（指向 yuanbao-bookmarks）：
```
# 元宝收藏索引
> 完整数据在私有仓库 yuanbao-bookmarks
> 使用：git clone <repo-url> → 按分类查找
> 最近更新：2026-05-11
```

**资源清单.md**（在体系内引用 netdisk-resources）：
```
# 网盘资源清单
> 完整资源在私有仓库 netdisk-resources
> 使用：git clone <repo-url> → 按分类查找
> 最近更新：2026-05-11
```

**项目索引**（如 项目/PhoneAgent/）：
```
# PhoneAgent
> 源码在私有仓库 phone-agent
> 关联角色：系统开发者
> 关联技能：[无]
```

### 与"框架与私有数据分离"的关系

分层原则是**框架分离原则的进阶版**。框架分离原则解决了"通用框架 vs 私有数据"的分离问题，分层原则在此基础上更细化地处理"会膨胀的素材"问题——不是所有私有数据都需要独立仓库，只有**会持续增长的内容**才需要。框架分离是方向，仓库分层是落地策略。

## 经验唯一真相源原则

> 新增时间：2026-05-10
> 解决：经验重复存储导致不一致和维护负担

### 问题

同一个踩坑/经验可能出现多处：
- 项目hot-rules.md 写了
- 角色knowledge/ 也写了
- 共享知识/踩坑记录/ 又写了一遍

结果：
1. 更新时只改了一处，其他地方过时了
2. 新手看多个地方，信息不一致
3. 维护成本翻倍

### 原则

**踩坑/经验只存一处，由"性质"决定存哪：**

| 经验性质 | 存放位置 | 原则 |
|----------|----------|------|
| 通用经验（所有项目都可能遇到） | `共享知识/踩坑记录/` | 一处存储，处处引用 |
| 领域经验（特定角色才关注） | `角色/<角色名>/knowledge/` | 只有该角色用 |
| 项目特定经验（只跟这个项目有关） | `项目/规划/hot-rules/` | 只有该项目用 |
| 交接文档（部署文档/配置说明） | `项目/docs/knowledge/` | 一处存储，关联角色通过项目INDEX定位 |

### 引用规则

其他地方只写引用，不写内容：

```
→ 详见共享知识/踩坑记录/xxx.md
```

禁止：
- ❌ 双写：既引用又复制内容
- ❌ 三写：在项目/角色/共享知识各写一遍
- ❌ 同步：写完一处还要同步到其他地方

交接场景：
- ✅ 部署文档只存在项目/docs/knowledge/，运维角色通过项目INDEX"部署信息"段定位
- ❌ 把部署文档复制到运维角色knowledge/（双写=维护噩梦）
- ❌ 在运维RULES中内嵌部署步骤（过时快，更新不同步）

### 执行流程

沉淀新经验时：

```
1. 判断性质：通用？领域？项目特定？
2. 写入对应位置（唯一真相源）
3. 其他地方只写引用：`→ 详见 xxx`
```

### 违反检查

沉淀时自检：
- 这个经验之前存过吗？
- 应该存在哪个位置？
- 其他地方有重复吗？

如有重复，删除重复内容，保留引用。
