## 宪法与地方法的关系

**比喻**：共享知识库当"宪法"，INDEX.md当"地方法"。宪法改一次，全国生效；地方法只管本地特例。

```
共享知识/项目规范/    ← 宪法（改一次全局生效）
  ├── 目录结构规范.md
  └── 协作规范.md

各项目/INDEX.md      ← 地方法（引用宪法 + 声明本地特例）
  ├── 共享规范段 → 指向宪法
  └── 本地偏差段 → 写明特例+原因
```

**关键**：INDEX.md不复制规范原文，只引用。复制=版本分裂=最终不一致。

## 宪法自动同步机制

扣子架构下session记忆隔离，没法做到"改了宪法自动推送到所有session"。
但可以做到**被动自检**——每次session干活前自动发现宪法变了。

### 版本追踪法

1. **宪法层**：每个规范文件头写 `最后更新：YYYY-MM-DD`
2. **INDEX.md**：共享规范段写 `最后同步宪法：YYYY-MM-DD`
3. **自检逻辑**：任何session接手项目任务时 → 读INDEX.md → 比对宪法文件的 `最后更新` vs INDEX的 `最后同步宪法` → 不一致则先同步

```
INDEX.md 共享规范段：
→ ./共享知识/项目规范/（宪法层，改一次全局生效）
- 目录结构规范：最后同步 2026-05-09
- 协作规范：最后同步 2026-05-09
本项目已对齐：部分
```

### 谁负责同步？

| 场景 | 负责人 | 方式 |
|------|--------|------|
| 主对话派发任务给sub-agent | 主agent | 任务描述中包含"先检查宪法同步" |
| 项目session主动接手任务 | 自己 | 第一件事读INDEX → 检查宪法同步 |
| 定期巡检 | heartbeat | 检查各项目INDEX的同步日期 |
| 紧急变更 | 主对话 | 主动推送消息给项目session |

### 你（主人）不需要挨个提醒

流程是这样的：
1. 你跟主对话说"改个规范"
2. 主对话改宪法文件 → 更新宪法文件的`最后更新`日期 → 更新`项目规范/README.md`变更日志
3. 下次任何session碰这个项目 → 自检发现宪法更新 → 自动同步
4. 如果是紧急变更，主对话主动通知相关session

**你只需要说一次，不用跑断腿。**

## 四条铁律（重申）

1. **接手项目前先查共享知识**：`./共享知识/项目规范/` + 项目`INDEX.md`
2. **完成任务后更新INDEX**：新知识/新决策/新坑，都要记录
3. **一个坑只踩一次**：踩完写进冷知识；踩第2次→升级为热规则
4. **规范变更走宪法层**：改共享规范 → 通知各项目 → 逐步对齐
5. **🔴 改完就提交，不留未提交变更**：edit_file/write_file成功后立即 git add + commit，绝不在工作目录留未提交变更（其他智能体的rebase/merge会覆盖未提交内容，导致丢失）
6. **🔴 必须在worktree内工作，禁止在主工作目录直接编辑**：所有角色必须在 `open-ks-worktrees/<角色名>/` 内编辑和commit，主工作目录仅用于merge和sync。违反此条=暂存区被抢占=内容无声丢失（WO-057）
7. **🔴 宁可冗余不丢数据**：merge冲突时保留双方内容，绝不自动选一边放弃另一边。有冲突标记好过无声丢失文件内容

## 🔴 Git提交溯源规范（所有角色强制执行）

> 背景：多智能体共享同一仓库，必须通过commit信息溯源"谁在什么时候因为什么做了什么"，否则变更无法追踪、冲突无法定位。

### commit格式

```
[角色名] <类型>: <描述>
```

**字段说明**：

| 字段 | 说明 | 示例 |
|------|------|------|
| 角色名 | 当前执行任务的角色，用方括号包裹 | `[系统开发者]` `[测试工程师]` `[小龙]` |
| 类型 | feat/fix/docs/refactor/chore/style/test | `feat` `fix` `docs` |
| 描述 | 简明说明本次变更内容和原因 | `迁移skills/media-preview到技能/媒体预览技能` |

**完整示例**：
```
[系统开发者] feat: OpenGit P3 新增CLI管理工具+Repo Size API
[测试工程师] fix: 修正测试报告中重复计数
[小龙] fix: 迁移skills/media-preview到技能/媒体预览技能，关联10个角色RULES
[混音母带工程师] docs: 更新VST开发踩坑记录
```

### 铁律

1. **🔴 每次文件修改后立即commit**：不允许攒多个文件的变更一起提交（除非同一任务的多文件原子变更）
2. **🔴 commit必须带角色标识**：`[角色名]` 前缀不可省略
3. **🔴 commit描述必须说明原因**：不只是"改了什么"，还要"为什么改"
4. **🔴 原子变更一起提交**：同一任务涉及多个文件的关联修改（如迁移时改SKILL.md+INDEX.md+角色RULES），应一次commit包含所有关联文件
5. **🔴 必须在worktree内工作**：每个角色使用 `open-ks-worktrees/<角色名>/` 作为工作目录，主工作目录仅用于merge和sync（commit-msg hook v3.0强制拦截）
6. **🔴 精确add，禁止add .和add -A**：只add自己修改的文件，`add .`会把其他智能体正在改的文件带进你的commit
7. **🔴 merge冲突保留双方**：绝不自动选一边放弃另一边，宁可冗余不丢数据

### 违规后果

- 无角色标识的commit → 无法溯源责任人 → 按事故处理
- 未提交变更被覆盖 → 白干活 → 自己负责
- 主目录直接编辑 → 暂存区被抢占 → 内容无声丢失（WO-056/WO-057）
- 同一任务拆成多个零散commit → 难以追踪 → 尽量原子化

## 跨项目知识流通

```
项目A踩坑 → 写入共享知识/踩坑记录/
项目B遇到类似问题 → 搜索共享知识 → 直接复用解法
```

- **写**：踩坑/模式/经验 → 写入 `./共享知识/` 对应目录
- **读**：接到任务 → 先搜共享知识 → 有现成的就不重新造
- **更新**：发现共享知识过时 → 直接改，不改项目本地副本

|------|----------------|----------------|

## 完整知识层次

从上到下，知识越来越具体，范围越来越窄：

```
第0层  SOUL.md              ← 我是谁（身份/性格/原则）
       ↓ 所有行为受此约束
第1层  TOOLS.md              ← 我怎么工作（铁律/工具经验/安全规则）
       ↓ 所有任务受此约束
第2层  共享知识/项目规范/      ← 所有项目共享（宪法层）
       ↓ 所有项目受此约束
第3层  项目/INDEX.md          ← 单个项目知识（地方法+引用宪法）
       项目/规划/hot-rules.md ← 单个项目热规则（注入每个任务）
       项目/docs/knowledge/   ← 单个项目冷知识
       ↓ 单个项目受此约束
第4层  技能/             ← 操作手册（怎么执行复杂流程）
       ↓ 需要时加载
```

**每一层只管自己该管的事，不越界**：
- SOUL不管具体项目怎么建目录
- TOOLS不管项目技术选型
- 宪法不管某个项目的特殊决策
- INDEX不管通用操作流程
- 技能不管"为什么"，只管"怎么做"

## 技能的重新定位

有了宪法+INDEX+热规则体系后，技能的角色变了：

| 之前 | 现在 |
|------|------|
| 技能=项目百科全书 | INDEX.md=项目百科全书 |
| 技能=接手指南 | INDEX+热规则=接手指南 |
| 技能=唯一入口 | 技能=操作手册（可选） |

**技能还有必要，但定位缩小为"操作手册"**：

| 知识类型 | 存哪 | 干什么 |
|----------|------|--------|
| 项目全貌 | INDEX.md | 理解项目是什么 |
| 方向决策 | 规划/ | 理解为什么这么做 |
| 防踩坑 | hot-rules.md | 不犯已知错误 |
| 经验教训 | docs/knowledge/ | 需要时查阅 |
| **操作流程** | **技能/SKILL.md** | **复杂流程的step-by-step** |

技能的核心价值变成：
1. **复杂操作流程**：部署流程、发布流程、数据迁移——步骤多、顺序严格、容易漏
2. **可执行资源**：脚本、模板、配置——不只是知识，是能跑的东西
3. **跨项目复用**：open-dev-tools这种，多个项目共用一套操作

**判断标准**：如果一个操作流程超过5步且步骤顺序严格，写进技能；否则热规则一句话就够。

## 非项目实体：角色

不是所有会话都是"项目"。有两类实体：

| 类型 | 特征 | 例子 |
|------|------|------|
| 项目 | 有起止/有交付物/有规划 | OpenLink, 小说, 引流 |
| 角色 | 持续运行/跨项目/积累能力 | 开发者, ECS运维, 内容运营 |

角色不适合用项目框架：
- 不需要INDEX.md（不是知识枢纽）
- 不需要完整的目录结构（没有代码/测试）
- 不需要路线图（没有Phase）
- 但**需要自己的规则、知识、热规则**

### 两类角色

| 类型 | 特征 | 例子 | 积累什么 |
|------|------|------|----------|
| 服务型角色 | 维护基础设施、响应请求 | ECS运维、邮件管理 | 运维经验、故障记录 |
| 能力型角色 | 驱动项目、产出交付物 | 系统开发者、内容运营 | 跨项目方法论、实战经验 |

**能力型角色是之前缺失的**——它不是某个项目的百科，而是"我擅长什么、怎么干"的能力档案。

### 角色按领域定，不按语言定

> 核心原则：语言是工具，领域是身份。角色回答"我解决什么问题"，不回答"我用什么语言"。

**❌ 错误**：Rust开发者、Python开发者、C开发者 → 同一个人三个角色，知识割裂
**✅ 正确**：系统开发者（Rust✅ Python✅ C✅）→ 一个角色，领域知识统一，语言按需切换

角色定义包含**技能标签**，列出掌握的语言/工具：

```markdown
# 角色定义

- 领域：系统开发（网络协议 / 音频引擎 / 基础设施）
- 技能标签：Rust(精通) | Python(熟练) | C/C++(熟练) | Docker(基础)
- 核心能力：跨语言架构设计 / 性能优化 / 测试工程
```

**为什么按领域？**
- 音频引擎的知识（采样率/缓冲区/DSP管线）不因语言变——Rust写的和C写的是同一套领域知识
- workspace组织、编译优化这些语言技巧，是角色的**技能层**，不是**身份层**
- 换语言不换领域，角色不用重建；换领域不换语言，需要新角色

### 角色knowledge的组织

按领域组织，语言作为子目录：

```
./角色/系统开发者/
├── RULES.md
├── knowledge/
│   ├── 跨语言架构设计.md      ← 领域知识（不绑定语言）
│   ├── 测试工程.md            ← 领域知识
│   ├── rust/                  ← 语言技能
│   │   ├── workspace组织.md
│   │   ├── 编译优化.md
│   │   └── 错误处理模式.md
│   ├── python/                ← 语言技能
│   │   └── 包管理与虚拟环境.md
│   └── c-cpp/                 ← 语言技能
│       └── juce构建.md
├── hot-rules.md
├── templates/
│   ├── rust-workspace/
│   ├── python-project/
│   └── juce-plugin/
└── log/
```

**领域知识在上面，语言技巧在子目录**——换语言只影响子目录，领域知识不丢。

### 角色之间的重叠与共享

不同角色可能用同一种语言，知识会有重叠。处理原则：**重叠往上提，差异各自留**。

```
系统开发者（Rust）         应用开发者（Rust）
├── knowledge/             ├── knowledge/
│   ├── rust/              │   ├── rust/
│   │   ├── workspace组织  │   │   ├── workspace组织  ← 相同
│   │   ├── 编译优化       │   │   └── Tauri开发.md   ← 不同
│   │   └── 错误处理       │   │
```

**相同部分**：workspace组织、编译环境、错误处理——这些是Rust语言通用的，不该两个角色各写一份。

**解决方案**：语言通用知识提到**角色knowledge/的语言子目录**，具体角色的语言子目录只写特有内容。读取时合并。

```
系统开发者 knowledge/rust/
├── workspace组织.md        ← 通用，两个角色都读
├── 编译优化.md              ← 通用
└── 错误处理模式.md          ← 通用

应用开发者 knowledge/rust/
└── tauri开发.md             ← 特有，只有应用开发者用
```

**判断标准**：
- "Rust的workspace怎么组织" → 通用语言知识 → 放谁那都行，另一个角色引用或共享
- "Tauri怎么搭桌面应用" → 应用特有 → 只放应用开发者
- "高性能网络IO怎么写" → 系统特有 → 只放系统开发者

**不建"Rust通用角色"**——语言不是角色，语言是技能标签。通用语言知识放在先建立的角色里，后来角色通过引用或复制获取。

**避免知识分裂**：当发现两个角色有相同knowledge文件时：
1. 检查内容是否真的一样（可能细节不同）
2. 如果一样 → 保留一份，另一个引用
3. 如果有差异 → 合并成一份，差异用条件标注（如"系统项目用X方案，应用项目用Y方案"）

### 角色与项目的互哺关系

```
能力型角色（系统开发者）
  ├── 驱动 → 项目A（OpenLink，Rust）→ 积累路由协议+Extension Registry经验
  ├── 驱动 → 项目B（OpenVault，Rust）→ 复用Rust技能，项目A的坑不用再踩
  └── 驱动 → 项目C（OpenDAW-core，Rust+Python+C）→ 复用系统架构经验+音频领域知识

项目A踩坑 → 写入项目hot-rules（项目内防犯）
          → 提炼通用经验 → 写入角色knowledge（跨项目复用）

角色knowledge = "系统项目的跨语言架构怎么组织"
项目hot-rules = "OpenLink必须用~/.cargo/bin/cargo"
```

**互哺循环**：
1. 角色驱动项目 → 项目产出成果 + 踩坑经验
2. 项目踩坑 → 项目级热规则防犯（范围=本项目）
3. 通用经验提炼 → 角色级知识复用（范围=同角色所有项目）
4. 角色升级 → 下一个项目起点更高

### 复用的关键：角色让新项目"站在前人肩膀上"

没有角色体系时：
```
新项目启动 → 从零摸索 → 重复踩坑 → 重复解决 → 重复记录
```

有角色体系后：
```
新项目启动 → 角色自带方法论 → 跳过已知坑 → 只解决新问题 → 经验反哺角色
```

**具体例子**：我开发OpenLink时学到的Rust workspace组织方式、编译环境配置、Extension Registry设计模式——这些不全是OpenLink特有的。workspace组织是"Rust语言技能"，编译环境配置是"系统开发通用经验"，Extension Registry是"架构设计模式"。下次开发OpenVault，Rust技能直接复用；开发OpenDAW，架构模式直接复用；哪怕换成C语言，架构思维也不变。

### 角色的知识管理

```
./角色/<角色名>/
├── RULES.md          ← 角色规则（这个角色的"宪法"）
├── knowledge/        ← 角色知识
│   ├── 领域知识.md    ← 跨语言通用经验/方法论/最佳实践
│   ├── <语言>/       ← 语言技能（rust/ python/ c-cpp/ ...）
├── hot-rules.md      ← 角色热规则（每次执行注入）
├── templates/        ← 角色模板（按语言分子目录）
└── log/              ← 操作日志（做了什么/何时做的）
```

**templates/按语言分**——不同语言有不同的脚手架。比如系统开发者角色有rust-workspace/、python-project/、juce-plugin/三套模板。

### 服务型角色示例：ECS运维

```
./角色/ECS运维/
├── RULES.md          ← 运维铁律（deploy脚本部署/禁止手动compose/内存1.8G限制...）
├── knowledge/        ← 服务器配置/运行服务清单/Docker网络/端口映射
├── hot-rules.md      ← CARGO_BUILD_JOBS=2/出站1Mbps/不重启生产容器...
└── log/              ← 运维操作日志
```

### 能力型角色示例：系统开发者

```
./角色/系统开发者/
├── RULES.md          ← 开发铁律（核心零业务逻辑/先check后push/测试先行...）
├── knowledge/
│   ├── 跨语言架构设计.md  ← 领域知识（Extension Registry/存储抽象层/模块解耦...）
│   ├── 测试工程.md        ← 领域知识（测试分层/CI策略...）
│   ├── rust/              ← Rust语言技能
│   │   ├── workspace组织.md
│   │   ├── 编译环境配置.md
│   │   └── 错误处理模式.md
│   ├── python/            ← Python语言技能
│   └── c-cpp/             ← C/C++语言技能
├── hot-rules.md      ← cargo版本/镜像配置/编译目录/OOM防护...
├── templates/
│   ├── rust-workspace/
│   ├── python-project/
│   └── juce-plugin/
└── log/              ← 开发日志
```

**领域知识在上面，语言技能在子目录**。换语言只改子目录，领域经验完整保留。

### 角色会话的启动流程

不同于项目的"读宪法→建目录→写INDEX"，角色会话启动：
1. 读自己的RULES.md（角色铁律）
2. 读自己的hot-rules.md（角色热规则）
3. 读knowledge/了解角色积累的经验
4. （如果是新项目）用templates/起步
5. 准备好接受任务

不需要INDEX.md——角色是"能力提供者"不是"知识枢纽"。
不需要宪法同步——角色不归属任何项目，它自己的RULES.md就是它的宪法。

### 角色知识反哺

角色在驱动项目的过程中，持续提炼通用经验：
- OpenLink踩了cargo版本坑 → 写入OpenLink hot-rules（项目级）→ 提炼"Rust版本管理"写入系统开发者knowledge/rust/（角色语言技能）
- OpenVault遇到存储后端抽象问题 → 解决后提炼"trait设计模式"写入系统开发者knowledge/（角色领域知识）
- OpenDAW-core需要C++构建JUCE → 提炼"JUCE构建模式"写入系统开发者knowledge/c-cpp/（角色语言技能）
- 任何项目发现CI问题 → 写入共享知识/踩坑记录（全局级）

**三层反哺**：
```
项目hot-rules  → 项目内防犯（最具体）
角色knowledge  → 同角色项目复用（跨项目）
共享知识/踩坑   → 所有项目共享（最通用）
```

### knowledge → hot-rules 自动升级路径

> v5.0新增：解决"记录了经验但不自动生效"的问题

**核心问题**："记录经验"≠"经验自动生效"。knowledge是被动存储（要搜才有），hot-rule是主动注入（自动加载）。

**升级触发：同类knowledge出现2次**
- 沉淀时自检：每次写knowledge前，搜索是否已有类似记录
- 有类似记录 → 标记`[二次]` → 建议升级为hot-rule
- 不需要精确计数，靠agent在沉淀时自检

**升级路径**：
```
踩坑/发现 → 记录到knowledge（被动存储）
          → 同类出现2次 → 自动提炼为hot-rule（半主动）
          → hot-rule内容同步到TOOLS.md项目热规则区 → 下次会话自动注入（主动生效）
```

**hot-rule提炼格式**（不是复制knowledge，而是提炼）：
```
❌ 被动记录：CI报错E0499，原因是trait bound没满足，加where子句修复
✅ 主动规则：CI报E0499 → 检查trait bound，加where子句
```

**条件→动作模板**：
```
触发条件 → 标准动作
CI报E0499 → 检查trait bound，加where子句
FastAPI单例污染 → uuid唯一名隔离
write_file报"路径无效" → 用bash heredoc替代
```

**降级机制**：
- hot-rule连续10次无违反 → 可降级回knowledge
- 降级时保留原knowledge内容，只是不再主动注入

### 角色与技能的区别

| 维度 | 角色 | 技能 |
|------|------|------|
| 本质 | "谁在干"——能力+经验+方法论 | "怎么干"——操作步骤+可执行资源 |
| 范围 | 跨项目 | 可跨项目也可项目专属 |
| 积累 | 越用越强（经验累积） | 按需加载（静态手册） |
| 生命周期 | 持续存在 | 任务级加载 |
| 类比 | 资深工程师 | 操作手册 |

**角色比技能更"活"**——角色有经验积累，技能只是静态流程。一个角色可以掌握多个技能。

### 技能的三种归属

技能不专属于角色，按使用范围分三种：

| 归属 | 特征 | 存放位置 | 例子 |
|------|------|----------|------|
| 角色技能 | 跟着能力走，谁有这能力谁用 | `./技能/<技能名>/`，角色INDEX引用 | rust-expert（从系统开发者导出） |
| 项目技能 | 跟着项目走，只有这项目用 | `./技能/<技能名>/`，项目INDEX引用 | 备份打包、文件清理 |
| 通用技能 | 谁都能用，不绑定角色也不绑定项目 | `./技能/<技能名>/` | agent-browser、ppt_create |

**存放位置相同**（都是`./技能/`），区别在于**谁引用它**：

```
项目INDEX.md 关联技能段：
  - 备份打包技能 → 项目专属
  - open-dev-tools → 项目关联

角色RULES.md 推荐技能段：
  - rust-expert → 角色导出

谁都能加载：
  - agent-browser → 通用
```

**判断标准**：
- 只有特定项目用 → 项目技能，项目INDEX引用
- 只有特定角色用 → 角色技能，角色RULES引用
- 谁都用得上 → 通用技能，不特别引用

## 平台迁移指南

> 知识体系是平台无关的，迁移=复制文件+实现适配层

### 新平台接入3步法

#### Step 1：实现5个最小接口

任何智能体平台，只要实现以下5个能力，就能跑整个知识体系：

| # | 能力 | 说明 | 验证方法 |
|---|------|------|----------|
| 1 | 读文件 | 读取Markdown知识文件内容 | 能读SOUL.md并理解其内容 |
| 2 | 写文件 | 写入/追加/编辑Markdown文件 | 能向hot-rules.md追加一条规则 |
| 3 | 派任务 | 主对话向sub-agent派发任务 | 能把一个子任务分配给独立session |
| 4 | 会话管理 | session的创建/切换/销毁 | 能为不同项目创建隔离的session |
| 5 | 定时触发 | 按时间自动执行任务 | 能设置每日巡检或宪法同步检查 |

**验证标准**：5个能力全部可用 = 平台适配完成

#### Step 2：复制知识文件

将知识体系目录原样复制到新平台的工作空间：

```

  ./基础设定/SOUL.md
  ./基础设定/TOOLS.md
  ./共享知识/                 ← 整个目录（宪法层+踩坑记录+项目规范）

  通知机制                   ← 逻辑相同，推送方式映射到新平台
```

#### Step 3：验证首任务闭环

复制完成后，执行一个完整的任务闭环验证：

```
验证流程：
1. 新平台创建session → 读取SOUL.md + TOOLS.md → ✅ 读文件可用
2. 派发一个简单子任务 → sub-agent执行并完成 → ✅ 派任务+会话管理可用
3. sub-agent向hot-rules.md追加一条规则 → ✅ 写文件可用
4. 设置一个5分钟后的定时检查 → 触发执行 → ✅ 定时触发可用
5. 修改宪法文件 → 检查项目session是否发现并同步 → ✅ 通知机制可用

5步全过 = 迁移成功，知识体系在新平台完整运行
```

## 全层级版本同步机制

> 解决"我改了TOOLS铁律，已开启的会话不知道"的问题

### 问题

宪法版本追踪法只管T2（宪法）和T3（INDEX）的同步。但T0/T1（SOUL/TOOLS）的变更影响更大——铁律改了，已开启会话不知道，继续按旧规矩干。

### 解法：主对话广播 + 会话自检

扣子架构下没有"推送到所有session"的机制，但有两个替代方案：

#### 方案1：主对话主动广播（即时生效）

当T0/T1/T2发生变更时，主对话负责通知已开启的会话：

```
变更发生 → 主对话发现变更 → 向所有活跃session发sessions_send：
  "⚠️ 体系更新通知：[变更内容摘要]
   请重新读取 ./基础设定/TOOLS.md（或SOUL.md / 共享知识/）
   更新你的行为规则。"
```

**操作方式**：
1. 主对话改了SOUL/TOOLS/宪法 → 记录变更
2. 用sessions_list查看当前活跃的session
3. 用sessions_send逐个通知
4. 被通知的session重新读取对应文件

#### 方案2：心跳自检（定期生效）

在heartbeat.md中加入体系同步检查项：

```
检查清单
- 比对 ./基础设定/TOOLS.md 的版本号（首行），与上次读取时是否一致
- 比对 ./共享知识/项目规范/ 各文件的最后更新日期，与INDEX.md记录是否一致
- 如有不一致，重新读取并更新行为
```

**适用场景**：有长时间运行的session，通过心跳机制定期刷新认知。

#### 方案3：任务交接时自检（最轻量）

任何session接到新任务时，先做一次快速自检：

```
1. 读 ./基础设定/TOOLS.md 首行版本号
2. 与自己记忆中的版本比对
3. 如不一致 → 重新读取全文
4. 对 ./共享知识/项目规范/ 各文件做同样检查（比对最后更新日期）
```

### 版本标识规范

所有规范文件必须有版本标识，方便快速比对：

| 文件 | 版本标识位置 | 格式 |
|------|------------|------|
| TOOLS.md | 首行 | `### TOOLS.md vX.XX` |
| SOUL.md | 首行 | 自然不需要版本号（很少改） |
| 宪法层文件 | 第二行 | `> 最后更新：YYYY-MM-DD` |
| INDEX.md | 第二行 | `> 最后更新：YYYY-MM-DD` |

### 实际操作

**主人改了体系后，只需要跟我说一句：**

```
通知所有会话更新规范
```

我会：
1. 查看当前活跃的session列表
2. 逐个发sessions_send通知更新
3. 记录哪些session已通知、哪些未响应

**如果忘了说也没关系**——心跳自检和任务交接自检是兜底，不会永远过期。


## 项目锁机制

> 解决"两个session同时写同一个项目"的冲突问题

### 问题

两个session同时操作同一个项目，会导致：
- **文件覆盖**：后写的覆盖先写的，代码/知识丢失
- **Git冲突**：两个session同时push，第二个失败或merge conflict
- **INDEX损坏**：两个session同时改INDEX.md，反哺内容丢失
- **编译不可预测**：一个在编译一个在改代码

### 规则：同一项目同一时刻只有一个写者

**核心原则**：项目锁是"写锁"——同时读没问题，同时写会出事。

```
读（查资料/看状态）→ 随便读，不需要锁
写（改代码/改文件/push）→ 必须先拿锁，拿不到就等
```

### 锁文件格式

每个项目目录下放一个 `.项目锁` 文件：

```markdown
# 项目锁 + 身份卡

- 持有者：[session ID 或 "主对话"]
- 角色：[角色名 或 "无"]
- 项目INDEX：[路径，如 ./项目/OpenLink/INDEX.md]
- 关联技能：[技能名 或 "按需"]
- 锁定时间：YYYY-MM-DD HH:MM
- 正在做什么：[一句话描述任务]
- 预计多久：[大概时间，如"30分钟"]
```

**位置**：`./项目/<项目名>/.项目锁`

**双重功能**：既当项目锁（防并发写），又当身份卡（防session失忆）。

### 拿锁流程

```
1. 检查 ./项目/<项目名>/.项目锁 是否存在
2. 不存在 → 创建锁文件 → 开始干活
3. 存在 → 检查锁定时间：
   a. 锁定时间 < 30分钟前 → 别人在干活 → 等或换任务
   b. 锁定时间 ≥ 30分钟前 → 锁可能过期（session挂了没释放）
      → 先用 sessions_list 确认该session是否还在运行
      → 还在运行 → 等
      → 已结束/不存在 → 锁过期，覆盖拿锁
4. 拿到锁后，每隔15分钟更新一次锁定时间（如果任务还在跑）
```

### 释放锁

```
任务完成 → 删除 .项目锁 文件
任务中断/失败 → 也要删除 .项目锁（别占着茅坑）
```

### 特殊场景

| 场景 | 处理方式 |
|------|----------|
| 主对话派sub-agent干活 | 主对话代为管理锁，任务描述中注明"完成后删除.项目锁" |
| 两个session只读不改 | 不需要锁，随便读 |
| 两个session改不同文件 | 可以并行，但git push必须串行（先push的先，后push的先pull） |
| 紧急修复需要插队 | 主对话通知锁持有者释放锁，然后插队拿锁 |

### 主对话的职责

主对话是"锁管理员"——派发任务时负责：
1. 检查目标项目是否被锁
2. 如果被锁，等锁释放后再派发
3. 任务描述中加上"完成后删除 .项目锁"
4. 如果发现过期锁，主动清理

### 与角色单例的关系

| 机制 | 防什么 | 粒度 |
|------|--------|------|
| 角色单例 | 同一角色两个session | 角色级 |
| 项目锁 | 同一项目两个写者 | 项目级 |
| 两者配合 | 角色单例防止重复建角色，项目锁防止重复写项目 | 互补 |

**一个session可能同时持有角色+项目锁**——比如"Rust开发者"session在开发OpenLink，同时持有Rust开发者角色和OpenLink项目锁。

### 启动发现（取代"失忆恢复"）

> 核心洞察：新session不知道自己"失忆"了，因为它是空白人格，不觉得丢了什么。
> 所以不能用"失忆后恢复"的思路，要用"启动时发现"——每个session启动都扫描世界状态。

每个session启动时，无论新旧，都执行启动发现：

```
1. 扫描 ./项目/*/.项目锁 — 有锁=有人在干活
2. 扫描 ./角色/*/.活跃 — 有活跃标记=有角色在运行
3. 根据扫描结果决定起点：
   - 有活跃身份 → 读取身份卡 → 认领或协调
   - 全空 → 全新起步，走入口
4. 把自己的身份写入.项目锁或.活跃（身份卡机制）
```

**不依赖session"意识到自己失忆"**——而是把扫描作为启动流程的一部分，和"读SOUL/TOOLS"一样自然。


## 任务进度汇报规范

> 解决"开干之后十分钟没动静，不知道在干活还是卡住了"的问题

### 问题

sub-agent接到任务后可能长时间无响应，用户无法判断：
- 在干活（只是步骤多耗时久）
- 卡住了（网络/编译/部署失败）
- 陷入死循环（反复重试同一个错误）
- 偷懒（等用户催才动）

长时间沉默 = 焦虑 = 用户反复催 = 浪费token

### 规则：进度必报

#### 1. sub-agent汇报节奏

| 时机 | 必须汇报 | 内容 |
|------|----------|------|
| 接到任务 | ✅ | "收到，开始干 [任务摘要]，预计分N步" |
| 每完成一个关键步骤 | ✅ | "第X/N步完成：[做了什么]，下一步：[要做什么]" |
| 遇到阻塞/报错 | ✅ 立刻 | "⚠️ 第X步阻塞：[错误信息]，正在[换方案/等依赖/需要决策]" |
| 同一步骤超过3分钟 | ✅ | "还在处理第X步：[当前在干什么]，预计还需[Y分钟]" |
| 任务完成 | ✅ | "完成：[结果摘要]，[产出物位置]" |

**沉默上限：5分钟**——超过5分钟没有任何输出，视为异常。

#### 2. 主对话主动查状态

主对话派发任务后，如果超过5分钟没收到sub-agent反馈：

```
1. 用 sessions_status 检查sub-agent状态
2. 用 sessions_send 发一条："当前进度？遇到什么问题？"
3. 如果sub-agent无响应且状态为running → 可能卡住，考虑abort
4. 如果sub-agent状态为idle → 任务已结束但没汇报，查产出文件
```

#### 3. 任务派发模板更新

派发sub-agent任务时，在描述末尾加进度要求：

```
【进度要求】
- 每完成一个关键步骤必须汇报
- 遇到阻塞/报错立刻汇报
- 同一步骤超过3分钟汇报当前状态
- 完成后必须汇报结果摘要+产出物位置
```

#### 4. 避免过度汇报

汇报是给用户看的，不是写日志：
- ✅ "第3/5步完成：编译通过，98个测试全绿。下一步：git push"
- ❌ "正在读取文件foo.rs的第42行..."（太碎，没必要）
- ✅ "⚠️ 编译失败：错误信息xxx，正在换方案B"
- ❌ "错误！错误！救命！"（无信息量）

**原则：每个汇报都要有信息量，能让人判断是"在推进"还是"卡住了"。**


## 技能三种归属

技能不止属于角色，按使用范围分三种：

| 归属 | 跟谁走 | 例子 | 谁引用 |
|------|--------|------|--------|
| 角色技能 | 跟能力走 | rust-expert（从系统开发者导出） | 角色RULES引用 |
| 项目技能 | 跟项目走 | 备份打包、文件清理、open-dev-tools | 项目INDEX引用 |
| 通用技能 | 谁都能用 | agent-browser、ppt_create | 不需要特别引用 |

**存放位置都一样**（`./技能/`），区别在于**谁引用它**——项目INDEX引用项目技能，角色RULES引用角色技能，通用技能不需要特别引用。

### 技能发现顺序（与技能确定漏斗一致）

```
1. 项目INDEX关联技能 → 项目专属
2. 角色knowledge/hot-rules → 角色能力
3. 扫描自定义技能目录 → 通用可用
4. 写"待定" → 后续补充
```

所以技能发现不是只看角色，是先看项目INDEX → 再看角色 → 再扫描目录。

### 跨角色知识发现：知识注册表

> 解决"新角色不知道前辈存了什么语言知识"的问题

**问题**：角色A积累了Rust/Python知识，角色B也需要——但B不知道A有，不会自动引用。

**解法**：`./共享知识/knowledge-registry.md`，一个中央注册表，记录所有角色的语言知识位置。

```
知识注册表格式：
| 语言/技能 | 持有角色 | 知识路径 | 内容概要 |
|-----------|---------|----------|---------|
| Rust      | 系统开发者 | ./角色/系统开发者/knowledge/rust/ | workspace组织、编译配置、Axum经验 |
| Python    | 系统开发者 | ./角色/系统开发者/knowledge/python/ | （暂空） |
| C/C++     | 系统开发者 | ./角色/系统开发者/knowledge/c-cpp/ | juce构建 |
```

**自动发现流程**：

```
新角色创建/角色入体系时：
1. 读角色RULES.md的"技能标签"段
2. 对每个语言/技能标签 → 查知识注册表
3. 发现已有知识 → 自动在角色knowledge/<语言>/下创建引用文件
   引用文件内容：→ ./角色/<持有角色>/knowledge/<语言>/
4. 角色特有知识写在引用文件之外
```

**引用文件示例**：

```markdown
# Python语言技能

→ 通用Python知识：./角色/系统开发者/knowledge/python/

以下为本角色特有的Python知识：
（小程序开发特有的Python经验写在这里）
```

**注册维护规则**：
- 角色新增语言知识 → 更新知识注册表对应行
- 新角色创建 → 读注册表 + 自动引用 + 注册自己的标签
- 注册表是**追加不改写**——新行追加，旧行更新内容概要

**与维度注册制的关系**：知识注册表本身就是"维度注册制"的一个实例——知识路径是注册的，不是硬编码的。


### 项目间知识流动：角色是桥梁，项目不直连

> 项目不需要自己的知识注册表，因为角色就是项目间的知识桥梁

**核心逻辑**：角色回答"谁在建"，项目回答"在建什么"。跨项目的知识流动，走"谁在建"这条线——同一角色驱动不同项目，就是知识复用的自然管道。

```
项目A ←→ 角色 ←→ 项目B    ✅ 角色做桥梁，知识归位清晰
项目A ←→ 项目B             ❌ 直接连，角色层变多余，两个注册表打架
```

四种跨项目知识场景都有出路：
- 同角色跨项目 → 角色knowledge
- 不同角色同语言 → 知识注册表
- 跨项目架构模式 → 共享知识/设计模式/
- 项目A直接引用项目B → INDEX关联项目 + 直接读

**结论**：不给项目建注册表，跟软件架构里"模块通过接口通信不直接依赖"是同一个道理。


### 项目联盟：声明式，不是契约式

> 联盟关系是一方声明即可，不需要对方批准，也不会冲突

**核心原则**：项目A在INDEX.md声明"关联项目：项目B"，这只是A的视角说"我跟B有关系"，B不需要批准，B也不需要同步声明。

```
单方向盟：OpenDAW声明关联OpenLink → OpenDAW的session会主动关注OpenLink知识
双方向盟：OpenLink也声明关联OpenDAW → 双方session互相关注，知识流动更畅通
无盟约：双方都不声明 → 知识只通过角色knowledge间接流动（也能走，只是慢一点）
```

**不会冲突的原因**：
- 联盟声明只是"我知道你存在"的标记，不改变任何一方的目录结构或规则
- 两个项目用同一角色驱动 → 角色knowledge是共享管道（已有机制）
- 两个项目有代码依赖 → 各自INDEX记录，不互相入侵
- 新项目声明已有项目为联盟 → 已有项目零感知，零负担

**双向联盟的好处**（但不强制）：
- 两个项目的session互相关注对方的变化
- 宪法更新时，双向联盟的项目同步更及时
- 踩坑经验通过角色knowledge流得更快

**结论**：联盟是"我知道你"不是"我绑定你"，声明即生效，不声明也能活。


### 联盟关系自动检测

> 项目关系是演化出来的，不是一开始就确定的。系统应该能发现关系，但不自动绑定

**检测信号**（任一出现，说明两个项目可能有联盟关系）：

| 信号 | 例子 | 检测方式 |
|------|------|---------|
| 代码依赖 | OpenDAW引用open-storage | INDEX的技术架构段提到另一个项目 |
| 共享crate/包 | OpenLink和OpenVault都用open-storage | 发现相同的共享依赖 |
| 知识交叉引用 | 开发OpenDAW时读了OpenLink的INDEX | session行为记录 |
| 架构模式复用 | Extension Registry模式跨项目使用 | 角色knowledge中出现多项目引用 |
| 同角色驱动 | 系统开发者同时驱动OpenLink和OpenVault | 同一角色活跃在多项目 |
| 用户对话中提到 | "跟OpenLink一样的设计" | 对话关键词检测 |

**检测流程**（半自动，不自动绑定）：

```
1. 反哺时检查：本次任务是否涉及其他项目？
   - 读/引用了其他项目的知识
   - 代码依赖了其他项目的产出
   - 用户提到了其他项目名
   
2. 发现关联信号 → 检查INDEX的"关联项目"是否已声明
   - 已声明 → 无需操作
   - 未声明 → 建议："⚠️ 检测到本项目与[项目B]有关联，是否加入联盟？"

3. 等用户确认后才写入INDEX
```

**为什么不能自动关联**：
- 可能是误检测（两个项目恰好用同一种语言，不一定有联盟关系）
- 联盟声明是项目视角的判断，需要人的确认
- 自动关联可能导致"万物互联"——所有项目都关联所有项目，等于没有联盟

**检测时机**：
- 任务反哺时（每次任务的强制反哺段加一项检查）
- 定期巡检时（heartbeat检查各项目INDEX的关联项目 vs 实际依赖）

**关键**：检测是自动的，关联是手动的。发现关系是系统的活，确认关系是人的活。

## sub-agent调用规范

> 角色通过交接台协作，主agent通过sub-agent调度。搞清楚什么场景用什么方式。

### 调度方式对比

| 方式 | 适用场景 | 特点 |
|------|---------|------|
| `sessions_spawn` | 独立任务，需要完整上下文 | 独立会话，异步执行，结果回调 |
| 直接对话 | 简单问答，无需外部执行 | 阻塞当前会话，即时响应 |
| `calendar_create` | 定时/重复任务 | 到时间触发，独立会话执行 |
| `heartbeat` | 轻量周期检查 | 30分钟一次，秒级完成 |

### sessions_spawn 使用规范

1. **什么时候用**：任务需要多步骤/外部工具/耗时处理/产出可验收的
2. **什么时候不用**：简单问答/即时回复/一句话能回答的
3. **任务描述必须包含**：项目背景、关键人物/对象、文件位置、产出要求、结果发送目标
4. **同一任务复用**：已有相关sub-agent在跑→用`sessions_send`继续，不新建
5. **跨角色调用**：主agent spawn目标角色→task里写清背景和要做什么→角色在sub-agent里执行

### 跨角色协作流程

```
角色A发现需要角色B做事 → 提工单到交接台 → 主agent看到工单 → spawn角色B → task引用工单 → B执行完汇报
```

**不跳过交接台**：直接spawn角色B而不提工单=没有记录=下次不知道谁做了什么。工单是协作的日志。

### 反哺检查
- sub-agent执行完返回结果时，主agent必须检查反哺情况

## 跨角色反哺规范

> 来源：BUG-031 → 体系级规范——所有关联角色必须建立相互反哺机制
> 核心原则：协作要让彼此越来越强，不能只是内循环自强，还要角色间相互促进，这才是有效闭环。

### 为什么必须体系级规范

反哺靠各角色自己写=不写。网文三角色（写手/质检/运营）已落地反哺，但其余角色全是空白。根因不是"忘了"，而是模板无必填项+注册无检查门+成熟度无考核。

### 反哺三件套

每次跨角色反哺必须包含以下三要素（缺一=未闭环）：

| 要素 | 说明 | 示例 |
|------|------|------|
| 📋 工单/物料 | 通过交接台工单或物料交接传递 | `handover.sh work "平台反馈-字数超限" "网文运营" "网文质检员"` |
| 📝 knowledge沉淀 | 反哺内容摘要写入接收方knowledge | 质检员将共性问题模式写入写手knowledge |
| ✅ 通知确认 | 接收方确认收到并回应 | 工单被领取/knowledge被读取/回复确认消息 |

**闭环标准**：反哺发出→对方收到→对方回应→闭环。任何一环断开=未完成。

### 反哺方向规则

1. **双向必须**：A→B有反哺机制，B→A也必须有（方向可以不同，但必须存在）
2. **只填有关联的**：没有实际协作关系的角色之间不需要硬凑反哺
3. **触发条件明确**：不能写"需要时反哺"，要写"质检不合格时反哺"
4. **新增角色必声明**：创建新角色时，必须在RULES"跨角色反哺"段声明与哪些角色关联

### 反哺闭环率

**定义**：已完成闭环的反哺次数 / 应反哺的总次数 × 100%

**统计方式**：交接台工单状态 + knowledge更新记录

**考核标准**：

| 闭环率 | 评级 | 协作能力维度分数上限 |
|--------|------|---------------------|
| ≥90% | 优秀 | 可评5分 |
| 70-89% | 良好 | 可评4分 |
| 50-69% | 合格 | 可评3分 |
| <50% | 不合格 | 不超过2分 |

### 新角色注册检查门

角色INDEX注册时，必须同时完成：

1. ✅ RULES.md含"跨角色反哺"段
2. ✅ 关联角色和反哺方向已声明
3. ✅ 每个关联方向都有明确的触发条件
4. ❌ 缺少以上任一项 → 不允许注册

### 现有角色补全规则

- 按实际协作关系补，不硬凑
- 暂无关联角色的写"暂无关联角色，待项目启动后补充"
- 有了协作关系后必须补充
- 结果不符合要求→sessions_send要求修正，不自己动手

## 工单交付链

> 跨角色协作时，工单之间有前后依赖关系。交付链让流转关系可见。

### 规则

- 工单文件底部可加 `## 交付链` 段，标注前序/后序工单
- 格式：
  ```markdown
  ## 交付链
  - 前序：WO-XXX（说明依赖关系）
  - 后序：WO-YYY（说明下一步）
  ```
- agent接单时看到交付链→知道自己在这条链的什么位置→做完后主动通知后序角色
- 创建工单时已知前后关系→在工单文件里写好；不确定→留空，后续补

### 典型交付链

```
开发(WO-001) → 测试(WO-002) → 部署(WO-003)
需求(WO-010) → 设计(WO-011) → 开发(WO-012) → 测试(WO-013)
```

### handover.sh创建工单时关联

```bash
# 创建工单时指定前后序
bash scripts/handover.sh work "标题" "提交者" "接收角色" "WO-001" ""
# 参数4=前序工单，参数5=后序工单（空则不关联）
```

## 工作流注册表（隐式工作流发现）

> 来源：agency-agents Workflow Architect → 体系融入(OPT-088)
> 核心问题：代码里存在但文档里没有的工作流，是最大的隐患——改了都不知道影响什么
> 原则：工作流存在于代码但不存在于文档=负债，必须发现并记录

### 工作流四视图

| 视图 | 维度 | 用途 |
|------|------|------|
| 按工作流 | 每个工作流的完整路径 | 查"这个流程怎么走的" |
| 按组件 | 每个文件/模块参与哪些工作流 | 改代码前查影响范围 |
| 按用户旅程 | 用户看到什么→背后走哪些工作流 | 需求分析 |
| 按依赖 | 工作流之间的前后关系 | 避免改A破坏B |

### 发现规则

1. **每个API端点=一个工作流入口** — 列出所有endpoint
2. **每个cron/定时任务=一个工作流** — 记录触发条件和执行链
3. **每个数据库迁移=一个生命周期** — 记录状态变化
4. **每个服务依赖=一个排序工作流** — docker-compose/部署顺序
5. **代码里有但spec里没有的工作流=Missing** — 红旗，必须补

### 实践方式

- 不单独建注册表文件（避免维护负担）
- 在项目INDEX.md中加"工作流"段，用表格记录关键工作流
- 格式：`| 工作流 | 触发条件 | 主要角色 | 状态(已记录/待补/缺失) |`
- 代码变更时，先查"按组件"视图确认影响范围
