# 目录结构规范 v2

> 最后更新：2026-05-09
> 统一所有项目的目录布局，任何智能体拿到项目都能0成本找到东西。
> v2：从"代码仓库+文档"升级为"项目即生命体"，适配开发/内容/运营/混合项目，优化AI检索体验。

## 核心原则

1. **约定大于配置**：目录名固定，不靠README解释就知道去哪找
2. **项目即生命体**：项目不只是代码，它有规划/输入/处理/输出/记忆/反馈
3. **三层分离**：规划层（方向）→ 执行层（干活）→ 记忆层（沉淀），互不干扰
4. **AI检索友好**：扁平优先、命名可搜索、INDEX当入口
5. **人机双读**：人能直觉导航，AI能语义定位
6. **目录自愈**：目录不存在时必须自动创建，不因清理/迁移/新平台而断裂
7. **回收站必选**：删除=移到回收站，禁止rm；回收站是项目结构的一部分，不是可选的

## 通用项目结构（所有项目适用）

```
./回收站/                        ← 🗑️ 全局回收站（任何删除操作的目标，禁止rm）
  ├── <项目名>-<日期>/           ← 按项目+日期归档被删除的内容

project-root/
├── INDEX.md                # 📖 知识索引（必读入口，AI和人的第一站）
├── README.md               # 项目说明（一句话定位+快速上手）
│
├── 规划/                    # 🧭 方向层——项目往哪走、为什么这么走
│   ├── roadmap.md          # 路线图（阶段/里程碑/优先级）
│   └── decisions.md        # 关键决策记录（为什么选A不选B）
│
├── src/                     # 🔧 源——项目的核心产出物来源
│   └── (按项目类型不同，见下方变体)
│
├── docs/                    # 📚 记忆层——项目的大脑
│   ├── dev-log/            # 过程记录（干了什么、遇到了什么）
│   └── knowledge/          # 知识沉淀（学到了什么、踩了什么坑）
│
├── assets/                  # 🎨 资源层——项目用到的原材料
│   ├── images/             # 图片（封面/截图/素材）
│   ├── templates/          # 模板（可复用的起步文件）
│   └── data/               # 数据（参考数据/基准数据）
│

│
├── config/                  # ⚙️ 配置层——项目行为控制
│   └── (配置文件/环境变量模板)
│
├── scripts/                 # 🤖 自动化——让机器干的活
│   └── (部署/迁移/数据/批处理脚本)
│
└── feedback/                # 🔄 反馈层——外部世界怎么看你
    └── (用户反馈/运营数据/评审意见)
```

## 共性部分（必须统一，所有项目必须有）

不管什么类型，以下结构**必须一致**：

| 路径 | 用途 | 必须有 | AI检索价值 |
|------|------|--------|-----------|
| `INDEX.md` | 知识索引（全局入口） | ✅ | ⭐⭐⭐ AI第一站 |
| `README.md` | 项目说明 | ✅ | ⭐⭐ 快速理解 |
| `规划/` | 方向+决策 | ✅ | ⭐⭐⭐ 理解为什么 |
| `docs/dev-log/` | 过程记录 | ✅ | ⭐⭐ 理解做了什么 |
| `docs/knowledge/` | 知识沉淀 | ✅ | ⭐⭐⭐ 理解学到了什么 |
| `assets/` | 资源 | ✅ | ⭐ 素材定位 |
| `output/` | 产出物 | ✅ | ⭐⭐ 成品定位 |
| `scripts/` | 自动化 | ✅ | ⭐ 操作复用 |

## 多子项目模式（一个项目包含多个强关联子项目）

> 适用场景：DAW+插件、平台+SDK、框架+组件 等强绑定关系

当一个项目内有多个独立但强关联的子项目时，**不拆成两个项目，用子目录区分**：

```
project-root/
├── INDEX.md                # 项目总索引（含子项目清单）
├── README.md
├── 规划/
│   ├── roadmap.md          # 整体路线图
│   └── decisions.md        # 全局决策
├── src/
│   ├── daw-core/           # 子项目1：DAW核心
│   ├── vc-plugins/         # 子项目2：VC插件系列
│   └── shared/             # 子项目间共享代码
├── docs/
│   ├── dev-log/
│   │   ├── daw-core/       # 各子项目独立日志
│   │   └── vc-plugins/
│   └── knowledge/          # 共享知识（DSP/混音/插件设计）
├── tests/
│   ├── daw-core/
│   └── vc-plugins/
├── output/
│   ├── releases/           # 插件发布包
│   └── mixes/              # 混音成品
├── config/
├── scripts/
├── feedback/
└── Cargo.toml / justfile
```

### INDEX.md 中的子项目声明

多子项目必须在INDEX.md顶部声明子项目清单：

```markdown
## 子项目

| 子项目 | 路径 | 状态 | 说明 |
|--------|------|------|------|
| DAW核心 | src/daw-core/ | 开发中 | VCMix宿主+音频引擎 |
| VC插件 | src/vc-plugins/ | 已完成26个 | 23效果器+3乐器 |
| 共享库 | src/shared/ | 开发中 | 两个子项目共用 |
```

### 判断要不要拆的3个标准

| 标准 | 作为一个项目 | 拆成两个项目 |
|------|------------|------------|
| 代码共享 | 共享代码（src/shared/） | 各自独立，共享需发布crate |
| 知识共享 | 直接共享docs/knowledge/ | 需要跨项目流动（角色→共享知识） |
| 发布节奏 | 一起发布/版本绑定 | 独立发布/独立版本 |

**一条判断原则**：如果A的变更经常影响B → 一个项目；如果A几乎不需要改B → 两个项目。

### 项目联盟（两个独立仓库，但知识强关联）

当两个项目是独立Git仓库但知识强关联时，用**联盟**关系绑定：

在各自INDEX.md中声明联盟关系：

```markdown
## 联盟项目

| 项目 | 路径 | 关系 | 共享知识 |
|------|------|------|----------|
| VC插件(AudioFX) | ./项目/AudioFX/ | 插件→DAW的扩展 | DSP/插件设计/混音经验 |
| DAW核心(OpenDAW) | ./项目/OpenDAW/ | 宿主→插件的平台 | 音频引擎/信号链/扩展注册 |
```

联盟项目的知识流动走**角色knowledge**——"Rust开发者"角色同时服务两个项目，插件踩的坑提炼到角色knowledge，DAW开发时自动带上。不需要额外的跨项目流动。

**判断要不要组联盟**：如果两个项目的开发日志/知识沉淀里经常互相引用 → 组联盟；如果各干各的 → 独立项目。

**开发类额外必须**：

| 路径 | 用途 | 必须有 |
|------|------|--------|
| `src/` | 代码 | ✅ |
| `tests/` | 测试 | ✅ |
| `config/` | 配置 | ✅ |
| `justfile` | 任务入口 | ✅ |
| `.github/workflows/` | CI/CD | ✅ |

## AI检索优化设计

### 为什么目录结构影响AI检索？

虽然AI主要靠语义搜索，但目录结构决定了：
1. **搜索范围**：`docs/knowledge/`下搜"踩坑"比全项目搜精准10倍
2. **上下文理解**：看到`规划/decisions.md`就知道里面是决策，不是代码
3. **关联发现**：INDEX.md作为入口，AI一次读取就能建立项目全貌

### 优化原则

| 原则 | 做法 | 原因 |
|------|------|------|
| 扁平优先 | 目录不超过3层 | AI读深层路径成本高 |
| 语义命名 | `knowledge/`不用`kb/` | 语义搜索直接命中 |
| INDEX当入口 | INDEX.md包含所有关键链接 | AI一次读取定位所有 |
| 日期前缀 | `2026-05-09-Phase2.md` | 时间排序+搜索友好 |
| 类型隔离 | docs/和src/分开 | 搜知识不搜到代码，搜代码不搜到文档 |

### INDEX.md的AI检索角色

INDEX.md不是给人看的目录，是**给AI的检索地图**：
- AI接到任务 → 第一步读INDEX → 知道去哪找 → 精准定位
- 不读INDEX = 全项目盲搜 = 浪费token = 结果不准

## 用户开发体验

### 人怎么看项目？

```
人脑导航路径：INDEX.md → 看概况 → 点链接 → 到具体文件
```

关键：
1. **INDEX.md是人也是AI的首页**——一个入口两套用法
2. **`规划/`让人快速理解项目走向**——新人onboard先读这个
3. **`output/`让非技术人员找到成品**——不需要翻代码
4. **`assets/templates/`让人复用前人经验**——不用从零开始

## 整体项目布局（扣子文件系统层面）

一个项目在扣子文件系统中的完整结构：

```
./项目/<项目名>/           ← 项目知识中心（规划/文档/知识/资源/产出）
├── INDEX.md
├── 规划/
├── docs/
├── assets/
├── output/
├── config/
├── scripts/
├── feedback/
└── (代码可选放这，也可仅存GitHub)

./技能/<项目名>-dev/     ← 项目技能（AI如何操作这个项目）
├── SKILL.md
└── references/

./SECRET.md                    ← 保密信息（API Key等，集中管理）
```

**为什么技能不放项目目录下？**
- 技能是"操作方法"，不是"项目知识"
- 技能可能跨项目复用（比如open-dev-tools）
- 保密信息集中管理更安全

**为什么代码可放可不放？**
- 纯代码项目：代码在GitHub，项目目录只放文档和知识
- 混合项目：如果需要在扣子内编译调试，代码也放进来
- 原则：项目目录是"知识中心"，代码是"执行层"，两者可分离

## 项目还缺什么？（完整清单）

| 维度 | 目录 | 当前有 | 缺什么 |
|------|------|--------|--------|
| 方向 | `规划/` | ❌ | roadmap + decisions |
| 源 | `src/` | ✅ | — |
| 验证 | `tests/` | ✅(开发类) | — |
| 过程记忆 | `docs/dev-log/` | ✅ | — |
| 知识记忆 | `docs/knowledge/` | ✅ | — |
| 资源 | `assets/` | ❌ | images + templates + data |
| 产出 | `output/` | ❌ | 交付物/成品 |
| 配置 | `config/` | ✅ | — |
| 自动化 | `scripts/` | ✅ | — |
| 反馈 | `feedback/` | ❌ | 用户反馈/运营数据 |
| 操作方法 | 技能目录 | ✅ | — |
| 保密 | SECRET.md | ✅ | — |
| 入口 | INDEX.md | ✅ | — |

**三个关键缺失**：
1. **`规划/`**——没有方向记录，换了session就不知道为什么这么做
2. **`assets/`**——图片/模板/数据散落各处，每次都重新找
3. **`feedback/`**——没有外部反馈收集，闭门造车

## 迁移策略

- **新项目**：严格按v2规范建
- **现有项目**：优先补齐三缺失（规划 + assets + feedback），其余逐步对齐
- **不强制一刀切**：每轮迭代补一点，不搞大迁移

## 文件命名规范

- 开发日志：`YYYY-MM-DD.md` 或 `YYYY-MM-DD-主题.md`
- 知识沉淀：`主题.md`（中文可，英文也行，看项目调性）
- 脚本：`kebab-case.sh` / `kebab-case.py`
- 配置：`snake_case.toml` / `snake_case.yaml`
- 规划文档：`roadmap.md` / `decisions.md`（固定名，不换）

## 目录自愈机制

> 解决问题：定时清理任务可能删除空目录，项目执行时必须能自我修复

### 原则

**任何session操作项目文件前，先验证目录结构完整性，缺失则自动创建。**

这不是"建议"——这是铁律。目录不存在 ≠ 项目不存在，只是目录被清理了，文件还在（空目录没东西自然会被清理工具删掉）。

### 自愈模板

每个项目的INDEX.md中必须包含**目录自愈块**，格式如下：

```markdown
## 目录自愈

此项目按 [开发类] 目录结构规范，执行任何操作前运行：

```bash
mkdir -p ./项目/<项目名>/{
  规划,src,tests/{integration,fixtures},
  docs/{dev-log,knowledge},
  assets/{images,templates,data},
  output,config,scripts,feedback
}
```
```

### 四种类型的自愈命令

**开发类**：
```bash
mkdir -p ./回收站 && \
mkdir -p ./项目/<项目名>/{
  规划,src,tests/{integration,fixtures},
  docs/{dev-log,knowledge},
  assets/{images,data},
  output,config,scripts,feedback,
  .github/workflows
}
```

**内容类**：
```bash
mkdir -p ./回收站 && \
mkdir -p ./项目/<项目名>/{
  规划,src/{drafts,outlines},
  docs/{dev-log,knowledge},
  assets/{images,templates,data},
  output,config,scripts,feedback
}
```

**运营类**：
```bash
mkdir -p ./回收站 && \
mkdir -p ./项目/<项目名>/{
  规划,src/{素材,文案},
  docs/{dev-log,knowledge},
  assets/{images,templates,data},
  output,config,scripts,feedback
}
```

**混合类**：
```bash
mkdir -p ./回收站 && \
mkdir -p ./项目/<项目名>/{
  规划,src,tests,
  docs/{dev-log,knowledge},
  assets/{images,templates,data},
  output,config,scripts,feedback
}
```

### 角色目录自愈

角色也需要自愈，但结构更轻：
```bash
mkdir -p ./角色/<角色名>/{knowledge,templates,log}
# RULES.md 和 hot-rules.md 是文件不是目录，不需要mkdir
```

### 角色是单例

⚠️ **同一角色只能有一个活跃会话。** 角色目录是共享的，多个会话同时写会冲突。

```
✅ 正确：新会话读已有角色的RULES+hot-rules，按里面的规矩干活，经验追加到knowledge/
❌ 错误：新会话再建一个同名角色，覆盖已有的RULES/hot-rules
```

判断角色是否已存在：`ls ./角色/` 查看目录列表。已存在就复用，不存在才新建。
