# 机制省token设计规范

> 最后更新：2026-05-14
> 体系架构级规范——所有脚本、模板、角色的设计必须遵循

## 核心原则

**让合规成为阻力最小的路径，违规需要额外努力。**

多智能体体系的token消耗 = prompt长度 × 执行次数。省token的本质不是少说话，而是：
- 不该做的 → 物理上做不到，不用prompt提醒
- 该做的 → 一键完成，不用prompt指导
- 做过的 → 缓存结果，不用重复思考

## 六大方式

### 1. 脚本——做对了省力

把重复的多步判断固化成一键脚本，角色不需要在prompt里学习"怎么做"，只需知道"跑哪个"。

| 场景 | 没脚本 | 有脚本 |
|------|--------|--------|
| 推送代码 | prompt写：先commit→再pull→检查冲突→push→检查双推 | `bash scripts/push.sh "描述"` |
| 创建仓库 | prompt写：先GitHub API创建→再Gitee API创建→配remote→装hooks→首次推送 | `bash scripts/create-repo.sh 名 "描述" 平台` |
| 初始化平台 | prompt写：配git用户→配remote→配双推→装hooks | `bash scripts/init.sh 扣子` |

**设计要点**：
- 脚本内置铁律（禁止强推、Gitee强制私有），不用靠prompt提醒
- 脚本内置幂等，重复执行无副作用
- 角色的prompt只需要：脚本名 + 参数，不用写步骤

### 2. 模板——填空代替从零写

模板用结构约束质量，角色填空就行，不用从零构思格式。

| 场景 | 没模板 | 有模板 |
|------|--------|--------|
| 角色规则 | 每次从头想该写什么段落 | RULES.template.md 填空 |
| 优化追踪 | 每次想该记哪些字段 | OPT条目格式固定：问题/讨论/方案/决策原因 |
| 踩坑记录 | 每次想该记什么 | knowledge/ 固定格式：场景/坑/解法/预防 |

**设计要点**：
- 模板里必填项用 `{占位符}` 标记，漏填一眼看出
- 模板即校验——格式不对=内容不全
- 新角色/新项目自动从模板生成，不用手动写

### 3. 钩子(Hook)——做错了自动拦

不用prompt说"请不要XX"，做了直接被挡回来。

| Hook | 拦截什么 | 省了什么 |
|------|---------|---------|
| pre-commit | 格式错误、明文凭据回写 | 不用prompt提醒"检查格式""别写密码" |
| commit-msg | OPT编号缺失/格式错 | 不用prompt提醒"记得写OPT编号" |
| pre-push | 强推、删远程分支 | 不用prompt提醒"禁止--force" |

**设计要点**：
- Hook是物理拦截，绕不过去（比铁律更硬）
- Hook不消耗token——它是bash脚本，不是prompt
- 出错信息直接告诉角色怎么修，不需要角色自己猜

### 4. 凭据隔离——没钥匙物理上做不到

不用prompt说"未经允许不要连服务器"，没密码就是连不上。

| 机制 | 防什么 | 省了什么 |
|------|--------|---------|
| ECS token单独加密 | 角色自行连服务器 | 不用prompt写"连服务器前要审批" |
| 密码不存文件 | 新会话自动遗忘 | 不用prompt写"用完清除密码" |
| pre-commit检测明文 | 凭据写回文件 | 不用prompt写"解密后不要存回去" |

**设计要点**：
- 靠机制不靠自觉——prompt可以说"请遵守"，但角色可能忘
- 密码每次找主人要=每次都有一次人工审批点
- 物理隔离 > 流程约束 > prompt提醒

### 5. 索引驱动——看一眼就够，不用搜

分层索引让角色只读自己需要的，不用大海捞针。

| 层级 | 读什么 | 大小 |
|------|--------|------|
| 入口.md | 全局目录+当前状态 | ~1页 |
| 项目INDEX.md | 只跟本项目相关的 | ~1页 |
| 角色RULES | 只跟自己角色相关的 | 按需 |
| knowledge/ | 踩坑缓存，按需查 | 按条 |

**设计要点**：
- 索引是"地图"，不是"百科全书"
- 角色不需要知道别的角色的规矩
- 读1KB索引代替读10KB全量文件

### 6. 缓存(踩坑记录)——踩过的坑不用再踩

knowledge/的本质是token缓存：相同问题不花两次token。

| 级别 | 位置 | 触发条件 | 作用 |
|------|------|---------|------|
| knowledge | 角色或项目下 | 踩坑1次 | 被动防御，搜到才省 |
| hot-rules | 角色根目录 | 踩坑2次 | 每次加载可见，主动省 |
| 脚本/铁律 | scripts/ 或 RULES | 系统性风险 | 物理不可绕过，最省 |

**设计要点**：
- 踩坑记录 = 最省token的机制（避免重复犯错和重复思考）
- 升级路径：knowledge → hot-rules → 脚本/铁律
- 判断标准：靠人记住才能避免的错误 = 系统性风险 = 必须升成脚本

## 设计检查清单

新功能/新脚本上线前，逐项检查：

- [ ] **脚本化**：重复3次以上的操作是否可脚本化？
- [ ] **模板化**：产出物是否有固定格式？有就做模板
- [ ] **Hook拦截**：有什么错误是prompt拦不住的？加Hook
- [ ] **权限隔离**：有什么操作不该所有人都能做？加凭据隔离
- [ ] **索引精简**：角色加载时读了不需要的内容？精简索引
- [ ] **缓存复用**：踩坑2次了？升hot-rules或脚本

## 与五步门的关系

五步门是流程骨架，六大方式是技术手段。两者配合：

```
①查后定方案 → 索引驱动（快速定位现有方案）
②执行方案   → 脚本+模板（一键执行，填空输出）
③测试验证   → Hook拦截（做错了自动挡）
④反哺沉淀   → 缓存升级（踩坑记录→hot-rules→脚本）
⑤交付闭环   → 凭据隔离（敏感操作需审批）
```
