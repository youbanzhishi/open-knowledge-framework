## 2026-05-10 大重构：从散乱到分诊制

### 起因

体系原始版本是简单的清单式导航（项目清单+角色清单+技能清单+使用流程），存在多个严重问题：
- 导航路径全部404
- 3个项目不存在（傲娇小汤圆、掘金签到、短链项目——后掘金签到作为运营项目保留）
- 4组重复技能
- 5个角色缺文件
- 25个散文件在知识沉淀目录下无人认领
- 没有知识归属判断标准，经验放哪全凭感觉

### 关键决策记录

#### 1. 经验在人身上（14:26）

**问题**：经验应该归项目还是归角色？

**决策**：角色=人，项目=工作台。通用经验跟着角色走，项目只存项目特有事实。

**判断标准**：问"换个项目还有用吗？"没用→项目，还有用→角色，换角色还有用→共享知识。

**类比**：一个工程师做了三个项目，项目归档了但人变强了。

**写入位置**：架构规范第三章"知识归属与自动流动"

#### 2. 项目技能不存在（14:52）

**问题**：只对某个项目有用的操作手册要不要封装成技能？

**决策**：不应该。项目技能=项目文档，不叫技能。内容还回项目INDEX。

**技能判断标准**：去掉判断力还能干吗？能干=工作流，干不了=技能。

#### 3. 工作流=轻量运营项目（15:27）

**问题**：工作流归角色还是归项目？

**决策**：工作流不会让角色变聪明，放项目。有独立状态需要管理的流程归项目目录，最简结构：INDEX+scripts+docs/knowledge+config。

**核心洞察**：能让角色变聪明的经验才更值得归到角色中。

#### 4. 三层单向关联（16:16）

**问题**：项目、角色、技能之间要不要双向关联？

**决策**：不需要。单向就够了。
```
项目INDEX → 声明关联角色（消费者找提供者）
角色RULES → 声明关联技能（人拿工具）
技能 → 独立存在（工具不关心谁在用）
```

**原因**：角色不需要知道自己服务哪些项目。双向关联维护成本高，收益低。

#### 5. 铁律判断标准：违反=不可逆损害（16:33）

**问题**：铁律条目太多（8条），agent记不住。

**决策**：铁律只保留"违反=不可逆损害"的条目，其余降为工作守则。

| 保留为铁律 | 降为工作守则 |
|-----------|-------------|
| 删除→回收站（不可逆） | 先查再干（可逆） |
| 改前备份（不可逆） | 经验跟着人走（可修正） |
| 做完立即记录（失忆不可逆） | 新工具要注册（可补） |
| 禁止扫描无关目录（资源不可逆） | 同样的错不犯两次（效率问题） |

**类比**：百姓不需要通读法律，但犯罪率很低——因为知道哪些事一定不能做。

#### 6. 体系修改者必须检查模板（16:50）

**问题**：改了架构规范忘了同步更新模板，导致新建项目用的还是旧模板。

**根因**：修改影响表里没有列模板文件，依赖链断了。

**修复**：影响表补上模板依赖，体系修改者分诊台从2行扩展为完整操作规程，自检清单新增"模板已同步"。

#### 7. 规范热更新：三层广播（17:04）

**问题**：规范改了，正在跑的agent还在用旧版。

**现实**：智能体平台没有广播API，真正的推模式做不到。

**方案**：三层覆盖
- 一次性任务：派任务时加"先读入口.md"
- 定时任务：改规范后更新任务描述
- 心跳检查：加一条查版本号的检查项

**版本机制**：导航顶部加版本号+变更摘要，agent只需对比版本号，变了读摘要。

#### 8. 导航瘦身（17:14）

**问题**：项目越来越多，导航文件越来越大。

**方案**：四大清单（项目/角色/技能/共享知识）从导航拆到各自目录的INDEX.md，导航只保留4行链接。

### 学到的教训

1. **体系缺了规则，agent就凭感觉，凭感觉就会搞反**——经验放错位置、改了规范忘了同步模板，都是因为没有显式规则
2. **铁律条目越多越没用**——8条记不住，4条刻在脑子里。判断标准必须是"违反=不可逆损害"
3. **改架构必须同步模板**——否则新项目用旧模板，架构改了等于白改
4. **清单内嵌在导航里会导致文件膨胀**——必须拆到各自INDEX
5. **规范热更新是平台限制**——没有广播API，只能靠版本号+任务描述手动更新

### 2026-05-10 20:45 | 安全加固：角色越界防护

**事件**：开发角色越界操作ECS服务器，差点删除内容
**决策**：
1. 铁律新增第3条"禁止越界操作服务器"——违反=不可逆损害
2. 所有开发项目INDEX加禁区声明——开工前必读，双重保险
3. 编写了Remote Gateway安全代理（scripts/gateway-proxy/），待安全组开放端口后部署
**教训**：文本规则是软约束，越界和删错一样不可逆，必须提到铁律级别。长期方案是服务器端加技术限制（命令过滤代理）
