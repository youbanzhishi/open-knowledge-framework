# 热规则规范

> 最后更新：2026-05-09
> 解决"知识沉淀了但还是会犯"的问题：把最关键的规则从冷知识升级为热规则。

## 问题

手册写了经验，但过一段时间碰到类似问题又犯错——因为知识是"被动"的，躺在文档里等被发现。
session不会主动去翻手册，尤其sub-agent拿到任务就开干，根本不读历史经验。

## 解法：冷知识 vs 热规则

| 类型 | 存放 | 触发方式 | 类比 |
|------|------|----------|------|
| 冷知识 | docs/knowledge/ | 需要时去查（被动） | 百科全书 |
| 热规则 | 项目热规则.md | 注入每个任务描述（主动） | 交通法规 |

**热规则 = 每次派任务时必须注入的规则，跟TOOLS.md文件安全铁律同一个机制。**

## 冷知识升级为热规则的条件

不是所有知识都值得升级。只有满足以下条件的才升级：

1. **犯过2次以上**：说明被动阅读不够，必须主动注入
2. **犯错代价高**：覆盖文件、数据丢失、安全漏洞、生产事故
3. **违反直觉**：正确做法跟"第一反应"相反，容易踩坑
4. **跨session通用**：不只一个项目会遇到

升级流程：
```
踩坑 → 记录到docs/knowledge/（冷知识）
     → 犯了第2次 → 升级为热规则
     → 从此每次派任务都注入，不可能再犯
```

## 热规则文件结构

每个项目在 `规划/hot-rules.md` 中维护热规则：

```markdown
# [项目名] 热规则

> ⚠️ 此文件内容必须在每次派发任务时注入任务描述
> 主对话负责维护，其他session只读

## 🔴 铁律（违反=任务失败）

1. [规则1]：[具体描述]
2. [规则2]：[具体描述]

## 🟡 警告（容易犯错）

1. [警告1]：[为什么容易犯 + 正确做法]
2. [警告2]：[为什么容易犯 + 正确做法]

## 🟢 提醒（容易忘）

1. [提醒1]：[什么时候需要想起这个]
2. [提醒2]：[什么时候需要想起这个]
```

### 优先级说明

| 级别 | 含义 | 注入方式 |
|------|------|----------|
| 🔴 铁律 | 违反=严重后果 | 每次任务描述开头，全量注入 |
| 🟡 警告 | 容易犯但后果可修复 | 每次任务描述中段，全量注入 |
| 🟢 提醒 | 容易忘但影响可控 | 相关任务时注入，不相关可省略 |

## 注入机制

### 主对话派发任务时

任务描述的标准格式：

```
【强制·铁律】
1. [从hot-rules.md复制铁律]
2. [从hot-rules.md复制铁律]
3. TOOLS.md文件安全铁律（已有的）

【任务背景】
...

【任务要求】
...

【警告】
1. [从hot-rules.md复制警告]

【提醒】（本任务相关）
1. [从hot-rules.md复制提醒]
```

### INDEX.md引用

INDEX.md的共享规范段增加热规则引用：

```markdown
## 共享规范
→ ./共享知识/项目规范/（宪法层）
- 目录结构规范：最后同步 2026-05-09
- 协作规范：最后同步 2026-05-09
本项目已对齐：部分

## 热规则
→ 规划/hot-rules.md（派发任务时必须注入）
```

### 自检：任务完成后

sub-agent完成任务后，主对话验收时检查：
- 是否违反了热规则？
- 如果违反了，热规则描述是否不够清晰？需要补充什么？

## 热规则 vs 冷知识 对照

| 维度 | 冷知识（docs/knowledge/） | 热规则（规划/hot-rules.md） |
|------|--------------------------|---------------------------|
| 触发 | 需要时去查 | 每次任务自动注入 |
| 数量 | 不限，越多越好 | 精简，只留最关键的 |
| 更新频率 | 随时追加 | 踩坑升级，定期精简 |
| 读者 | 主动去找的人 | 所有执行任务的session |
| 类比 | 百科全书 | 交通法规 |

## 热规则的"冷却"

如果一条热规则连续10次任务都没被违反，说明已经内化为习惯，可以降级回冷知识：

```
🔴铁律 → 连续10次无违反 → 🟡警告 → 连续10次 → 🟢提醒 → 移回docs/knowledge/
```

但不能直接删除——冷知识仍保留，只是不再每次注入。

## 全局热规则（所有项目共享）

以下热规则适用于所有项目，已经在TOOLS.md和宪法层维护：
1. 文件安全铁律（覆盖=犯罪，先读后动，删除→回收站）
2. 协作铁律（接手先查共享知识，产出必更新INDEX）
3. 不碰其他项目文件（除非主人授权）

项目级热规则只放**本项目特有的**，不放全局的（避免重复注入）。

## knowledge → hot-rules 自动升级机制

> v2.0新增：同类knowledge出现2次时自动升级为hot-rule

### 核心问题

"记录经验"≠"经验自动生效"。
- knowledge是被动存储：要搜才有，躺在文档里等人翻
- hot-rule是主动注入：每次任务自动加载，不搜也知道

### 升级条件

以下任一条件满足，建议升级：
1. **同类knowledge出现2次**：同一个"错误模式"或"最佳实践"在knowledge中被记录2次以上
2. **沉淀时自检发现类似记录**：写新knowledge前搜索，发现已有类似内容
3. **经验多次被引用**：knowledge中的内容被其他项目/角色频繁引用

### 升级流程

```
1. 沉淀时自检
   └─ 写knowledge前，搜索是否已有类似记录
   
2. 发现同类记录
   └─ 标记"[二次]"
   └─ 建议升级为hot-rule
   
3. 提炼hot-rule格式
   └─ 不是复制knowledge内容
   └─ 提炼为"触发条件→标准动作"
   
4. 写入hot-rules.md
   └─ 新增条目按优先级分类（铁律/警告/提醒）
   
5. 同步到TOOLS.md项目热规则区
   └─ 通用规则提取到TOOLS.md
   └─ 下次会话自动注入
```

### 升级记录格式

在knowledge-registry.md的升级记录区域登记：

```markdown
| 升级时间 | 来源knowledge | 提炼为hot-rule | 触发次数 | 状态 |
|----------|---------------|----------------|----------|------|
| 2026-05-09 | 角色/系统开发者/knowledge/rust/错误处理模式.md | CI报E0499→检查trait bound | 2 | 活跃 |
```

### hot-rule提炼格式规范

**原则**：一行能看懂，不需要读大段背景

```
❌ 被动记录（knowledge风格）：
CI报错E0499，原因是trait bound没满足，解决方案是加where子句约束

✅ 主动规则（hot-rule风格）：
CI报E0499 → 检查trait bound，加where子句
```

**提炼步骤**：
1. 提取触发条件（什么时候触发这条规则）
2. 提取标准动作（触发后怎么做）
3. 去掉背景描述（为什么/怎么发现的/踩坑过程）
4. 格式化为"条件→动作"

### 降级机制

如果hot-rule连续10次任务无违反，说明已内化为习惯：
```
🔴铁律 → 连续10次无违反 → 🟡警告 → 连续10次 → 🟢提醒 → 移回knowledge
```

降级时：
- 保留原knowledge内容
- 保留hot-rule历史记录（在升级记录中标记"已降级"）
- 只是不再主动注入

### 与现有升级路径的关系

| 触发条件 | 原有路径 | 新增路径 |
|----------|----------|----------|
| 踩坑2次 | 项目踩坑→项目hot-rules | 保持不变 |
| 同类knowledge沉淀2次 | 无 | **新增**：knowledge→hot-rule→TOOLS.md |
| 跨项目经验 | 项目→角色knowledge | 保持不变 |

> **注意**：两条路径独立运行。踩坑2次是直接升级为项目hot-rule；knowledge二次出现是升级为更通用的规则，可能进入TOOLS.md。
