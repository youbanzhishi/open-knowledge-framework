# Dev↔QA循环规范

> 来源：agency-agents NEXUS Phase 3 → 体系融入(OPT-087)
> 核心发现：Dev↔QA循环3次重试能捕获95%的缺陷，比管线末端测试效率高50%
> 原则：每个任务独立验证后再推进下一个，不攒着最后一起测

## 核心机制

```
对于每个任务（按优先级排序）:

  1. 指派 → 开发角色实现
  2. 验证 → QA角色/自检验证
     - 功能：验收条件逐条过
     - 证据：截图/测试输出/运行结果
  3. 判定：
     通过 → 标记完成，推进下一任务
     失败 且 尝试<3 → 把QA反馈返给开发，只修反馈的问题
     失败 且 尝试≥3 → 升级！不允许第4次自作主张
```

## 三次重试熔断

| 尝试 | 行为 | 指引 |
|------|------|------|
| 第1次失败 | 发QA反馈给开发 | 反馈要具体：哪个条件不通过、预期vs实际、怎么修 |
| 第2次失败 | 发累积反馈+考虑换人 | 问：开发角色选对了吗？ |
| 第3次失败 | 强制升级 | 填升级报告（见交接模板规范#4），调教师/主对话决策 |

**关键**：开发角色修bug时只修QA指出的具体问题，不要趁机"顺手改进"其他东西。

## 任务指派矩阵

| 任务类型 | 首选角色 | 备选角色 | 验证方式 |
|----------|---------|---------|----------|
| Rust核心开发 | 系统开发者 | — | cargo test + 功能冒烟 |
| 前端UI | 前端开发 | — | 浏览器验证+响应式检查 |
| C++/JUCE | 系统开发者 | — | cmake build + 测试 |
| Docker/部署 | ECS运维 | 本地运维 | 容器启动+API可达 |
| 文案/内容 | 自媒体运营 | — | 平台发布验证 |
| 体系优化 | AI调教师 | — | 脚本可执行性+规范一致性 |

## 质量门控清单

阶段切换前必须全部通过：

| # | 条件 | 证据来源 |
|---|------|----------|
| 1 | 所有任务通过QA(100%完成) | 任务列表逐条验证 |
| 2 | 无P0/P1级别开放bug | bug单列表 |
| 3 | 所有验收条件满足 | 任务验收记录 |
| 4 | 代码已review | git历史 |
| 5 | 文档已同步 | INDEX最近变更 |

## 并行执行规则

- 无依赖的任务可以并行派发给不同角色
- 有依赖的任务必须等前置任务QA通过后再派发
- 并行任务数≤2（避免资源竞争）
- 每个并行任务独立走Dev↔QA循环

## 与现有体系的关系

- **五步门**：Dev↔QA循环是五步门③验证的展开，不是替代
- **重试熔断**：TOOLS.md已有"失败2次上报"，这里细化到3次+升级报告
- **工单系统**：QA反馈写进工单文件，升级报告提新工单
