# 跨职能协作方案

> 问题来源：2026-05-09 讨论
> 用户原话："有些会话可能和开发无关，但是归属同一个项目，他们之间有的负责运营有的负责开发等，知识上会有交叉点，在扣子上如何实现协作呢？"

## 问题本质

扣子平台上，一个项目涉及多个职能：

```
某项目
├── 开发session — 写代码、修bug、架构设计
├── 运营session — 推广文案、用户增长、数据分析
├── 内容session — 知识整理、教程写作、社区运营
└── 你(主人) — 决策、方向、需求输入
```

它们共享同一套产品知识，但关注点不同。扣子的session记忆是隔离的，知识天然不流通。

## 解决方案：INDEX.md + 协作协议

### 核心设计

每个项目一个INDEX.md，作为项目知识索引，所有session的协作入口。

INDEX.md回答：项目有什么知识？在哪？谁负责？最近有什么变化？

### 知识流向

```
开发session                    运营session
  │                              │
  │ 代码+架构 ──→ 知识沉淀 ←── 需求+反馈
  │                  │            │
  │                  ↓            │
  │              INDEX.md ←───────┘
  │                  ↑
  │                  │            │
  │  技术教程 ←── 知识沉淀 ──→ 推广文案
  │                              │
内容session

主对话：协调冲突 + 分配任务 + 维护INDEX.md最终真相
```

### 协作4条铁律

1. **接手先读INDEX** — 任何session接手项目任务前，先读`./项目/<项目名>/INDEX.md`
2. **产出必更新INDEX** — 完成任务后更新INDEX的最近变更
3. **跨职能知识写沉淀** — 开发的产品亮点、运营的用户反馈，写入知识沉淀
4. **主对话协调冲突** — 知识冲突时由主对话裁决，INDEX.md为最终真相

## 与共享知识库的关系

```
./共享知识/              ← 跨项目通用（所有项目共享）
./项目/<项目名>/INDEX.md  ← 项目内跨职能（开发/运营/内容共享）
```

共享知识解决跨项目的经验复用，INDEX.md解决项目内跨职能的知识协作。

## 为什么不用其他方式

| 方案 | 为什么不行 |
|------|-----------|
| 每个session独立搜索memory | 搜索不精确，大量噪音 |
| 主对话手动转发消息 | 主对话是瓶颈，信息碎片化 |
| 数据库/外部服务 | 引入额外依赖 |
| 所有session共享一个MEMORY.md | 不同职能关注点不同，混在一起反而混乱 |

INDEX.md方案优势：零额外依赖、文件系统天然共享、结构化可搜索、人类也能读。
