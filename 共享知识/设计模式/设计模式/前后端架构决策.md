# 前后端架构决策

> 沉淀日期：2026-05-11
> 适用于：OpenDev生态下所有项目

## 核心原则

**场景决定架构，不搞统一方案。**

## 各项目前后端方案

| 项目 | 部署场景 | 前后端方案 | 原因 |
|------|---------|-----------|------|
| **OpenDAW** | 桌面+Web | **混合集成** | 天生Web应用：opendaw-api serve静态文件→浏览器直接用；Tauri打包→桌面单二进制。一套前端代码两种部署 |
| **胶带** | 小程序+H5+API | **分离**（没得选） | 微信小程序跑在微信里，H5分享页跑在浏览器里，后端只能是纯API |
| **OpenMind** | 自托管服务 | **混合**（二进制内嵌） | 一个二进制启动就有管理界面，部署极简。API同时对外暴露，前端也能独立开发 |
| **OpenLink** | 基础设施 | **纯后端** | 无用户直面界面，只有API和Agent发现端点 |
| **OpenVault** | 基础设施 | **纯后端** | 同上，3-2-1备份引擎无需UI |

## 三种方案详解

### 方案1：混合集成（OpenDAW / OpenMind）

```
一个二进制 = API服务器 + 静态文件

浏览器访问 → http://host:port/ → 静态文件（Web UI）
             http://host:port/api/v1/* → API端点
桌面应用 → Tauri壳 + 同一份Web UI（仅OpenDAW）
```

**实现要点**：
- Axum路由：`/api/v1/*` → API，其余 → 静态文件fallback
- 静态文件用 `include_dir!` 编译进二进制（OpenMind方案）
- 或运行时从 `./static/` 目录读取（OpenDAW方案，方便开发热更新）
- 前端开发时用Vite dev server连后端API，发布时build产物塞回二进制

**优势**：部署极简，一个文件就完事。用户不需要装nginx。

### 方案2：前后端分离（胶带）

```
后端：纯API服务器（Axum，端口3000）
前端A：微信小程序（独立开发/发布）
前端B：H5分享页（独立部署/CDN）
前端C：管理后台（可选）
```

**实现要点**：
- 后端只暴露JSON API + WebSocket
- CORS配置允许前端跨域访问
- JWT认证，无状态
- 小程序通过wx.request调API
- H5页面可部署到CDN/对象存储

**优势**：各端独立迭代，互不阻塞。

### 方案3：纯后端（OpenLink / OpenVault）

```
只有API + Agent发现端点
无用户界面
```

**实现要点**：
- `/.well-known/agent.json` 暴露能力
- RESTful API + gRPC（可选）
- 其他项目/Agent通过API调用

## OpenDAW的双形态部署

OpenDAW的特殊之处：**同一套前端代码，两种部署形态**。

### Web形态（手机/平板/任意浏览器）
```
opendaw serve --host 0.0.0.0 --port 8080
→ 用户浏览器访问 http://服务器:8080
→ 触控优化版UI自动启用（检测触摸设备）
→ 无需安装任何东西
```

### 桌面形态（Windows/macOS/Linux）
```
Tauri打包 → AppImage/dmg/exe/msi
→ 本地运行，原生窗口
→ 可离线使用（音频引擎本地）
→ 系统级MIDI设备访问
```

**关键**：触控支持不是"移动端特供"，是Web UI的基础能力。桌面用户用触摸屏一样受益。

## 前端开发角色如何接手

1. 读项目的 `docs/agent-guide.md` → API契约
2. 读项目的 `docs/蓝图与设计文档.md` → UI架构章节
3. 读本文档 → 前后端方案
4. 开发：前端连API开发，不用碰后端代码
