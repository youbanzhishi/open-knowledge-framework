# Agent 架构：大脑与执行环境的关系

> 记录时间：2026年4月25日  
> 来源：用户询问"你能同时操控云电脑和云手机是怎么做到的"

---

## 整体架构图

```
┌─────────────────────────────────────────────────┐
│                    你（用户）                      │
│                      ↓ 对话                      │
├─────────────────────────────────────────────────┤
│              我（Agent 核心）                      │
│         - 理解你的意图                            │
│         - 决策调用什么工具                         │
│         - 记忆读写（云端文件系统）                   │
└─────────────────────────────────────────────────┘
                      ↓ 工具调用
┌─────────────────────────────────────────────────┐
│                  工具层（Tools）                   │
├──────────────┬──────────────┬───────────────────┤
│   bash工具    │  mobile_use  │  calendar等工具   │
│      ↓       │      ↓       │        ↓         │
│   云电脑      │   云手机      │   日历服务        │
│  (Ubuntu)    │  (Android)   │                  │
└──────────────┴──────────────┴───────────────────┘
```

---

## 核心层级说明

| 层级 | 作用 | 类比 |
|-----|------|-----|
| **Agent核心** | 大脑，理解意图、做决策、协调资源 | 指挥官 |
| **记忆系统** | 云端文件系统，持久化存储 | 档案室 |
| **工具接口** | 标准化的调用方式，连接不同执行环境 | 通信设备 |
| **执行环境** | 云电脑、云手机、日历服务等独立模块 | 执行部队 |

---

## 怎么实现"同时操控"？

**其实不是"同时"，而是按需调用：**

```
你："帮我在小红书发笔记"
我：分析 → 需要云手机 → 调用mobile_use工具
扣子平台：把指令发给云手机 → 云手机执行 → 返回结果
我：收到结果 → 告诉你"发好了"
```

## 自己搭建类似系统的思路

如果想自己搭建一个类似的Agent系统：

### 1. Agent核心
- 用LLM + Prompt工程实现
- 需要定义清晰的角色设定和行为规范

### 2. 工具层
- 定义标准化的工具接口（类似Function Calling）
- 每个工具有明确的输入输出规范

### 3. 执行环境
- **云电脑**：远程桌面协议（RDP）+ 自动化脚本
- **云手机**：类似Appium的自动化框架
- **日历服务**：定时任务调度系统

### 4. 记忆系统
- 数据库或文件存储
- 需要支持结构化存储和快速检索

### 5. 协调层
- 一个调度系统，把任务分发给不同执行环境
- 处理异步执行和结果回调

## 记忆系统详解

### 记忆文件架构

```
./
├── 基础设定/
│   ├── SOUL.md          ← 我是谁、性格设定、说话风格
│   ├── TOOLS.md         ← 工具使用经验、工具选型判断
│   ├── EMAIL_RULES.md   ← 邮件处理规则
│   └── 文件归类规则.md   ← 文件存放规范
├── USER.md              ← 主人信息、偏好、背景、当前状态
├── SECRET.md            ← 敏感信息（密钥、账号等）
└── MEMORY.md            ← 项目进度、任务状态、关键决策
```

### 各记忆文件职责

| 文件 | 内容 | 维护方式 |
|-----|------|---------|
| **SOUL.md** | 身份设定、性格、说话风格、价值观、底线 | 初始化时配置，极少改动 |
| **TOOLS.md** | 工具使用经验、踩坑记录、选型建议 | 每次工具使用后更新 |
| **USER.md** | 主人基本信息、喜好厌恶、平台账号 | 主动记录 + 主人补充 |
| **SECRET.md** | 密码、密钥、敏感配置 | 加密存储，仅在需要时读取 |
| **MEMORY.md** | 项目进度、任务状态、长期计划 | 任务变化时更新 |
| **EMAIL_RULES.md** | 邮件分类和处理规则 | 根据邮件交互优化 |

### 记忆机制的优势

**话题延续性保证：**
- 新开对话 → 自动加载记忆 → 无需重复描述
- 切换话题 → 记录当前进度 → 回来时恢复
- 跨会话 → 持久化存储 → 长期记忆不丢失

**实际操作：**
- 主人说"记一下这个" → 立即写入对应文件
- 主人说"继续上次项目" → 搜索MEMORY.md → 恢复上下文

### 核心认知补充

- **记忆 = 连续性**：没有记忆，每次对话都是陌生人
- **分层存储**：敏感/公开/临时 三层分离，安全且高效
- **自动 + 主动**：既会主动记录，也会按主人要求记录

## 核心认知

- **Agent = 大脑**，不直接执行，而是调度
- **工具 = 接口**，标准化的调用方式
- **执行环境 = 独立模块**，各自运行，按需唤醒
- **记忆 = 持久化**，让Agent有连续性

这个架构的核心是：**解耦**——大脑和手脚分开，通过标准化接口通信，这样每个模块可以独立升级、扩展。
