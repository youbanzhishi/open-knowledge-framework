## 1. 决策背景

### 三台基础设施的能力画像

2026年5月，手上可用的基础设施有三台：

| 维度 | 云电脑 | ECS | GitHub/CDN |
|------|--------|-----|------------|
| **地址** | 115.190.127.67 | 39.103.203.162 | 全球节点 |
| **CPU/内存** | 2核/3.8G RAM | 1核/1.8G RAM（已用约1.3G） | N/A |
| **出站带宽** | 不限速 ✅ | 1Mbps ❌ | 不限速 ✅ |
| **入站带宽** | 无端口，入站不可达 ❌ | 不限速 ✅ | N/A |
| **开放端口** | 无（NAT/防火墙封死） | 有公网端口 ✅ | N/A |
| **桌面环境** | VNC桌面 ✅ | 无 ❌ | N/A |
| **适合** | 编译/构建/自动化/推数据 | 轻量API路由/Nginx反代 | 静态资源不限速分发 |

### 各自的致命短板

- **云电脑**：没有开放端口，外部无法主动访问。乍一看——这台机器"废了"，只能当个编译机用。
- **ECS**：1Mbps出站带宽，分发一个大文件（比如100MB）要800秒≈13分钟。这对面向用户的下载服务是灾难性的。
- **GitHub/CDN**：只能分发静态资源，不能运行业务逻辑、不能跑API、不能编译。

### 核心问题

三台机器各有致命短板，但短板的方向不一样。问题是：**能不能让它们互补？**

## 2. 架构决策过程——从"废了"到"互补"

### 初始认知：云电脑没端口=废了

最初看到云电脑没有开放端口，第一反应是：

> "云电脑出站不限速有什么用？外面连不进来，只能往外推。但往外推到哪？推到ECS也是1Mbps卡死。所以云电脑就是个编译机，跑完构建任务就完事了。"

这个认知是**错误**的，但在当时很自然——因为思维被"服务必须被动接受请求"这个模式锁死了。

### 关键转折：主人提出"带宽互补"

主人在讨论中指出：

> "云电脑带宽大，ECS带宽小但下载不限制。"

这句话打破了僵局。它迫使重新审视三者的关系——**不是单独看每台机器能干什么，而是看它们在哪个方向上有优势。**

### 重新审视：三个方向、三种优势

把网络能力拆成**出站、入站、路由**三个方向：

| 方向 | 云电脑 | ECS | GitHub/CDN |
|------|--------|-----|------------|
| **出站（推数据）** | 不限速 ✅ | 1Mbps ❌ | 不限速 ✅ |
| **入站（拉数据）** | 无端口 ❌ | 不限速 ✅ | N/A |
| **路由（转发小数据）** | 需frp中转 ❌ | 有公网IP ✅ | N/A |

突然就清楚了：

- **云电脑出站不限速 → 适合"推"数据**（上传到GitHub/OSS/CDN）
- **ECS入站不限速 → 适合"拉"数据**（从云电脑拉取，入站不限速，比scp快60倍）
- **ECS有公网IP → 适合"路由"**（API网关，只传小数据）

### 核心洞察

> **1Mbps不是瓶颈，前提是大文件压根不走ECS。**

这句话是整个架构设计的基石。它改变了问题的定义——不是"1Mbps怎么够用"，而是"怎么让1Mbps根本不需要够用"。

## 3. 三种隧道方案对比

云电脑没有端口，外部无法主动访问。要让用户能访问云电脑上的服务，必须建隧道。我们评估了三种方案：

### 3.1 frp（推荐）

| 维度 | 评价 |
|------|------|
| **原理** | 云电脑frpc主动连ECS frps，ECS通过隧道转发用户请求 |
| **配置灵活度** | ✅ 高。支持多端口映射、TCP/UDP/HTTP/HTTPS、WebSocket |
| **Dashboard** | ✅ 有Web管理界面，查看隧道状态、流量统计 |
| **性能** | ✅ Go编写，轻量高效，frps内存占用约10MB |
| **可靠性** | ✅ 心跳检测+自动重连，断线后自动恢复 |
| **安全性** | ✅ 支持token认证+TLS加密 |
| **复杂度** | ⚠️ 需要在两端分别部署frps和frpc |

### 3.2 SSH反向隧道（应急备选）

| 维度 | 评价 |
|------|------|
| **原理** | `ssh -R` 在ECS上开端口，转发到云电脑本地端口 |
| **依赖** | ✅ 只需SSH，不需要额外软件 |
| **配置灵活度** | ❌ 差。一个-R只能映射一个端口，多端口要写多条命令 |
| **可靠性** | ❌ 需要autossh维持连接，容易断 |
| **安全性** | ✅ SSH本身就是加密的 |
| **适用场景** | frp挂了时的应急通道，或临时调试 |

### 3.3 ECS安全组直接放行云电脑IP（不推荐）

| 维度 | 评价 |
|------|------|
| **原理** | 在阿里云安全组中放行云电脑的公网IP，允许入站 |
| **简单度** | ✅ 最简单，不用装任何东西 |
| **暴露风险** | ❌ 云电脑IP暴露在安全组规则中，IP变化就得改 |
| **依赖** | ❌ 依赖云电脑有固定公网IP（实际上可能变化） |
| **灵活度** | ❌ 只能解决"连得上"的问题，无法做端口映射和路由 |

### 决策结论

- **frp为主**：配置灵活、有Dashboard、性能好、支持多协议
- **SSH隧道为应急备选**：frp挂了时快速建一个临时通道
- **安全组放行不推荐**：暴露云电脑IP、不灵活、依赖固定IP

### 为什么frp端口选7000？

之前旧FRP实例使用的是6010端口。这次新部署选择7000端口，原因：

1. **独立端口避免冲突**：旧实例可能还在运行（其他项目），新实例用不同端口互不干扰
2. **7000是frp默认端口**：文档和社区案例都用7000，降低认知负担
3. **运维第一步**：部署frp后，必须在阿里云安全组放行7000/TCP，否则云电脑连不上frps——这是最常见的"部署完连不上"的原因

## 4. 带宽互补策略推导

### 传统思路（错误）

```
用户请求 ──→ ECS ──→ 云电脑（通过frp隧道）
用户下载 ←── ECS ←── 云电脑（所有数据经ECS中转）
```

所有流量走ECS → 1Mbps出站卡死 → 大文件下载13分钟/100MB → 用户骂人

### 正确思路：按数据大小分流

```
                  ┌── 小数据(API响应/几十KB) ──→ 走ECS ──→ 用户（快！）
用户请求 ──→ ECS ┤
                  └── 大文件下载 ──→ ECS返回下载链接URL ──→ 用户从CDN拉（不限速！）

云电脑编译完成 ──→ 直推GitHub Release/CDN（出站不限速！）
```

核心思路转变：**ECS不传大文件，ECS只传"去哪拿大文件"的地址。**

### 类比：工厂、门面、物流

| 角色 | 类比 | 实际对应 | 干什么 |
|------|------|----------|--------|
| **工厂** | 云电脑 | 编译/构建/跑重任务 | 干重活，产出货物 |
| **门面** | ECS | API网关/Nginx | 只接待轻量咨询，不搬货 |
| **物流** | GitHub/CDN | 大文件分发 | 发大货，不限速 |

顾客（用户）到门面（ECS）问："我要下载OpenDAW"。门面说："货在这，去物流拿"，给个链接。顾客自己去物流（CDN）拿，门面不搬货。

### 关键API设计模式

ECS的API永远只返回**下载链接URL**，不返回文件本身：

```json
{
  "name": "OpenDAW",
  "version": "0.3.0",
  "download_url": "https://github.com/xxx/opendaw/releases/download/v0.3.0/opendaw-linux-x64.tar.gz",
  "size": "85MB",
  "sha256": "abc123..."
}
```

用户拿到URL后直接从GitHub CDN下载，ECS只传了几百字节的JSON。1Mbps传几百字节？零负担。

## 5. 七个应用场景的决策逻辑

### 5.1 OpenDAW演示站

| 维度 | 分析 |
|------|------|
| **需求** | 跑Web版DAW，需要WebSocket + 静态资源 + 可能的音频处理 |
| **适合部署在哪** | 云电脑（3.8G RAM勉强够单实例，ECS 1.8G不够） |
| **流量路径** | 用户 → ECS(Nginx反代) → frp隧道 → 云电脑(dev server) |
| **大文件怎么发** | 构建产物直推GitHub Release，用户从CDN拉 |
| **不适合的情况** | 多人同时使用（3.8G RAM只够单实例，并发会OOM） |
| **注意事项** | 云电脑Rust 1.75太旧，OpenDAW需1.86+（icu依赖），升级是前置条件 |

### 5.2 OpenLink名片

| 维度 | 分析 |
|------|------|
| **需求** | 轻量HTML页面 + API（读写名片数据） |
| **适合部署在哪** | ECS（HTML+API都是轻量数据，1Mbps无压力） |
| **流量路径** | 用户 → ECS(Nginx静态页面 + API) |
| **不需要frp** | 名片服务可以直接跑在ECS上，不需要隧道 |
| **不适合的情况** | 名片需要上传大头像 → 大文件走OSS，不走ECS |

### 5.3 预览/测试服务

| 维度 | 分析 |
|------|------|
| **需求** | 临时跑个dev server给外部预览 |
| **适合部署在哪** | 云电脑（dev server本来就跑在云电脑） |
| **流量路径** | 用户 → ECS(frp代理) → 云电脑(dev server) |
| **天然安全** | frpc断开即不可访问！不需要额外的下线流程 |
| **不适合的情况** | 需要长期在线的服务（frp断线期间不可访问） |

### 5.4 持续构建

| 维度 | 分析 |
|------|------|
| **需求** | GitHub推送触发构建，编译产物发布 |
| **构建在哪** | 云电脑（2核3.8G，比ECS强得多） |
| **webhook怎么触发** | GitHub webhook → ECS(轻量API接收) → 通知云电脑开始构建 |
| **产物怎么发** | 云电脑编译完 → 直推GitHub Release（出站不限速） |
| **不需要frp** | webhook不需要frp，ECS自己就能接收GitHub回调 |

### 5.5 浏览器自动化

| 维度 | 分析 |
|------|------|
| **需求** | 跑Playwright/puppeteer等浏览器自动化 |
| **适合部署在哪** | 云电脑（只有云电脑有VNC桌面，ECS干不了） |
| **流量路径** | 内部执行，不需要对外暴露 |
| **ECS完全帮不上** | ECS没有桌面环境，1.8G RAM也跑不动浏览器 |

### 5.6 文档预览

| 维度 | 分析 |
|------|------|
| **需求** | 本地改文档，外部可预览 |
| **适合部署在哪** | 云电脑（dev server在云电脑跑，改完即看） |
| **流量路径** | 用户 → ECS(frp代理) → 云电脑(dev server) |
| **不适合的情况** | 需要长期在线的文档站 → 应部署到GitHub Pages |

### 5.7 临时文件服务

| 维度 | 分析 |
|------|------|
| **需求** | 临时分享文件给外部 |
| **小文件** | 走frp隧道，通过ECS下载（入站不限速，ECS从云电脑拉，用户从ECS拉） |
| **大文件** | 走GitHub Release或OSS，ECS只给下载链接 |
| **file_to_url** | 扣子平台临时链接（coze.cn/s/xxx），有时效性，适合临时传脚本/配置给运维 |

## 6. file_to_url的定位

### 是什么

扣子平台提供的临时文件链接，格式为 `coze.cn/s/xxx`，有时效性（一段时间后失效）。

### 适合

- 临时传脚本/配置文件给运维执行
- 临时分享小文件
- Agent在对话中生成文件后让用户下载

### 不适合

- 长期分发地址（链接会失效）
- 大文件下载入口（有时效+性能限制）
- 作为API返回的下载地址

### 实测数据

192MB文件从ECS wget 20秒完成（入站不限速），对比scp同样文件约1200秒（1Mbps出站限制）。**快60倍。**

这验证了"ECS入站不限速"的价值——虽然ECS出站只有1Mbps，但入站不限速意味着从外部往ECS上传/拉取数据是很快的。结合frp隧道，用户请求到ECS → ECS通过隧道从云电脑拉数据（入站不限速）→ 响应用户。对于中小文件这个链路完全可行。

## 7. 风险与应对

### 7.1 云电脑重启

| 维度 | 说明 |
|------|------|
| **风险** | 云电脑重启后frpc和业务服务都需要重新启动，否则隧道断开 |
| **应对** | frpc和关键业务服务都配置systemd自启 |
| **关键配置** | 业务服务的service文件中写 `After=frpc.service`，确保frpc先启动再启动业务 |
| **验证** | 重启后执行 `systemctl status frpc` 确认隧道已恢复 |

### 7.2 frp隧道断开

| 维度 | 说明 |
|------|------|
| **风险** | frp隧道断开 → 所有云电脑服务对外不可访问 |
| **本质** | 这不是高可用架构，单点故障在frps |
| **应对** | frpc配置自动重连（默认支持）；紧急情况用SSH反向隧道应急 |
| **接受** | 对于个人项目/演示环境，非高可用是可以接受的 |

### 7.3 frp token泄露

| 维度 | 说明 |
|------|------|
| **风险** | 任何人拿到token都可以注册frp代理，把流量转发到任意地址 |
| **应对** | 强密码token + 定期轮换 |
| **注意** | token明文写在frpc.toml和frps.toml中，文件权限要控制好 |

### 7.4 ECS安全组没放行

| 维度 | 说明 |
|------|------|
| **风险** | 安全组没放行7000端口 → 云电脑连不上frps → 隧道建不起来 |
| **应对** | 运维部署第一步必须放行7000/TCP |
| **这是最常见的坑** | frp配置全对，但安全组忘放行，导致"部署完连不上" |

### 7.5 云电脑Rust版本过旧

| 维度 | 说明 |
|------|------|
| **现状** | 云电脑Rust 1.75，OpenDAW等新项目需要1.86+（icu等依赖） |
| **风险** | 编译失败，报 edition 2024 或依赖版本不兼容 |
| **应对** | 升级Rust是OpenDAW部署的前置条件 |
| **教训** | 部署前先检查工具链版本，别配完环境才发现编译不过 |

### 7.6 SSH被reset

| 维度 | 说明 |
|------|------|
| **风险** | SSH连接被reset，可能是fail2ban误封或安全组未放行 |
| **应对** | 检查fail2ban日志；确认安全组放行22端口 |
| **预防** | 将自己IP加入fail2ban白名单 |

## 8. 未来演进方向

### 8.1 云电脑换成有端口的环境

如果云电脑换了有开放端口的环境（比如换了供应商或升级套餐）：

- 不再需要frp隧道，可以直接对外服务
- ECS不再是唯一入口，可以做负载均衡
- 但当前的frp架构仍然兼容——有端口后frp只是"多了一个选项"而不是"必须换"

### 8.2 ECS升级带宽

如果ECS出站带宽升级（比如5Mbps甚至更高）：

- ECS可以承担更多分发任务
- 中等大小的文件可以直接通过ECS分发，不需要走CDN
- 但1Mbps→5Mbps的成本可能不如直接用OSS

### 8.3 加对象存储（OSS/S3）

如果加入OSS/S3：

- 大文件分发更专业（CDN加速、全球节点）
- 云电脑构建完直推OSS（出站不限速）
- ECS只做API路由，完全不碰文件
- 架构更标准，是生产环境的正确方向

### 8.4 多台云电脑

如果有多台云电脑：

- frp天然支持多客户端，每台云电脑跑一个frpc
- 不同服务部署在不同云电脑上，天然隔离
- 甚至可以做简单的负载均衡（frp支持负载均衡模式）
- 架构横向扩展能力好

## 9. 关键教训

### 教训1：不要被"无端口"限制思维

> 出站连接也能建隧道。

云电脑没有端口，不代表它不能对外服务。frp和SSH反向隧道都是云电脑**主动往外连**，不是外面往里连。就像NAT后面的手机没有公网IP，但微信照样收消息——因为是主动连出去的，回包顺着原路回来。

**思维陷阱**：看到"无端口"就认为"废了"，是被"服务必须被动监听"这个模式锁死了。主动出站同样可以建立双向通信。

### 教训2：不要被"1Mbps"吓到

> 只要大文件不走ECS就不是瓶颈。

1Mbps听起来很小，但如果只传API响应（几十KB），响应时间在毫秒级。1Mbps每秒能传约125KB，一个10KB的API响应不到0.1秒。瓶颈不是带宽的绝对值，而是**你让什么数据走了这条窄路**。

**思维陷阱**：看到"1Mbps"就想"升级带宽"，是被"所有流量走一条路"的默认假设锁死了。分开考虑不同数据的路径，窄路也能跑快。

### 教训3：架构设计要看方向

> 出站、入站、存储要分开考虑。

云电脑出站不限速但入站不可达，ECS入站不限速但出站1Mbps。如果只看"带宽"这个维度，会得出"都慢"的结论。但拆开看出站和入站，就发现互补点：云电脑擅长推，ECS擅长拉和路由。

**方法论**：评估基础设施能力时，不要只看总量指标（"带宽多少"），要看方向性指标（"出站多少、入站多少、延迟多少、并发多少"）。不同方向的能力组合才能设计互补架构。

### 教训4：三者互补 > 任何单独使用

> 没有一台机器是"够用"的，但三台互补就够了。

- 只用云电脑 → 外部无法访问
- 只用ECS → 跑不动重任务，分不了大文件
- 只用GitHub/CDN → 只能发静态资源
- 三者互补 → 重任务云电脑干，轻路由ECS干，大文件CDN发

**方法论**：资源有限时，不要试图让一台机器做所有事。识别每台机器的"长板方向"，让它在那个方向上承担主要职责，短板由其他机器弥补。
