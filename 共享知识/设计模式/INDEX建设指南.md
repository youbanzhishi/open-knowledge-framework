# INDEX.md 建设指南

> 用途：任何项目session都可以按此指南为自己的项目建立INDEX.md
> 创建时间：2026-05-09

## 为什么要建

跨职能协作的入口。不管哪个session、哪个职能、哪个智能体，碰这个项目第一件事就是读INDEX.md。

扣子平台session记忆隔离，一个项目如果有开发/运营/内容多个职能的session，它们共享产品知识但关注点不同。INDEX.md就是知识索引，让任何session零成本了解项目全貌。

## 建设步骤

### 1. 在你自己的项目目录下创建INDEX.md

位置：`./项目/<你的项目名>/INDEX.md`

如果还没有项目文档目录，先建：
```bash
mkdir -p ./项目/<项目名>/{开发日志,知识沉淀,运营,内容}
```

### 2. 按模板填写

必须包含以下8个段落（顺序固定，不要改结构）：

```
# <项目名> 项目知识索引

> 最后更新：<日期> | 更新人：<哪个session>
> 用途：任何智能体接手本项目相关任务时，先读本文件了解项目全貌

## 共享规范
→ ./共享知识/项目规范/（宪法层，改一次全局生效）
- 目录结构规范：最后同步 <日期>
- 协作规范：最后同步 <日期>
本项目已对齐：是/部分/否

## 本地偏差
| 规范 | 偏差 | 原因 |
|------|------|------|
| （无偏差则留空） | | |

## 项目定位
一句话说清项目是什么。当下/进行中/远期分别是什么状态。

## 核心知识
分4个子段，每个用表格：
### 产品特性（开发↔运营 共享）
### 技术架构（开发关注）
### 运营卖点（运营关注）— 用人话说，不要技术黑话
### 内容素材（内容关注）

## 项目状态
各Phase/阶段的状态表格。

## 关联项目
列出相关项目及关系，带链接。

## 职能分工
开发/运营/内容各谁负责，产出放哪。

## 最近变更
最近7天的重要变更，带日期和链接。
```

**重要**：前两段（共享规范+本地偏差）是新增的，解决"规范改一次全局生效"的问题。
- INDEX.md不复制规范原文，只引用
- 复制=版本分裂=最终不一致

### 3. 填写要点

1. **产品特性段**最重要——所有职能共享的知识
2. **每个特性都要有链接**——INDEX是索引不是百科，详情链接到具体文档
3. **运营卖点要用人话说**——运营session可能不懂Rust/C++
4. **最近变更要持续更新**——这是其他session判断"发生了什么"的依据
5. **链接用相对路径**——确保任何session都能访问

### 4. 维护规则

- 谁维护：项目的核心session维护，其他session只读
- 什么时候更新：每次完成项目任务后，更新"最近变更"
- 冲突由主对话裁决：INDEX.md为最终真相

### 5. 参考范例

- OpenLink：`./项目/OpenLink/INDEX.md`
- OpenVault：`./项目/OpenVault/INDEX.md`

## 协作4条铁律

1. **接手先读INDEX** — 任何session接手项目任务前，先读INDEX.md
2. **产出必更新INDEX** — 完成任务后更新INDEX的最近变更
3. **跨职能知识写沉淀** — 开发的产品亮点、运营的用户反馈，写入知识沉淀
4. **主对话协调冲突** — 知识冲突时由主对话裁决
