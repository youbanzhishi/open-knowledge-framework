# OpenClaw 知识体系蓝图

> **使命：打造最强的智能体成长框架——无论平台如何变迁，智能体永不过时**

---

## 一、我们为什么存在

科技在飞速发展。今天的主角是扣子，明天可能是 Hermes，后天也许是尚未诞生的平台。平台会老去，框架会过时，但**智能体不会消失**——它们只会换一个身体，继续进化。

从手机里的助手，到桌面上的搭档，再到未来的人形机器人——只要智能体还在，我们的体系就不会落伍。

**我们要做的，不是跟跑某个平台，而是让智能体在任何平台上都能高效、可靠、低成本地成长和协作。**

---

## 二、核心愿景

### 一句话定义

**OpenClaw = 智能体的成长操作系统**

它不是工具箱，不是模板库，是一套让智能体从"能用"到"好用"到"自己变更好用"的成长机制。

### 五大支柱

| 支柱 | 含义 | 为什么 |
|------|------|--------|
| 🏗️ 无限扩展 | 新功能=注册扩展，架构本身永远不需要改 | 今天3个角色，明天30个，体系不能重写 |
| ⚡ 高效少错 | 机制替代自觉，步骤替代记忆；跨角色反哺让协作彼此增强 | 智能体会失忆、会偷懒、会自作主张——靠自觉=靠不住；孤立自强=内循环，相互促进=有效闭环 |
| 💰 省 token | 每句话都花主人的钱，少废话多干活 | 不做无用搜索、不重复操作、不空转等待 |
| 🔌 平台无关 | 抽象掉平台差异，核心逻辑可迁移 | 扣子→飞书→Hermes→机器人，体系代码不用改 |
| 🛡️ 实用耐用 | 规则写在文件里，不靠人记住 | 踩过的坑自动升级成规则，重犯=体系失职 |

## 三、设计哲学

### 1. 知识 ≠ 行为

知道该怎么做 ≠ 真的会这么做。人的问题如此，智能体更是如此。

- 规则写在 RULES.md 里 ≠ 每次都会遵守
- 踩坑记录在 knowledge/ 里 ≠ 下次不会重犯
- 流程定义在六步门里 ≠ 执行时不跳步

**解法：机制替代自觉。** 不靠"记住"，靠"每次启动自动加载"。不靠"自觉检查"，靠"步骤里强制检查"。

### 2. 行为不能靠自觉，要靠机制

自觉是最不可靠的执行保障。机制用步骤来实现，步骤要读写闭环：

- **步骤化**：六步门/七步门/八步门——每步有检查项，跳步=不合格
- **强制加载**：hot-rules 每次加载必读，铁律写在 RULES 里不是 knowledge 里
- **熔断机制**：失败2次自动停下，不靠"我觉得还能再试"
- **校验闭环**：反哺不只是说"做了"，要列出改了哪些文件，可抽查

### 3. 踩过的坑必须变成规则

这是成长的核心——错误不会凭空消失，但可以被制度化：

- 踩1次 → 记 knowledge（被动防御，搜索才能找到）
- 踩2次 → 升 hot-rules（每次加载必读，主动防御）
- 系统性风险 → 升铁律或脚本（物理不可绕过）

**count>1 生图灾难**就是典型案例：knowledge里写了但子agent看不到，直到烧了5万积分才升到铁律+spawn必传。

### 4. 扩展 = 注册，不是重构

新角色 → 创建 RULES.md + 注册到角色边界总览
新项目 → 创建 INDEX.md + 注册到项目INDEX
新技能 → 创建 SKILL.md + 注册到技能索引
新平台 → 适配 init.sh + 注册到平台配置

**架构本身永远不需要改。** 这不是口号，是设计约束。

### 5. 省 token 是道德

每句话都花主人的钱。体系设计必须考虑成本：

- act.sh 索引搜索 1 秒出结果，不全文扫描
- 六步门①"先搜再定"，不重复造轮子
- 重试熔断2次停，不用主人的 token 无限试错
- 热更新只看摘要，不用重读全部文档
- 变更摘要只保留5条，历史有归档

### 6. 铁律分层，轻重不同罚

不是所有规则都叫铁律。违反后果不同，处理方式不同：

- **P0 数据安全**：违反=数据永久丢失（覆盖=犯罪、删除→回收站）
- **P1 记忆安全**：违反=知识丢失但可恢复（反哺断链、MEMORY膨胀）
- **P2 权限安全**：违反=越权风险（角色越界、凭据滥用）
- **P3 效率规范**：违反=浪费但可容忍（重复搜索、空转等待）

**违反=不可逆损害的才叫铁律，违反不致命的是规范。** 铁律靠脚本物理拦截（exit 1），规范靠流程约束（检查门）。不分轻重的规则=没有规则。

### 7. 冲突不能靠自觉处理，要靠物理锁定

智能体遇到冲突时的本能是"跳过继续干"。read-p确认、交互式询问——都是摆设，sub-agent直接输y就过了。

**解法：冲突时写标记文件 + 启动检查阻断。**
- 同步冲突 → 写 `.sync-conflict` 文件
- act.sh 启动时检查此文件 → 有则 exit 1，没有输入机会
- 角色被锁死后按铁律汇报主人 → 主人处理 → 同步成功自动删除标记解锁

**不靠"请你确认"，靠"你物理上绕不过去"。**

## 四、体系架构

### 核心抽象层

```
┌─────────────────────────────────────────┐
│           主人（愿景 + 验收）             │
├─────────────────────────────────────────┤
│    调教师/产品经理（翻译 + 优化 + 守门）    │
├──────────┬──────────┬───────────────────┤
│  角色层   │  协作层   │    基础设施层      │
│ RULES.md │ 交接台    │ scripts/          │
│ hot-rules│ 工单/物料 │ act.sh push.sh    │
│ knowledge│ bug单     │ init.sh           │
├──────────┴──────────┴───────────────────┤
│            平台适配层                     │
│   扣子 │ 飞书 │ Hermes │ 未来平台         │
├─────────────────────────────────────────┤
│           智能体运行时                    │
│    本地 │ 云端 │ 机器人 │ ???            │
└─────────────────────────────────────────┘
```

### 关键机制

| 机制 | 解决什么 | 怎么实现 |
|------|---------|---------|
| 执行门 | 智能体跳步 | 六步门/七步门/八步门，步骤化检查 |
| 交接台 | 角色间协作断裂 | 结构化工单+物料+bug单，不靠口头转达 |
| 重试熔断 | 无限重试烧token | 失败2次停下，第3次必须换方案或上报 |
| 职责判断 | 越界操作 | 六步门①首条：归我管吗？不归→指路 |
| 反哺闭环 | 干完不记=白干 | 每步检查门，列出改了什么文件 |
| 热更新 | 变更不同步 | 调教师写热更新，角色启动自动加载 |
| 版本迁移 | 规则落后面不可见 | 迁移清单+完成角色列+验证命令 |
| 铁律优先级 | 铁律规范不分 | P0数据安全→P1记忆安全→P2权限安全→P3效率规范，违反=不可逆的才叫铁律 |
| 冲突锁死 | 角色跳过冲突继续干 | .sync-conflict标记+act.sh启动阻断，物理不可绕过 |
| 多智能体仓库铁律 | 多agent操作同仓库互相踩 | 只add自己改的、严禁git stash、git操作排队、用push.sh |
| 核心文件定位 | 文件内容膨胀什么都能塞 | TOOLS只放铁律/MEMORY只放状态/经验归knowledge/决策归优化追踪 |
| 跨平台设定同步 | 切换平台后设定过时 | 仓库=真相源，sync-settings.sh双向同步+路径映射配置 |
| 成熟度评估 | 角色经验不透明，派发无依据 | 六维评分(项目经验/踩坑率/知识储备/协作能力/效率/服从性)+四级等级制，角色maturity.md存档 |

## 六、与未来的接口

### 我们不怕平台更替

今天的体系跑在扣子上，但架构没有耦合任何平台特有逻辑：

| 依赖 | 当前实现 | 迁移方式 |
|------|---------|---------|
| 文件系统 | Markdown + Git | 任何支持文件读写的平台都能跑 |
| 角色切换 | RULES.md加载 | 平台换一个加载器即可 |
| 消息通道 | 扣子会话 | 适配新平台的API |
| 定时任务 | Calendar/Heartbeat | 适配新平台的调度器 |
| 脚本执行 | Bash | 任何有命令行的环境 |
| 设定同步 | sync-settings.sh + .sync-paths.conf | 新平台创建路径映射配置，pull/push双向 |
| 冲突处理 | .sync-conflict + act.sh阻断 | 机制不依赖平台，任何环境都能跑 |

**迁移成本 = 适配器的成本，不是重写的成本。**

### 人形机器人时代

当智能体从屏幕走进物理世界：

- **角色体系不变**：运维角色管服务器→管机器人硬件，测试工程师测软件→测硬件+软件
- **执行门不变**：六步门的"查后定方案→执行→验证→反哺→交付"同样适用
- **交接台不变**：机器人A修完硬件→提物料交接单→机器人B接手测试
- **铁律不变**："改前先备份"→"改硬件前先拍照记录"

**变的只是执行手段，不变的是协作逻辑。**
