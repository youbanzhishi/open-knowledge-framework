# 共享知识库

> 跨项目、跨语言的可复用知识和经验。所有项目通用。
> 原则：一个项目踩过的坑，其他项目绝对不能再踩。

## 为什么需要这个？

扣子平台的智能体协作痛点（2026-05-09讨论确定）：

### 痛点分析

1. **记忆隔离**：每个session记忆独立，A知道的B不知道
   - 例：主对话讨论的架构决策，子agent执行时完全不知道
   - 例：你上次跟另一个session聊的技术选型，这次新session可能重新来过
   
2. **重复造轮子**：不同项目独立解决相同问题
   - 例：AudioFX写了Docker GHA缓存（`cache-from: type=gha`），OpenLink又从头摸索
   - 例：cargo国内镜像配置，每个项目都配一遍
   
3. **经验不流通**：这个项目踩的坑，那个项目再踩一次
   - 例：OpenLink踩了Rust编译OOM的坑，如果下次OpenVault也在本地编译，还得再踩
   - 例：编译环境PATH问题（rustup cargo vs 系统cargo），修了一次下次可能忘

4. **上下文断裂**：session上下文耗尽后，新session零上下文重新来过
   - 例：对话到11万token被迫中断，新session从summary重启，很多细节丢失

### 讨论过程

> 用户原话："其实我觉得扣子最大的问题那就是协作不到位，容易重复造轮子"

这个观察非常准确。扣子平台上每个智能体/会话是独立记忆的，天然就存在知识孤岛。不能指望平台来解决这个问题，得自己建。

### 解决方案：三层知识共享体系

**第一层：共享知识库**（治本）
- 跨项目通用知识集中存放
- CI模式、踩坑记录、设计模式
- 任何智能体接手任何项目前，先搜索本知识库

**第二层：open-dev-tools**（治标）
- 跨语言统一工具链和模板
- Rust/Python/C++三套justfile模板
- CI workflow模板可跨项目复用
- 不强推同一套脚本，而是统一命令入口+沉淀经验

**第三层：项目级技能**（治根）
- 每个项目一个技能，确保任何智能体能0成本接手
- 技能里引用共享知识，不用每个项目重复写
- 技能即交接协议

### 设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 知识放哪 | ./共享知识/ | 与项目文档同级，全局可见 |
| 强推统一脚本？ | 不强推 | AudioFX/DAW的CI已成熟，替换反而引入风险 |
| 统一什么 | 命令入口+经验沉淀 | `just build/test/lint` 所有项目一样，里面是cargo/cmake/pip无所谓 |
| CI模板怎么用 | 复制+修改 | 不用git submodule，每个项目独立维护 |

## 目录说明

| 目录 | 内容 | 示例 |
|------|------|------|
| CI模式库/ | 不绑定具体项目的CI/CD实践 | Rust CI标准模式、Docker缓存、多平台Release |
| 踩坑记录/ | 所有项目踩过的坑及解法 | Rust编译OOM、cargo国内镜像、JUCE缓存 |
| 设计模式/ | 可复用的架构模式 | Extension Registry、trait抽象、公私分治 |

## 使用方式

1. 智能体接手任何项目前，先搜索本知识库相关内容
2. 解决新问题后，如果具有通用性，沉淀到这里
3. 项目特定知识放在项目自己的知识沉淀目录，通用知识放这里
4. 每次派发sub-agent任务时，在任务描述中引用本知识库相关文档

## 与项目知识沉淀的关系

```
共享知识/           ← 跨项目通用（CI模式/踩坑/设计模式）
项目/XXX/知识沉淀/  ← 项目特定（技术选型依据/架构决策）
```

项目知识沉淀引用共享知识时用链接，不重复写。
