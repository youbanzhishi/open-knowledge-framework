# 三项目并行开发经验

> 沉淀时间：2026-05-10
> 来源：OpenDAW/OpenLink/OpenVault 三个Rust项目同日并行开发
> 适用：任何多Rust项目并行开发场景

## 一、并行编译管控（最关键的生存法则）

**问题**：云电脑3.8G RAM，多个sub-agent同时cargo build会OOM崩溃。

**解法**：
1. 先改代码不编译，统一编译时才cargo check
2. 同时编译≤2个sub-agent
3. 超过2线并行→排队等前两线完成再启动
4. CARGO_BUILD_JOBS=1 限制编译并发
5. CARGO_TARGET_DIR=/tmp/{项目名}-target 避免仓库膨胀

**原则**：2跑3排策略，每次只派2个sub-agent，等完成后再派下一批。

## 二、Sub-agent代码质量管控

**问题**：sub-agent生成的代码常有编译错误，类型推导/参数数量/缺失derive/路径问题。

**解法**：
1. 主对话统一cargo check验证，不信任sub-agent的"编译通过"声明
2. 先拿全部错误，批量修
3. 常见错误模式：format trait(f64无f)/fold闭包类型/模糊float/解引用/参数不匹配
4. 修复策略：cargo check→收集错误→逐一修复→再check→0错误才commit

## 三、Sub-agent路径限制

**问题**：read_file/write_file工具只能访问/app/data/下的路径，不能访问/tmp/。

**解法**：
1. 任务描述中明确告知：仓库在/tmp/下，用bash cat/heredoc操作
2. 不要让sub-agent用read_file读取/tmp/下的代码文件

## 四、Cargo编译范围控制

**问题**：多crate workspace全量编译太慢且可能OOM。

**解法**：
1. 只cargo check -p指定crate，不编译整个workspace
2. 每个项目确定"核心编译集"：
   - OpenDAW: -p opendaw-core -p opendaw-api -p opendaw-cli
   - OpenLink: -p openlink-core -p openlink-api -p openlink-sdk -p openlink-a2a
   - OpenVault: -p openvault-core -p openvault-server -p openvault-intel
3. 避开有系统依赖的crate（如desktop/src-tauri需要glib）

## 五、Workspace版本统一

**问题**：多crate项目版本号不统一，Release时混乱。

**解法**：
1. 根Cargo.toml加[workspace.package]段定义version/authors/license
2. 子crate用version.workspace=true继承
3. 一处改全部生效

## 六、派发五段式（任务描述模板）

1. 文件安全铁律：覆盖=犯罪，先读后动，删除→回收站
2. 项目热规则：仓库路径/Rust版本/编译命令/编译范围
3. 效率铁律：止损≤2次重试，开工前查现有方案
4. 进度汇报：完成后的报告格式
5. 强制反哺：结果发送回主对话

## 七、开发交付六步门

| 步 | 门 | 做什么 | 在哪做 | 谁管 |
|---|---|---|---|---|
| ① | cargo check 0错误 | 编译通过 | 本地+CI | 开发 |
| ② | cargo test 0失败 | 功能正确 | 本地+CI | 开发 |
| ③ | 本地构建验证 | `cargo build --release` + smoke test（`./binary --help`） | 本地 | 开发 |
| ④ | CI全平台构建 | GitHub Actions 4平台构建+打包+推Release | CI | 开发 |
| ⑤ | 桌面应用构建 | Tauri v2三平台构建(仅GUI项目) | CI | 开发 |
| ⑥ | 文档同步更新 | README/CHANGELOG/知识体系INDEX+roadmap/运维文档 | 本地 | 开发 |

六步全过才算开发完成。开发不管运维的事。

**CI只做本地做不了的事**：
- clippy -D warnings → 本地跑，不放CI（太严格，小问题让CI红，读日志费token）
- smoke test → 第③步本地做，CI浪费分钟
- 多平台构建+推Release → 必须CI，本地只有Linux
- 用 `workflow_dispatch` 手动触发，不打tag（tag多容易乱）

## 八、GitHub Release流程

1. `gh workflow run release.yml --repo user/repo` 手动触发（或push触发CI）
2. CI自动：4平台构建 → 打包tar.gz/zip → 创建GitHub Release → 上传产物
3. gh CLI需要单独安装（curl下载tar包），不在PATH中
4. 不打tag触发：`workflow_dispatch`手动触发，避免tag混乱
