# 胶带 项目路线图

> 最后更新：2026-05-11

## 总览

胶带是"现在写，条件到了才开"的时间封存平台。核心三要素：封存物（内容）+ 解封条件（什么时候开）+ 查看人（谁能看）。按Phase递进，先建骨架再叠场景，最后接入生态。

---

## Phase 1: ✅ 项目骨架+数据模型+核心trait

**目标**：搭建可运行的最小骨架，定义核心trait和数据模型

**关键交付**：
- Rust项目骨架（Axum + SQLite + migrations）
- 核心trait定义：`Sealable`（封存物）、`TriggerCondition`（解封条件）、`Viewer`（查看人）
- 数据模型：Tape / SealCondition / Viewer / TriggerEvent
- 基础API：创建封存/查询封存/更新状态
- 单元测试框架

**完成标准**：`cargo test` 全绿，可创建一条封存记录并查询

---

## Phase 2: ✅ 账号体系

**目标**：用户注册/登录/手机号绑定/换号找回/实名认证兜底

**关键交付**：
- 手机号+验证码注册/登录
- 账号→手机号绑定（一对多，一个账号可绑多个手机号）
- 换号找回：旧手机号验证 → 绑定新手机号 → 已有胶带自动迁移
- 实名认证（身份证OCR + 活体检测）作为兜底找回手段
- JWT token管理 + 刷新机制

**完成标准**：注册→登录→换号→找回全链路跑通

**设计参考**：详见ADR-003（关联键是胶带账号而非手机号）

---

## Phase 3: ✅ 封存核心

**目标**：内容加密存储 + hash上链 + 封存凭证生成

**关键交付**：
- 端到端加密：客户端加密→服务端存密文，平台零知识
- 内容hash计算（SHA-256），存入数据库 + 准备上链
- 封存凭证生成：包含内容hash + 封存时间 + 解封条件摘要
- 封存凭证可分享（短链/二维码）

**完成标准**：创建封存→内容加密存储→生成凭证→hash可验证

**设计参考**：详见ADR-007（零知识存储）

---

## Phase 4: ✅ 解封引擎

**目标**：四种触发机制的状态机实现

**关键交付**：
- 心跳失联触发：用户定期发送心跳，N天未心跳→触发解封流程
- 双向匹配触发：A→B封存 + B→A封存 → 双方匹配成功→触发解封
- 指定日期触发：到达指定日期时间→触发解封
- 多人确认触发：指定N人中有M人确认→触发解封
- 触发事件记录（TriggerEvent）
- 解封后内容解密+通知查看人

**完成标准**：四种条件各有一个端到端测试通过

**设计参考**：详见ADR-006（四种解封条件）

---

## Phase 5: ✅ 暗恋表白场景

**目标**：手机号搜索 → 被动注册 → 双向hash匹配 → 触发通知

**关键交付**：
- 手机号搜索功能（搜索对方是否已注册）
- 被动注册流程：A搜索B的手机号→B未注册→发短信邀请→B注册→自动关联
- 双向hash匹配：A封存"喜欢B" + B封存"喜欢A" → hash匹配 → 双方同时收到通知
- 只有一方 → 永远静默，对方不知道
- 隐私保护：搜索不留痕，匹配前不泄露

**完成标准**：A向B表白 + B向A表白 → 双方同时收到通知；只有一方 → 无任何通知

**设计参考**：详见ADR-005（双向封印机制）

---

## Phase 6: ✅ 遗嘱交代场景

**目标**：身份证实名 + 查看人验证 + 心跳失联

**关键交付**：
- 遗嘱封存流程：实名认证 → 创建遗嘱 → 指定查看人 → 心跳设置
- 查看人验证：查看人需实名认证+身份验证后才能查看
- 心跳失联检测：用户按设定频率（7天/14天/30天/90天）发送心跳
- 失联后解封流程：失联N天→短信/邮件提醒→宽限期→正式解封→通知查看人
- 宽限期机制：防止误触发

**完成标准**：创建遗嘱→正常心跳→停止心跳→宽限期→解封→查看人收到通知并验证身份后查看

---

## Phase 7: ✅ 时间胶囊场景

**目标**：短链邀请 + 指定日期解封

**关键交付**：
- 时间胶囊创建：选择日期 → 写内容 → 指定查看人（或公开）
- 短链邀请：生成OpenLink短链，分享给查看人
- 倒计时展示：查看人可看到"还有X天开启"（内容不可见）
- 到期解封：到达指定日期 → 自动解封 → 通知查看人
- 给自己的胶囊：设定未来某个时间给自己看

**完成标准**：创建胶囊→分享短链→到期解封→查看人打开看到内容

---

## Phase 8: ✅ 区块链时间戳集成 ✅

**目标**：Ethereum L2 / ESA 时间戳证明

**关键交付**：
- Merkle树批量上链：多条封存hash合并为一棵Merkle树，只提交root
- L2合约部署：简单的TimestampRegistry合约
- 时间戳验证API：给定封存ID → 返回链上时间戳证明
- ESA集成：作为L2的备选方案
- 批量上链调度：定时（如每小时）将新封存批量上链

**完成标准**：创建封存→hash上链→验证API返回链上证明

**设计参考**：详见ADR-004（为什么这次区块链真的合适）

---

## Phase 9: ✅ OpenLink集成 ✅

**目标**：封存凭证 = Identity Card + 短链分享

**关键交付**：
- 封存凭证作为OpenLink的Identity Card
- 凭证包含：内容hash + 封存时间 + 解封条件摘要 + 链上证明引用
- OpenLink短链分享：凭证生成短链，可分享给查看人
- 凭证验证端点：`GET /.well-known/agent.json` 暴露验证能力
- Agent发现：其他Agent可通过agent.json发现胶带的封存验证能力

**完成标准**：封存凭证可通过OpenLink短链访问并验证

---

## Phase 10: ✅ OpenVault集成 ✅

**目标**：大文件加密存储引用

**关键交付**：
- 大文件（视频/音频/大量图片）存入OpenVault
- 胶带只存：Vault文件引用 + 加密密钥分片
- 解封时：从Vault取文件 + 用密钥分片解密
- 密钥分片方案：Shamir's Secret Sharing，M-of-N分片

**完成标准**：封存含大文件的胶带→解封→完整获取大文件

**设计参考**：详见ADR-007（加密策略/大文件方案）

---

## Phase 11: ✅ Web前端API准备 ✅

**目标**：API就绪，前端可用

**关键交付**：
- 微信小程序：核心场景（创建/查看/心跳/搜索）
- H5页面：分享页（查看人打开短链查看内容）
- 端到端加密：客户端加密后再提交，服务端只存密文
- 实名认证集成：OCR + 活体检测

**完成标准**：小程序可完成完整封存-解封流程

---

## Phase 12: ✅ Agent Action Protocol接入 ✅

**目标**：Agent可操作胶带平台

**关键交付**：
- `/.well-known/agent.json` 完整定义
- Agent可创建封存、查询封存状态、触发解封
- Agent可验证封存凭证（内容hash + 链上时间戳）
- 与OpenMind联动：Agent可搜索知识库中的封存记录

**完成标准**：外部Agent通过agent.json发现胶带能力并完成一次封存操作

---

## 优先级与依赖关系

```
Phase 1 ──→ Phase 2 ──→ Phase 3 ──→ Phase 4
                                      ├──→ Phase 5 (暗恋)
                                      ├──→ Phase 6 (遗嘱)
                                      └──→ Phase 7 (胶囊)

Phase 3 ──→ Phase 8 (区块链)
Phase 8 ──→ Phase 9 (OpenLink)
Phase 3 ──→ Phase 10 (OpenVault)
Phase 5+6+7 ──→ Phase 11 (前端)
Phase 9 ──→ Phase 12 (Agent)
```

## 里程碑

| 里程碑 | 包含Phase | 价值 |
|--------|----------|------|
| M1: 最小可用 | P1-P4 | 核心封存+解封能力可用 |
| M2: 场景闭环 | P5-P7 | 三大场景端到端可用 |
| M3: 生态集成 | P8-P10 | 区块链+OpenLink+OpenVault |
| M4: 用户可达 | P11-P12 | 前端+Agent接入 |
