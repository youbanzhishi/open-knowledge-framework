## ADR-001: 为什么叫"胶带"

**状态**：已决定

**背景**：项目需要一个名字，既要传达"封存"的含义，又要有画面感、好记。

**决策**：叫"胶带"（Tape）

**理由**：
- 像用胶带封住的信封——封口上有签名，撕开就知道有没有人动过
- "封口"= 封存，"签名"= hash上链，"撕开"= 解封
- 画面感极强：一卷胶带，贴住信封，封口有指纹
- 英文Tape也暗示"磁带"——记录、回放，时间胶囊的隐喻
- 简短好记，中文两个字，英文四个字母

**影响**：品牌名、域名、Logo设计都围绕"胶带封存"意象展开

## ADR-002: 产品定位不是遗嘱工具而是时间封存平台

**状态**：已决定

**背景**：封存类产品很容易被归类为"遗嘱工具"，但遗嘱只是场景之一。

**决策**：定位为时间封存平台，遗嘱是场景之一

**理由**：
- 遗嘱只是场景之一，不是全部。暗恋表白、时间胶囊、毕业寄语都是场景
- 封存物 + 解封条件 + 查看人 三要素组合可以覆盖所有场景：
  - 遗嘱：封存物=遗嘱内容，解封条件=心跳失联，查看人=法定继承人
  - 暗恋：封存物=表白内容，解封条件=双向匹配，查看人=对方
  - 胶囊：封存物=任意内容，解封条件=指定日期，查看人=指定人/自己
  - 寄语：封存物=祝福内容，解封条件=多人确认，查看人=接收人
- 如果定位为遗嘱工具，用户心智被锁定，增长天花板低
- 时间封存平台的心智更宽，每个用户都有多种场景

**影响**：
- 产品设计以三要素为核心抽象，不做场景特化
- 营销策略围绕多场景展开
- 架构上解封引擎是可扩展的，支持任意条件组合

## ADR-003: 关联键是胶带账号而非手机号

**状态**：已决定

**背景**：用户身份用什么做关联键？手机号？身份证？还是账号？

**决策**：胶带账号是关联键，手机号是入口，身份证是兜底

**理由**：
- 手机号会换——这是封存场景的致命问题：一个封存5年的胶带，5年后手机号换了，胶带怎么找回来？
- 身份证号难拿到——需要OCR+活体检测，用户体验差，不能作为日常入口
- 账号是锚——换手机号不影响已有胶带，换号只需绑定新手机号
- 手机号是入口——日常登录用手机号+验证码，便捷
- 身份证是兜底——手机号丢了、换了、注销了，用实名认证找回账号
- 三层身份：账号（永久锚）> 手机号（日常入口）> 身份证（找回兜底）

**影响**：
- 数据模型中胶带关联的是account_id而非phone_number
- 账号系统支持一个账号绑定多个手机号
- 换号流程：旧号验证 → 绑定新号 → 自动迁移
- 找回流程：实名认证 → 证明身份 → 重绑手机号

## ADR-004: 为什么这次区块链真的合适

**状态**：已决定

**背景**：区块链在很多场景是伪需求，但胶带场景有真实需求。

**决策**：使用区块链（Ethereum L2）做时间戳证明

**理由**：
- 用户需要证明"我没改过"→ 不可篡改是保护不是负担
  - 封存5年的遗嘱，解封时继承人说"你改过"，怎么证明没改？→ 链上hash不变就是证明
- 时间戳必须可信 → 链上说了算，不是自己说了算
  - "这个遗嘱是什么时候封存的？"→ 服务器时间可以伪造，链上时间戳不能
- 与OpenMind判断逻辑相反：
  - OpenMind的知识需要随时改，所以不用链
  - 胶带的记录不能改，所以用链
  - 同一个体系，不同项目，判断逻辑不同，这才是正确的
- 用L2而非L1：Gas费低，确认快，安全性与L1一致
- 只存hash不存内容：链上只有SHA-256 hash，不泄露任何内容

**影响**：
- 技术选型增加ethers-rs依赖
- 需要部署L2合约（TimestampRegistry）
- 批量上链策略：Merkle树root，节省Gas
- 解封时提供链上证明：hash + 区块号 + 时间戳

## ADR-005: 双向封印机制（暗恋表白）

**状态**：已决定

**背景**：暗恋表白的核心问题：如何让双方互相表白时触发通知，但只有一方表白时不泄露隐私？

**决策**：双向封印机制——A→B封存 + B→A封存 = 双向触发

**理由**：
- A→B封存 + B→A封存 → 双方匹配成功 → 同时收到通知
- 只有一方 → 永远静默，对方不知道
- 这是数学上唯一隐私安全的方案：
  - 不匹配 = 零信息泄露（对方完全不知道有人表白）
  - 匹配 = 双方同时知道（不存在"我知道你表白了但我不想让你知道"的不对称）
- 被动注册：搜索手机号→发短信→注册→关联，同时解决拉新
  - A搜索B的手机号，B未注册→发短信"有人想对你说些话，注册胶带查看"
  - B注册后→自动关联→如果B也封存了对A的表白→双向触发
  - 这是天然的拉新机制：好奇心驱动注册

**影响**：
- 需要手机号搜索功能（搜索不留痕）
- 需要被动注册流程
- 匹配用hash而非明文：A的封存条件里存的是hash(B的手机号)，匹配时比较hash
- 隐私保护：即使数据库泄露，也无法知道谁对谁表白

## ADR-006: 四种解封条件

**状态**：已决定

**背景**：解封条件是胶带的核心，需要覆盖所有场景。

**决策**：四种基础解封条件，可组合

**理由**：
1. **心跳失联N天**（遗嘱场景）
   - 用户定期发送心跳（7天/14天/30天/90天可配置）
   - N天未心跳 → 短信/邮件提醒 → 宽限期 → 正式解封
   - 宽限期防止误触发（住院、出差等）

2. **双向匹配**（暗恋场景）
   - 详见ADR-005

3. **指定日期**（时间胶囊场景）
   - 到达指定日期时间 → 自动解封
   - 支持时区

4. **多人确认**（遗嘱/毕业寄语场景）
   - 指定N人中有M人确认 → 解封
   - 适用于"需要多位见证人"的场景

**组合能力**：四种条件可AND/OR组合
- 例：心跳失联90天 AND 多人确认(3人中有2人) → 遗嘱+见证
- 例：指定日期 OR 心跳失联365天 → 给自己的胶囊+意外保障

**影响**：
- 解封引擎设计为状态机，支持条件组合
- `TriggerCondition` trait 是核心抽象
- 定时任务：心跳检测、日期检测需要后台调度

## ADR-007: 加密策略——零知识存储

**状态**：已决定

**背景**：胶带存的是用户最私密的内容，平台必须无法查看。

**决策**：端到端加密，平台零知识

**理由**：
- 内容端到端加密，平台也看不了
  - 客户端加密后再提交密文
  - 服务端只存密文 + 加密元数据
  - 解封条件满足时，密钥分片重组 → 解密给查看人
- 只有解封条件满足时才解密给查看人
  - 非解封状态下，任何人（包括平台管理员）都无法查看内容
  - 这是核心信任基础：用户相信平台看不到内容
- 大文件存OpenVault，胶带只存引用 + 加密密钥分片
  - 视频遗嘱、大量图片等大文件不适合存在胶带自身
  - 用Shamir's Secret Sharing做密钥分片：M-of-N，任意M个分片可还原密钥
  - 分片分散存储：胶带存一部分、查看人持有一部分、链上存一部分

**影响**：
- 客户端必须实现加密（小程序/H5都需要加密库）
- 密钥管理是核心挑战
- 备份/恢复需要考虑密钥丢失的情况
- 性能影响：大文件加密解密耗时

## ADR-008: 蓝图放共享知识/设计模式/

**状态**：已决定

**背景**：蓝图文档放在哪里？项目内还是共享知识？

**决策**：蓝图与设计文档放在项目docs/内，同时将跨项目可参考的设计模式沉淀到共享知识/设计模式/

**理由**：
- 遵循v14体系规范，蓝图放项目docs/是规范要求
- 胶带的设计模式（如双向匹配、零知识存储、条件状态机）可能被其他项目参考
- 按文档双写规范：项目docs/是操作文档，共享知识/设计模式/是知识沉淀
- 交叉引用：项目蓝图顶部📖指向共享知识，共享知识🔧指向项目蓝图

**影响**：
- 项目docs/蓝图与设计文档.md是主文档
- 跨项目可参考的模式需额外写到共享知识/设计模式/下
- 维护两处文档需要保持同步

## ADR-010: 前后端分离架构

**状态**：已决定

**背景**：胶带需要同时服务微信小程序、H5分享页和API调用方，无法将前端集成到后端二进制中。

**决策**：采用前后端分离架构。后端只暴露JSON API + WebSocket，前端独立部署。

**理由**：
- 微信小程序运行在微信容器内，无法集成到后端
- H5分享页需要CDN分发，独立部署性能更好
- 前后端独立迭代互不阻塞

**影响**：
- 后端必须完善CORS配置
- API需要完整的JWT认证机制
- 详见共享知识/设计模式/前后端架构决策.md
