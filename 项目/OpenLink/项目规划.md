# OpenLink — 智能体时代的通用路由与编排协议

> 项目代号：OpenLink
> 创建时间：2026-05-09
> 状态：规划阶段
> 核心哲学：**新功能=注册扩展，架构本身永远不需要改**

---

## 一、愿景

**OpenLink不是短链服务，是智能体互联网的基础协议层。**

- 当下：短链重定向（必须保留，这是入口）
- 近未来：Agent间的发现、握手、协作
- 远未来：智能体互联网的DNS + 路由 + 编排

一句话：**URL是人类互联网的入口协议，OpenLink是智能体互联网的入口协议。**

---

## 二、核心原语（Primitives）

架构的基石，5个原语覆盖所有场景，永不需要新增：

### 2.1 Link（链接实体）

可寻址、可识别的实体。短链是Link的最简形态。

```
Link {
  id:          LinkID,        // 全局唯一标识
  code:        ShortCode,     // 人类可读短码 (d.aw/abc)
  payload:     JSON,          // 结构化元数据（链接即数据包）
  owner:       Identity,      // 创建者
  created_at:  Timestamp,
  metadata:    JSON,          // 扩展元数据（不影响路由）
}
```

### 2.2 Route（路由规则）

从Link到Action的映射，支持条件分支。

```
Route {
  link_id:     LinkID,
  rules: [     // 有序规则列表，命中即停
    {
      condition: Condition,   // 什么情况下走这条路
      target:    Target,      // 去哪
      priority:  i32,         // 优先级
    }
  ],
  default:     Target,        // 兜底目标（传统短链重定向就是只有default的Route）
}
```

### 2.3 Action（执行动作）

Link被解析后"做什么"。不只是重定向，一切皆Action。

```
Action = Redirect | Webhook | Workflow | Transform | Delegate | Custom(Extension)
```

| Action | 说明 | 场景 |
|--------|------|------|
| Redirect | 302/301重定向 | 传统短链 |
| FileTransfer | 跨设备/跨环境文件传输 | 文件中转、跨设备共享 |
| Webhook | 触发外部HTTP回调 | Agent通知、CI/CD触发 |
| Workflow | 执行多步编排 | Agent间协作链 |
| Transform | 请求改写后转发 | 协议转换、格式适配 |
| Delegate | 委托给另一个Link | 链式路由、A/B测试 |
| Custom | 扩展注册的自定义Action | 未来任何未知场景 |

**关键设计：Custom Action通过扩展注册，核心永远不需要改。**

### 2.4 Context（请求上下文）

路由决策的输入，决定走哪条路。

```
Context {
  identity:    Identity,      // 谁在访问（人类/Agent/服务）
  device:      DeviceInfo,    // 设备信息
  location:    GeoInfo,       // 地理位置
  time:        Timestamp,     // 时间
  intent:      JSON,          // 访问意图（Agent可显式声明）
  session:     SessionID,     // 会话追踪
  custom:      JSON,          // 扩展上下文（Extension填充）
}
```

### 2.5 Hook（钩子）

路由前后的拦截器，扩展系统的核心机制。

```
Hook {
  phase:       BeforeRoute | AfterRoute | OnError,
  action:      HookAction,
  priority:    i32,
}
```

- BeforeRoute：改写Context（注入身份、添加意图）
- AfterRoute：记录日志、触发通知、改写响应
- OnError：降级处理、告警

## 三、分层架构

```
┌─────────────────────────────────────────────┐
│            Protocol Layer（协议层）            │
│  HTTP/WebSocket/MCP/A2A/自定义协议适配         │
├─────────────────────────────────────────────┤
│            Routing Engine（路由引擎）           │
│  Context解析 → Rule匹配 → Action调度          │
├─────────────────────────────────────────────┤
│            Action Layer（动作层）              │
│  Redirect | Webhook | Workflow | Extension... │
├─────────────────────────────────────────────┤
│            Core Store（核心存储）              │
│  Link | Route | Context | Hook | Stats       │
├─────────────────────────────────────────────┤
│            Extension Registry（扩展注册表）     │
│  自定义Action | 自定义Condition | 自定义Hook   │
└─────────────────────────────────────────────┘
```

### 设计原则

1. **核心层零业务逻辑**：路由引擎只做"Context→Rule→Action"的通用调度，不关心具体是什么业务
2. **一切扩展通过Registry**：新Action类型、新Condition类型、新Hook，全部注册即用
3. **协议层可插拔**：今天HTTP，明天MCP/Agent-to-Agent，加个适配器就行
4. **存储层可替换**：核心逻辑不依赖具体存储实现

## 四、Extension Registry（扩展注册表）

与OpenDAW的Extension Registry同构，四柱模型：

```
┌──────────────────────────────────────┐
│         Extension Registry            │
│                                      │
│  ┌──────────┐  ┌───────────────┐    │
│  │ Action API│  │Condition API  │    │
│  │ 注册新动作 │  │注册新路由条件  │    │
│  └──────────┘  └───────────────┘    │
│  ┌──────────┐  ┌───────────────┐    │
│  │ Hook API  │  │Protocol API   │    │
│  │注册拦截器  │  │注册协议适配器  │    │
│  └──────────┘  └───────────────┘    │
└──────────────────────────────────────┘
```

### 4.1 Action扩展示例

```rust
// 注册一个"打开DAW工程"的Action
registry.register_action("open-daw-project", |ctx, target| {
    // 检测本地是否有OpenDAW
    // 有 → 深链接拉起
    // 没有 → Web预览 / 返回元数据
});
```

### 4.2 Condition扩展示例

```rust
// 注册一个"Agent类型判断"的Condition
registry.register_condition("agent-type", |ctx, params| {
    ctx.identity.agent_type == params["type"]
});
```

### 4.3 Hook扩展示例

```rust
// 注册一个"统计埋点"的Hook
registry.register_hook("analytics", BeforeRoute, |ctx| {
    stats.record(ctx);
    ctx  // 原样传递，不改写
});
```

### 4.4 Protocol扩展示例

```rust
// 注册MCP协议适配器
registry.register_protocol("mcp", |transport| {
    // 将MCP请求转换为OpenLink的Context
    // 将OpenLink的Action响应转换为MCP格式
});
```

## 五、场景覆盖（当下→未来）

### 5.1 当下：基础短链（必须保留）

```
d.aw/abc123 → 302 → https://example.com/long-url
```

Route只有default，Action只有Redirect。最简形态，兼容所有传统短链场景。

### 5.2 近期：Agent协作 + 文件中转

```
d.aw/agent-task → 根据Agent身份路由
  - Planner Agent → 返回任务分解JSON
  - Worker Agent  → 返回执行指令
  - 人类          → 跳转任务看板

d.aw/report-2026 → 文件中转（根据设备环境走不同策略）
  - 手机（低带宽）→ 压缩预览版传输
  - 云电脑        → 原文件直传
  - Agent         → 返回文件元数据+下载URL，不传文件本身

d.aw/project-audio → 多Agent并发访问同一文件（自动选择最优路径）
  - Agent A（云电脑，同LAN） → DirectTransfer，从NAS直传，Gbps
  - Agent B（手机，同LAN）   → DirectTransfer，从已下载的Agent A直传，Gbps
  - Agent C（异地Agent）     → P2PTransfer，Tailscale穿透，Mbps
  - Agent D（远程Agent）     → CloudRelay，R2中转，按带宽走
```

### 5.3 中期：DAW生态衔接

```
d.aw/vc-comp    → 安装VC压缩机插件
d.aw/project-x  → 打开DAW工程
d.aw/jsfx/tape  → 加载JSFX脚本
d.aw/mix-42     → 加入混音会话
```

### 5.4 远期：人形机器人 + 物理智能体

```
"帮我找一下昨天的会议录音" → 机器人调用OpenLink resolve → 定位文件 → OpenVault校验完整性 → 最优路径传输
"把这个文件发给小王" → 机器人理解意图 → 路由到小王设备 → 根据网络环境自动选传输方式
"家里的监控录像备份了吗" → 机器人查询OpenVault状态 → 汇报副本健康 → 异常则触发补备
```

人形机器人是特殊的Agent——有物理身体的Agent。它需要：
- **语音入口**：自然语言指令 → OpenLink API调用
- **实时响应**：机器人不能等，路由决策必须<100ms
- **物理上下文**：机器人知道自己在哪（GPS/室内定位），Context自动填充
- **隐私隔离**：机器人处理家庭私密文件时，端到端加密，不经云端

架构支持：注册`PhysicalAgent`类型的Condition和Action即可，核心不改。

```
d.aw/agent://planner-north-3 → 发现并连接指定Agent
d.aw/workflow://deploy-pipe  → 触发编排好的多Agent工作流
d.aw/skill://summarize       → 动态加载Agent能力
d.aw/market://plugin-search  → Agent能力的发现与交易
```

**所有这些场景，架构不需要改，只需要注册新的Action/Condition/Protocol扩展。**

## 六、数据模型

### 6.1 核心表

```sql
-- 链接实体
CREATE TABLE links (
    id          UUID PRIMARY KEY,
    code        VARCHAR(32) UNIQUE NOT NULL,  -- 短码
    payload     JSONB DEFAULT '{}',
    owner_id    VARCHAR(255) NOT NULL,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    metadata    JSONB DEFAULT '{}',
    is_active   BOOLEAN DEFAULT TRUE,
    visibility  VARCHAR(16) DEFAULT 'public'  -- public | private
);

-- 路由规则
CREATE TABLE routes (
    id          UUID PRIMARY KEY,
    link_id     UUID NOT NULL REFERENCES links(id),
    rules       JSONB DEFAULT '[]',           -- 有序规则列表
    default_target JSONB NOT NULL,             -- 兜底目标
    version     INTEGER NOT NULL DEFAULT 1,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- 扩展注册
CREATE TABLE extensions (
    id          UUID PRIMARY KEY,
    ext_type    VARCHAR(64) NOT NULL,          -- action/condition/hook/protocol
    name        VARCHAR(128) UNIQUE NOT NULL,
    config      JSONB DEFAULT '{}',
    is_active   BOOLEAN DEFAULT TRUE,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- 访问日志
CREATE TABLE access_logs (
    id          UUID PRIMARY KEY,
    link_id     UUID NOT NULL,
    context     JSONB NOT NULL,                -- 完整上下文快照
    matched_rule VARCHAR(255),                 -- 命中的规则
    action_taken VARCHAR(128),                 -- 执行的动作
    response_time_ms INTEGER,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### 6.2 索引策略

```sql
-- 短码查询（最高频）
CREATE INDEX idx_links_code ON links(code) WHERE is_active = TRUE;

-- 按owner查询
CREATE INDEX idx_links_owner ON links(owner_id);

-- 按时间范围统计
CREATE INDEX idx_access_logs_link_time ON access_logs(link_id, created_at);

-- JSONB索引（路由规则查询）
CREATE INDEX idx_routes_rules ON routes USING GIN(rules);
```

## 八、API设计

### 8.1 短链管理

```
POST   /v1/links              # 创建短链
GET    /v1/links/:code        # 查询短链信息
PUT    /v1/links/:code        # 更新短链
DELETE /v1/links/:code        # 删除短链
GET    /v1/links/:code/stats  # 访问统计
```

### 8.2 路由规则

```
POST   /v1/links/:code/routes        # 创建路由规则
PUT    /v1/links/:code/routes/:id    # 更新路由规则
DELETE /v1/links/:code/routes/:id    # 删除路由规则
```

### 8.3 扩展管理

```
POST   /v1/extensions          # 注册扩展
GET    /v1/extensions          # 列出扩展
PUT    /v1/extensions/:name    # 更新扩展
DELETE /v1/extensions/:name    # 卸载扩展
```

### 8.4 重定向（无需认证）

```
GET    /:code                   # 短链重定向（核心路径，最快）
```

### 8.5 Agent专用

```
POST   /v1/agent/resolve        # Agent解析短链（返回结构化数据而非重定向）
POST   /v1/agent/batch          # 批量创建短链
GET    /v1/agent/discover       # 发现可用的Action/Extension
```

## 九、MVP路线图

### Phase 1：地基 — 核心原语 + 基础短链

**目标：跑通最短路径，传统短链能力100%覆盖**

- [ ] 5个核心原语的数据结构定义
- [ ] 路由引擎核心：Context → Rule匹配 → Action调度
- [ ] Extension Registry框架（注册/查询/调用）
- [ ] Redirect Action（传统短链重定向）
- [ ] HTTP API：短链CRUD + 重定向
- [ ] SQLite存储
- [ ] Docker单容器部署

**验收标准：** `curl d.aw/abc` 返回302重定向，传统短链完全可用

### Phase 2：扩展骨架 — 动态路由 + Hook

**目标：证明架构的扩展性**

- [ ] 条件路由（identity/device/location匹配）
- [ ] Webhook Action扩展
- [ ] BeforeRoute/AfterRoute Hook机制
- [ ] 访问日志 + 基础统计
- [ ] API认证（Token）

**验收标准：** 同一短链，浏览器访问跳网页，curl访问返回JSON

### Phase 3：Agent接入 + 文件中转 — SDK + 跨设备传输

**目标：Agent能直接用 + 跨设备文件共享（云存储）**

- [ ] Rust Agent SDK
- [ ] Agent身份识别Condition扩展
- [ ] Workflow Action扩展（多步编排）
- [ ] Agent专用API（resolve/batch/discover）
- [ ] FileTransfer Action扩展 — CloudRelay模式
  - 存储后端抽象trait（StorageBackend: upload/download/presigned_url/delete）
  - Cloudflare R2实现（推荐主力，下行流量免费，S3兼容，免费10GB+10M请求/月）
  - 阿里云OSS实现（备选）
  - 城通网盘实现（归档场景，API已有）
  - WebDAV实现（加密中转）
  - SFTP实现（传统服务器）
  - ECS本地实现（测试用）
  - 自定义后端（Extension注册）
- [ ] 存储路由（Storage Router）— 跟Link路由引擎同构
  - 根据文件特征匹配存储后端（大小/类型/标签/来源/自定义）
  - 支持分流：按权重分配到不同后端（负载均衡/容灾）
  - 存储后端管理API / 存储路由规则管理API
- [ ] 文件上传：Presigned URL直传模式（ECS只管元数据，文件直传存储后端）
- [ ] 多Agent并发访问同一文件：每个Agent根据自身Context走不同路径
- [ ] PostgreSQL存储（可选切换）

**存储选型决策：**
- 推荐 Cloudflare R2：下行流量免费，解决ECS 1Mbps出站瓶颈；S3兼容API；免费额度起步
- 存储后端通过trait抽象可插拔，配置切换不改动代码
- 配置示例：`[storage] backend = "r2" bucket = "openlink-files"`

**文件上传设计：Presigned URL 直传模式 + 存储路由**
- 核心原则：ECS不碰文件本身，只处理元数据，文件直传存储后端
- 上传流程：客户端请求上传 → 存储路由匹配后端 → 生成presigned upload URL → 客户端直传 → 确认创建Link
- 下载流程：访问短链 → 存储路由匹配后端 → 生成presigned download URL → 客户端直连下载
- **存储路由（Storage Router）**：跟Link路由引擎同构，根据文件特征匹配存储后端
  - 路由规则示例：大文件→R2 / 归档文件→网盘 / 临时文件→本地 / 敏感文件→加密WebDAV
  - 支持分流：同一文件类型可按权重分配到不同后端（负载均衡/容灾）
  - 路由条件：文件大小 / 文件类型 / 用途标签 / 来源设备 / 自定义条件（可扩展）
- 存储后端（全部通过StorageBackend trait抽象，注册即用）：
  - Cloudflare R2（推荐主力，下行免费，S3兼容）
  - 阿里云OSS（备选）
  - 城通网盘（归档场景，API已有）
  - WebDAV（自建/第三方，适合加密中转）
  - SFTP（传统服务器）
  - 本地存储（ECS，测试用）
  - 自定义后端（Extension注册）
- 不同设备策略：云电脑原文件直传 / 手机可选压缩 / Agent程序化上传 / SDK一键上传
- API：
  - `POST /v1/files/upload-request` → 存储路由匹配后端，返回presigned URL + file_id
  - `POST /v1/files/confirm` → 上传完成确认，创建Link
  - `GET /:code` → 普通访问302到presigned URL
  - `POST /v1/agent/resolve` → Agent访问返回元数据+presigned URL
  - `GET/DELETE /v1/files/:id` → 文件元数据查询/删除
  - `GET/POST/DELETE /v1/storage/backends` → 存储后端管理（注册/查询/卸载）
  - `GET/POST/PUT/DELETE /v1/storage/routes` → 存储路由规则管理
- 安全：Presigned URL TTL 15分钟 / 上传需API Token / 文件大小可配置(默认100MB) / 文件类型白名单可配置
- 配置示例：
  ```toml
  [storage]
  default_backend = "r2"
  
  [storage.backends.r2]
  type = "r2"
  bucket = "openlink-files"
  
  [storage.backends.ctdisk]
  type = "ctdisk"          # 城通网盘
  ```

## 十一、设计哲学：去中心化与云端共存

**核心类比：PPS播放器的进化版——人越多越快，但永远不断流。**

OpenLink不是纯去中心化，也不是纯云端，而是**路由引擎动态决策的最优路径选择**：

| 模式 | 适用场景 | 特点 |
|------|---------|------|
| p2p-first | 大文件分享 | P2P优先，云端兜底 |
| cloud-first | 隐私敏感文件 | 云端优先，P2P加速 |
| adaptive（默认） | 通用 | 路由引擎动态选最优 |
| cloud-only | 关闭P2P | 传统CDN模式 |

**冷启动→热传播飞轮：**
1. 第1个Agent → CloudRelay（云端源）
2. 第2个Agent（同LAN） → DirectTransfer（从第1个直传，Gbps）
3. 第3个Agent（异地） → P2PTransfer（穿透直连，Mbps）
4. 越多节点做种 → 路径选择越多 → 云端源压力下降
5. 最终云端只服务冷启动和极端弱网

**关键原则：保底能力比峰值性能更重要。** P2P没人做种就废了，OpenLink最差走CloudRelay，速度有下限。

## 十二、链接与文件生命周期

### 链接过期

```rust
Link {
    // ... 已有字段
    expires_at: Option<Timestamp>,     // 过期时间
    max_clicks: Option<u64>,           // 最大点击次数
    expired_action: ExpiredAction,     // 过期后的行为
}

ExpiredAction = Gone | Redirect(String) | Custom(Extension)
```

- expires_at：到期自动失效
- max_clicks：达到点击上限自动失效
- 过期行为可配置：返回410 Gone / 跳转提示页 / 自定义Extension处理

### 文件过期

```toml
[file-lifecycle]
default_ttl = "7d"              # 默认7天
max_ttl = "90d"                 # 最长90天

[[file-lifecycle.overrides]]
tag = "temp"
ttl = "1d"                      # 临时文件1天

[[file-lifecycle.overrides]]
tag = "archive"
ttl = "365d"                    # 归档文件1年

on_expire = "delete"            # delete | archive-to-cold | custom
```

### Link与文件联动

- Link过期 → 文件不一定删（可能有其他Link指向同一文件）
- 文件过期 → 所有指向该文件的Link自动失效
- 引用计数：文件只在没有任何Link指向且自身过期时才真正删除
- 定时清理任务：扫描过期Link和文件，执行删除/归档

## 十三、安全防火墙 — 公私分治

**核心原则：公开内容过审，私密内容不管。谁还没点隐私小爱好呢。**

安全防火墙只作用于**公开内容**（is_private = false）。私密内容走端到端加密通道，系统不解密、不扫描、不过审。

### 公私分治模型

```
Link/文件 {
    visibility: Public | Private,
    // Public  → 过三道防线，合规才能发布
    // Private → 端到端加密，系统只管传输和存储，不管内容
}
```

| 维度 | 公开内容 (Public) | 私密内容 (Private) |
|------|-------------------|-------------------|
| 内容扫描 | ✅ 必须过审 | ❌ 不扫描 |
| URL过滤 | ✅ 域名黑名单+关键词 | ❌ 不过滤 |
| 访问限制 | 可匿名访问 | 仅授权身份访问 |
| 加密 | 传输加密(HTTPS) | 端到端加密(服务端不可读) |
| 举报机制 | ✅ 支持举报 | ❌ 无举报 |
| 合规责任 | 平台负责 | 用户自己负责 |
| 存储位置 | 公开存储(R2等) | 加密存储(独立bucket) |
| 传输路径 | CloudRelay/DirectTransfer | 加密直传/P2P |

### 三道防线（仅公开内容）

**第一道：URL过滤（链接创建时）**
- 域名黑名单 + 关键词过滤
- 集成第三方URL安全检测API（阿里云/腾讯云）
- 可疑URL进入人工审核队列
- 拦截：黄/赌/毒/诈骗/暴恐/政治敏感

**第二道：文件内容扫描（文件上传时）**
- 图片：色情/暴恐识别（接入阿里云内容安全/腾讯云天御）
- 文本：敏感词过滤 + 语意分析
- 文件哈希：MD5/SHA256对接违规文件黑名单
- 无法自动判断 → 暂存 + 人工审核

**第三道：访问行为监控（运行时）**
- 异常访问模式检测（短时间大量访问 → 疑似违规传播）
- 举报机制：每个Link页面带举报按钮
- 被举报链接自动降级（暂停访问 + 进入审核）

### 私密内容保障

- 端到端加密：客户端加密后上传，服务端只存密文，无密钥无法解密
- 零知识架构：即使服务端被入侵，私密内容也无法读取
- 密钥管理：用户自有密钥，可选用硬件密钥/密码派生/社交恢复
- 人形机器人访问私密内容：需用户授权+生物验证，机器人不缓存明文

## 十、与OpenDAW的架构映射

两个项目共享同一套设计哲学，架构同构：

## 十一、设计铁律

1. **核心层零业务逻辑** — 路由引擎不知道"短链"是什么，只知道Context→Action
2. **新功能=注册扩展** — 任何新场景都不改核心代码
3. **协议层可插拔** — HTTP只是第一个协议，不是唯一协议
4. **存储层可替换** — 核心逻辑通过trait抽象，不绑定具体数据库
5. **配置优于代码** — 路由规则、Hook顺序、扩展启停，全部可配置
6. **兼容性第一** — 传统短链(`GET /:code → 302`)永远是零配置开箱即用
7. **可观测内置** — 每次路由决策都有完整上下文记录，不可关闭
