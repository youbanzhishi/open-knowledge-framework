## 决策1：短链在AI时代的价值重估

**决策：短链的本质不是"长变短"，是可寻址的入口协议。**

- 传统短链 = 长URL变短 + 点击统计
- AI时代短链 = 可编程路由层 + Agent协作入口 + 扩展分发协议
- URL是人类互联网的入口协议 → OpenLink是智能体互联网的入口协议

**推导逻辑：** Agent执行动作、跳转、分发都需要可寻址、可路由的URL层 → 短链从"静态映射"升级为"动态路由引擎"

## 决策2：为什么选Rust而不是Go/Java

**决策：不是性能原因，是生态位统一。**

| 维度 | 分析 |
|------|------|
| 短链场景性能 | Go就能跑满，Rust是性能冗余 |
| AI基础设施生态 | Rust在吃份额（candle/burn/tract/PyO3/WASM），Go几乎缺席 |
| 与OpenDAW统一 | 两个项目共享技术栈 = 共享经验、组件、心智模型 |
| 长期扩展 | WASM边缘部署、Python桥接、共享crate |

**教训：** 技术选型不要只看单项目维度，要看整个技术版图的统一性。

## 决策3：Extension Registry四柱模型

**决策：跟OpenDAW同构的扩展架构，覆盖所有扩展场景永不需要新增。**

| 柱 | 功能 | 注册什么 |
|----|------|---------|
| Action API | 注册新动作 | 执行行为（重定向/文件传输/工作流...） |
| Condition API | 注册新条件 | 路由判断逻辑（设备类型/身份/网络...） |
| Hook API | 注册新拦截器 | 流程拦截（日志/认证/改写/降级...） |
| Protocol API | 注册新协议适配器 | 通信协议（HTTP/MCP/A2A/自定义...） |

**核心哲学：新功能=注册扩展，架构本身永远不需要改。**

## 决策4：文件中转不单开项目

**决策：Link可以指向任何资源，文件是最核心的资源。拆开反而制造人为边界。**

- 短链指向URL → 重定向
- 短链指向文件 → 传输
- 短链指向Agent → 握手
- 路由引擎天然支持，只是Action类型不同

**教训：** 架构边界应该沿着自然的数据流划分，不沿着功能标签划分。

## 决策5：Presigned URL直传模式

**决策：ECS只管元数据，文件直传存储后端。解决1Mbps出站瓶颈。**

- 上传：客户端请求 → ECS生成presigned URL → 客户端直传R2 → 确认创建Link
- 下载：访问短链 → ECS生成presigned URL → 客户端直连R2
- 安全：Presigned URL TTL 15分钟过期作废

**教训：** 低资源环境只做轻量调度，重流量走外部存储。

## 决策6：内网穿透选型

**决策：Tailscale（首选，零配置）→ STUN/TURN降级 → CloudRelay兜底**

选择逻辑：Tailscale零配置跟OpenLink的"自动最优路径"理念最契合。不可用时降级到STUN/TURN。

**教训：** 去中心化不是非黑即白，应该是一个可调的滑动条。保底能力比峰值性能更重要。

---

## 决策7：安全防火墙 — 公私分治

**决策：公开内容过审，私密内容不管。**

- 三层防御全部做成Hook扩展（不改核心代码），仅对公开内容（visibility=public）生效
- 私密内容走端到端加密，系统零知识
- visibility字段是分界线 → 一行配置决定走哪条处理管道

**教训：** 安全合规不是功能需求，是生存需求。隐私保护同样是基本权利。两者不矛盾。

---

## 决策8：链接与文件生命周期

**决策：链接支持时间过期+次数过期，文件按标签设TTL，引用计数联动。**

- 链接过期 ≠ 文件删除（可能有其他Link指向）
- 文件过期 → 所有关联Link失效
- 真正删除条件：引用计数=0 且 文件已过期

**教训：** 资源的生命周期管理是基础设施的必修课，不做就是定时炸弹。

---

## 决策9：GitHub Actions CI — 编译交给云端

**决策：不在云电脑重编译，交给GitHub Actions，ECS只拉产物部署。**

- 云电脑：开发调试
- GitHub Actions：自动编译+测试+Release
- ECS：部署运行Docker容器

**新流程：** push到GitHub → Actions自动编译测试 → 打tag → Actions出Release → ECS拉二进制部署

**教训：** 低资源环境只做`cargo check`，重编译交给CI。部署只拉产物，不自编译。

## 决策10：扣子协作痛点 — 三层知识共享体系

| 层级 | 方案 | 治什么 |
|------|------|--------|
| 治本 | 共享知识库 `./共享知识/` | 知识不流通 |
| 治标 | open-dev-tools升级 | 重复造轮子 |
| 治根 | 项目级技能 | 协作断裂 |

**原则：** 一个项目踩过的坑，其他项目绝对不能再踩。工具层面各项目可以不同，但经验和模式必须流通。
